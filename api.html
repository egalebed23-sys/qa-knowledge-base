01 Что такое API basics
L1 · Основы

API (Application Programming Interface) — это контракт, по которому одна программа обращается к другой. Он описывает, что можно запросить, что нужно передать и что вернётся в ответ — не раскрывая, как это устроено внутри.

АналогияAPI как сигнатура функции: ты знаешь её имя, какие аргументы подать и что она вернёт — и пользуешься, не зная и не видя реализации. get_user(id) → User. Что происходит внутри (поход в БД, кэш, другой сервис) — тебя не касается. Меняется реализация — API остаётся прежним.

Простой пример: приложение погоды на телефоне само не считает погоду. Оно дёргает API погодного сервиса: «дай погоду для Москвы» → получает JSON с температурой. Приложение — клиент, сервис погоды — сервер, API — правила их общения.

L2 · Детали

API бывают не только веб-овые: Web API (по сети, обычно HTTP), Library API (функции библиотеки — requests.get()), OS API (системные вызовы). В контексте Backend-тестирования «API» почти всегда значит Web API поверх HTTP — о нём и речь.

По доступу Web API делят на public (любой разработчик), partner (по договору, с ключом) и private/internal (между своими сервисами).

Клиент-серверКлиент отправляет запрос (браузер, приложение, другой сервис), сервер отвечает. QA обычно в роли клиента: шлёт запросы и проверяет ответы.
L3 · Senior

Почему QA так много работает именно с API, а не только через UI:

  • Быстрее — API-тест в десятки раз быстрее UI-теста (нет рендеринга, кликов, ожиданий)
  • Стабильнее — UI меняется постоянно (вёрстка, кнопки), API-контракт стабильнее
  • Глубже — через UI не проверишь то, что фронт не показывает: коды ошибок, граничные ответы, безопасность
  • Раньше — API готов до фронта, можно тестировать, пока UI ещё пишут
В реальном миреСовременный подход — API-first: сначала проектируют и согласуют контракт API (в OpenAPI), и только потом параллельно пишут бэкенд, фронтенд и тесты. Между клиентами и сервисами часто стоит API Gateway (Kong, Apigee, AWS API Gateway) — единая точка входа, которая занимается авторизацией, rate limiting, маршрутизацией и логированием. Для QA это значит: часть проверок (лимиты, auth) делается на уровне gateway, а не самого сервиса.
Каверзный вопрос«Чем API отличается от endpoint?» — API — это весь интерфейс сервиса целиком (все его возможности). Endpoint — конкретный адрес одной операции внутри API (GET /users/42). Один API состоит из множества endpoints. Грубо: API — это меню, endpoint — отдельное блюдо в нём.
02 HTTP: запрос и ответ protocol basics
L1 · Основы

HTTP — протокол, по которому клиент (браузер, мобильное приложение, другой сервис) общается с сервером. Всё взаимодействие — это пары запрос → ответ поверх TCP.

В каждом запросе есть четыре части:

  • Метод — что хотим сделать (GET, POST, ...)
  • URL — куда обращаемся (https://api.shop.com/orders/42)
  • Заголовки — метаданные (Content-Type, Authorization, ...)
  • Тело — данные запроса (обычно JSON; не у всех методов есть)

В ответе — три части: статус-код (200, 404, 500), заголовки, тело.

АналогияHTTP-запрос как команда в shell: метод = команда (cat, rm, cp) — что делаем. URL = путь к файлу — над чем. Заголовки = флаги и опции (-r, --force) — как именно. Тело = stdin — что передаём на вход. Ответ — это exit code (статус) + stdout (тело) + дополнительные метаданные (заголовки).
L2 · Детали

Стандартный вид сырого HTTP-запроса и ответа:

// Запрос
POST /api/orders HTTP/1.1 Host: api.shop.com Content-Type: application/json Authorization: Bearer eyJhbGc... { "product_id": 42, "quantity": 2 }
// Ответ
HTTP/1.1 201 Created Content-Type: application/json Location: /api/orders/789 { "id": 789, "status": "pending" }

Особенности протокола:

  • Stateless — каждый запрос независим, сервер не «помнит» предыдущий. Контекст передаётся явно (cookie, токен)
  • Текстовый (в HTTP/1.1) — можно посмотреть глазами через DevTools, curl -v, Wireshark
  • Поверх TCP — гарантирует доставку и порядок пакетов
  • HTTPS = HTTP + TLS — то же самое, но в зашифрованном туннеле
L3 · Senior

HTTP/1.1 vs HTTP/2 vs HTTP/3 — в чём разница:

ВерсияТранспортГлавное отличие
HTTP/1.1 (1997)TCPТекстовый, один запрос на соединение (keep-alive — несколько подряд, но не параллельно). Head-of-line blocking на уровне HTTP
HTTP/2 (2015)TCPБинарный, мультиплексирование (много запросов параллельно в одном соединении), сжатие заголовков (HPACK). Server Push был частью спеки, но на практике почти не используется и свёрнут в браузерах
HTTP/3 (2022)QUIC (UDP)Решает head-of-line blocking на уровне TCP — потеря одного пакета не блокирует все потоки. Быстрое установление соединения
В реальном миреНа собесе расскажи разницу, но в работе версия HTTP — почти никогда не предмет ручного выбора, это решает инфраструктура (load balancer, CDN). Нюанс для Python: requests работает по HTTP/1.1; для HTTP/2 и async-запросов используют httpx.
Каверзный вопрос«Можно ли в HTTP сделать запрос без тела?» — Да, у методов GET, HEAD, DELETE тело обычно не передаётся. Спецификация формально не запрещает, но прокси и серверы могут его игнорировать или отвергать. Если нужно передать данные с GET — используй query-параметры в URL.
03 HTTP-методы и идемпотентность verbs
L1 · Основы

Метод говорит серверу что мы хотим сделать с ресурсом. Основные:

  • GET получить ресурс. Без тела. Безопасный и идемпотентный
  • POST создать новый ресурс. С телом. Не идемпотентный (повторный = новая сущность)
  • PUT заменить ресурс целиком. Идемпотентный
  • PATCH обновить часть ресурса. Чаще идемпотентный, но не обязан
  • DELETE удалить ресурс. Идемпотентный
Идемпотентность простыми словами«Если повторить запрос несколько раз — результат на сервере будет тот же, что после первого». GET ничего не меняет — идемпотентный. DELETE удалит один раз, второй уже не найдёт (или вернёт 404) — но состояние сервера то же. POST «создать заказ» при повторе создаст ещё один — не идемпотентный.
L2 · Детали

Три свойства методов, которые часто путают:

МетодSafeIdempotentCacheable
GETдадада
HEADдадада
OPTIONSдаданет
POSTнетнетиногда
PUTнетданет
PATCHнетобычно нетнет

  • Safe — не меняет состояние сервера (только чтение)
  • Idempotent — повторение запроса не меняет результат
  • Cacheable — ответ можно кэшировать

Менее популярные, но реальные методы:

HEAD
То же, что GET, но без тела ответа — только заголовки. Используют чтобы проверить существование ресурса (HEAD на файл — есть ли) или его размер (Content-Length) без скачивания.
OPTIONS
Узнать, какие методы поддерживает эндпоинт. Главное практическое применение — CORS preflight: браузер автоматически шлёт OPTIONS перед «нестандартными» запросами с другого домена, чтобы проверить разрешения.

PUT vs PATCH — ключевая разница:

// PUT — заменить целиком
PUT /users/42 { "name": "Иван", "email": "i@ex.com", "phone": "+7..." } // Если поле опустить — оно // очистится / станет null
// PATCH — обновить часть
PATCH /users/42 { "phone": "+7-новый" } // Остальные поля остаются // без изменений
L3 · Senior

Идемпотентность для QA — это о тест-стратегии, а не теоретическое свойство. Что проверять:

  • Если документация говорит «метод идемпотентный» — повтори запрос 5 раз подряд. Результат на сервере должен быть тот же, что после первого вызова
  • На неидемпотентных POST — проверь, как сервер защищается от дублей: Idempotency-Key в заголовке (Stripe, Yandex) — клиент шлёт уникальный ключ, сервер запоминает и при повторе возвращает старый результат
  • В платёжных и финансовых API — без идемпотентности и Idempotency-Key никуда. Это вопрос денег пользователя при ретраях
В реальном миреВ большинстве REST API современных продуктов PATCH преобладает над PUT — частичные обновления удобнее и безопаснее. PUT в чистом виде встречается реже, чаще используется как «upsert» (создать или обновить целиком). Также набирает популярность JSON Merge Patch (RFC 7396) и JSON Patch (RFC 6902) — стандартизированные форматы для PATCH-операций.
Делегируй LLMГенерация cURL-запросов: «Напиши cURL для POST /api/orders с заголовками Authorization Bearer и Content-Type application/json, телом {product_id: 42, quantity: 2}.»

Анализ непонятного API: вставь OpenAPI-спеку или пример запроса/ответа → «Какие методы есть, какие идемпотентны, что вернёт сервер при повторном вызове DELETE?»

Генерация тестов на идемпотентность: «Для эндпоинта POST /payments с поддержкой Idempotency-Key напиши набор тест-кейсов: повторный запрос с тем же ключом, с другим, без ключа, с устаревшим.»
Каверзный вопрос«PATCH идемпотентен?» — Зависит от реализации. PATCH /counter {"increment": 1} — НЕ идемпотентен (каждый раз +1). PATCH /user {"name": "Иван"} — идемпотентен (всегда тот же результат). Senior-ответ: «PATCH может быть идемпотентным, но не обязан, в отличие от PUT, который идемпотентен по спецификации».
04 HTTP статус-коды response codes
L1 · Основы

Статус-код — трёхзначное число в ответе сервера. Первая цифра определяет класс:

1xx — информационные
  • 100 Continue
  • 101 Switching Protocols (WebSocket)
2xx — успех
  • 200 OK
  • 201 Created
  • 204 No Content
3xx — перенаправление
  • 301 Moved Permanently
  • 302 Found
  • 304 Not Modified (из кеша)
4xx — ошибка клиента
  • 400 Bad Request
  • 401 Unauthorized
  • 403 Forbidden
  • 404 Not Found
  • 409 Conflict
  • 422 Unprocessable Entity
  • 429 Too Many Requests
5xx — ошибка сервера
  • 500 Internal Server Error
  • 502 Bad Gateway
  • 503 Service Unavailable
  • 504 Gateway Timeout
АналогияКак exit codes в Unix, только детальнее: 2xx — успех (exit 0). 3xx — перенаправление, ходи в другое место. 4xx — ошибка на стороне клиента (невалидный ввод). 5xx — сервер упал, твоей вины нет.
L2 · Детали

2xx — успех, но с нюансами:

КодКогда возвращать

201 CreatedСоздан новый ресурс. Обычно после POST. Часто включает заголовок Location: /api/orders/789

204 No ContentУспех, но тело ответа пустое. Типично для DELETE и PUT без возврата

4xx — ошибки клиента, наиболее частые:

КодСмысл

401 Unauthorized«Кто ты?» — не передан токен / он невалидный / истёк
403 Forbidden«Я тебя знаю, но тебе нельзя» — авторизован, но прав нет

L3 · Senior

Каверзные пары кодов:

401 vs 403
401 — «кто ты вообще?» Не аутентифицирован. Решение: войти / обновить токен.
403 — «знаю тебя, но нельзя». Аутентифицирован, но не имеешь прав на этот ресурс.
400 vs 422
400 — невалидный синтаксис: битый JSON, отсутствие обязательного поля.
422 — синтаксис ок, но семантика плохая: email "abc", цена -100. В REST API чаще именно 422 для валидации.
301 vs 302 vs 307 vs 308
301 навсегда, можно поменять метод на GET.
302 временно, можно поменять метод (исторически).
307 временно, метод и тело сохраняются.
308 навсегда, метод и тело сохраняются.
404 vs 410
404 — не найдено (возможно, никогда и не было).
410 Gone — было, но удалено навсегда (для поисковиков сигнал «забудьте URL»).
В реальном миреВ современных API часто сознательно используют 404 вместо 403 для защиты приватности. Если бы сервер отвечал 403 — это бы выдавало факт существования приватного ресурса. Возвращая 404, мы скрываем сам факт. Пример — GitHub: на приватный репозиторий с чужого аккаунта отдаётся 404, а не 403.
Антипаттерн«200 OK на ошибку» — встречается часто: сервер возвращает HTTP 200, но в теле {"status": "error", "code": 500}. Это ломает все retry-механизмы (они смотрят на HTTP-код), мониторинг (5xx-метрики чистые, а сервис «лежит»), и противоречит спецификации. На собесе можно упомянуть как пример того, что встречаешь в legacy и аргументируешь рефакторинг.
Каверзный вопрос«Какой код вернуть, если пользователь хочет удалить ресурс, которого нет?» — Зависит от логики:
· 404 — если важно сообщить «такого нет»
· 204 — если хотим идемпотентности: DELETE дважды → оба раза 204, как будто всё ок
В стандартах REST оба варианта валидны. Команда выбирает по сценарию использования.
05 Заголовки HTTP headers
L1 · Основы

Заголовки — пары Имя: Значение с метаданными запроса или ответа. Тело — это «что», заголовки — это «как, кому, в каком формате». Их десятки, но на собесе важны ~15.

Чаще всего в запросе
Host — куда (домен)
Content-Type — формат тела (application/json)
Authorization — токен / креды
Accept — что мы готовы принять в ответ
User-Agent — кто клиент (браузер, бот)
Cookie — куки от сервера
Referer — откуда пришли
Чаще всего в ответе
Content-Type — формат ответа
Content-Length — размер тела в байтах
Set-Cookie — установить куку у клиента
Cache-Control — как кэшировать
Location — куда редирект / где новый ресурс
Retry-After — через сколько повторить
L2 · Детали

Content-Type — главный заголовок для тестировщика API. Без него сервер не знает, как парсить тело:

  • application/json — JSON. 90% современных REST-API
  • application/xml — XML. Legacy, SOAP, банковские API
  • application/x-www-form-urlencoded — формы HTML, key=val&key2=val2
  • multipart/form-data — загрузка файлов
  • text/plain — просто текст

Accept — клиент говорит, что готов принять. Сервер может отдать в другом формате (content negotiation): Accept: application/json, application/xml;q=0.5 — предпочту JSON, но XML тоже пойдёт.

Кэширование — три уровня заголовков:

ЗаголовокГдеСмысл
Cache-Controlв ответеГлавный. max-age=3600, no-cache, private, public
ETagв ответеУникальный хеш версии ресурса
If-None-Matchв запросеКлиент посылает прошлый ETag → сервер вернёт 304 Not Modified, если не изменилось
Last-Modifiedв ответеДата последнего изменения ресурса
If-Modified-Sinceв запросеАналог If-None-Match, но по дате

Безопасностные заголовки (часто в нагрузке для QA):

  • Strict-Transport-Security (HSTS) — заставляет браузер всегда ходить по HTTPS
  • Content-Security-Policy (CSP) — что разрешено выполнять (защита от XSS)
  • X-Frame-Options — можно ли встраивать страницу в iframe (защита от clickjacking)
  • X-Content-Type-Options: nosniff — не угадывать MIME-тип, верить заголовку
L3 · Senior

Кастомные X- заголовки — клиенты и серверы часто добавляют свои. Префикс X- исторически означал «нестандартный», сейчас от него отказались (RFC 6648), но в legacy его всё ещё много:

  • X-Request-ID — трейсинг запроса через микросервисы
  • X-Forwarded-For — реальный IP клиента, когда между ним и сервисом стоят прокси / load balancer
  • X-Rate-Limit-Remaining — сколько запросов осталось до лимита
  • X-API-Version — версия API (альтернатива версионированию через URL)

CORS-заголовки — без понимания не пройти ни одного собеса по фронтенду или Backend, даже как QA:

  • Access-Control-Allow-Origin: https://shop.com — какие домены могут обращаться
  • Access-Control-Allow-Methods: GET, POST, PUT — какие методы разрешены
  • Access-Control-Allow-Credentials: true — можно ли слать куки
  • Preflight — браузер автоматически шлёт OPTIONS перед «нестандартными» запросами (не-GET, нестандартный Content-Type) и смотрит на CORS-ответ
В реальном миреQA-автоматизатор регулярно сталкивается с заголовками в трёх контекстах: 1) аутентификация через Authorization или кастомный X-API-Key, 2) трейсинг через X-Request-ID — без него невозможно найти запрос в логах в микросервисной архитектуре, 3) rate limiting — твои тесты должны уметь читать X-Rate-Limit-* и тормозить себя, а не падать с 429.
Делегируй LLMРасшифровка непонятных заголовков: вставь любой curl -v или дамп DevTools → «Что делает каждый заголовок? Какие из них критичны для теста, какие можно игнорировать?»

Генерация тест-кейсов на заголовки: «Эндпоинт принимает Authorization Bearer и X-API-Key. Сгенерируй негативные тест-кейсы: оба отсутствуют, один отсутствует, оба невалидны, формат сломан, регистр не тот.»
Каверзный вопрос«Что вернёт сервер, если послать запрос с Content-Type: application/json, но в теле — XML?» — Обычно 400 Bad Request: парсер JSON не сможет прочитать тело. Реже сервер пытается угадать формат, но это плохая практика. Хорошее тест-покрытие — проверить эту комбинацию: правильный Content-Type / неправильное тело и наоборот.
06 REST API и его принципы architectural style
L1 · Основы

REST (Representational State Transfer) — архитектурный стиль для API. Не протокол и не стандарт, а набор принципов того, как организовать обмен между клиентом и сервером.

Главные принципы:

  • Ресурсы — всё, чем оперируем (заказы, пользователи, товары) — это ресурсы с уникальным URL
  • URL = имя ресурса, метод HTTP = действие. GET /users/42, а не GET /getUser?id=42
  • Stateless — сервер не хранит состояние клиента между запросами. Всё нужное передаётся в запросе
  • Унифицированный интерфейс — одинаковые правила для всех ресурсов: GET читает, POST создаёт, и т.д.
АналогияREST как файловая система: каждый ресурс — это файл с уникальным путём (URL). Операции стандартизированы: cat /users/42 = GET, echo data > /users = POST, rm /users/42 = DELETE. Правила одинаковые для всех ресурсов — не нужно учить отдельный синтаксис для каждого.

REST vs SOAP — единственное, что обычно спрашивают про SOAP. SOAP — старый тяжёлый протокол на XML, который ещё встречается в банках и госуслугах:

Главное для собеса: REST гибче и легче, SOAP строже и тяжелее. Новые проекты на SOAP почти не делают.

L2 · Детали

URL-дизайн — самая частая ошибка в REST. Правильно:

ДействиеПравильно (REST)Неправильно (RPC-стиль)

УдалитьDELETE /orders/789POST /deleteOrder?id=789

Правила: существительные во множественном числе (/users, не /user), иерархия через вложение (/users/42/orders), никаких глаголов (действие определяет HTTP-метод).

Версионирование API — три подхода:

В URL
/api/v1/users

Самый распространённый. Просто, видно глазами, легко переключаться.
В заголовке
Accept: application/vnd.api+json;version=2

«Чистый» REST по Fielding — URL не меняется. Сложнее тестировать.
В query-параметре
/api/users?version=2

Редкий вариант. Удобно для миграции, но «грязно».

Пагинация — три модели:

ПодходПримерКогда

Фильтрация и сортировка — общепринятые форматы:

# Фильтры GET /orders?status=paid&user_id=42&created_after=2026-01-01 # Сортировка (минус = по убыванию) GET /orders?sort=-created_at,price # Выбор полей (sparse fieldsets) GET /users/42?fields=id,name,email # Включение связанных ресурсов GET /orders/789?include=user,items
L3 · Senior

Richardson Maturity Model — 4 уровня «зрелости» REST API:

  • Level 0 — один URL, один метод (обычно POST), всё через тело. По сути — RPC поверх HTTP. SOAP сюда
  • Level 1 — появляются ресурсы (разные URL), но методы используют криво (POST /getUser)
  • Level 2 — используются разные HTTP-методы по назначению + правильные статус-коды. Большинство «REST API» в реальности — это Level 2
  • Level 3 — HATEOAS: в ответе сервер возвращает ссылки на возможные следующие действия. Полноценный REST по Fielding

HATEOAS (Hypermedia as the Engine of Application State) — в ответе сервер кладёт ссылки на возможные действия:

{ "id": 789, "status": "pending", "_links": { "self": "/orders/789", "cancel": "/orders/789/cancel", "pay": "/orders/789/pay" } }

Идея: клиент не «знает» URL заранее, а ходит по ссылкам. На практике почти никто так не делает — слишком сложно для клиента и слишком много трафика.

В реальном миреВ работе под «REST» обычно понимают Richardson Level 2: разные URL, осмысленные методы, JSON в теле, нормальные коды. HATEOAS редко встречается в прикладных JSON API — в продакшене его почти не найти, кроме отдельных банков и Spring HATEOAS / HAL-проектов. Если на собесе спросят «у нас REST» — уточни, что под этим понимают. Часто это просто «HTTP + JSON».
Типичные нарушения REST · Глаголы в URL: /api/createUser, /api/getOrders
· 200 OK на ошибки: {"error": "not found"} вместо 404
· Изменение состояния через GET: GET /users/42/delete
· Версионирование без плана — наплодили v1, v2, v3, никто не помнит, что где
· Возврат HTML вместо JSON при ошибке — клиент пытается распарсить, падает
Делегируй LLMРевью REST API: вставь список эндпоинтов → «Какие нарушают REST-принципы? Предложи правильные варианты с обоснованием.»

Дизайн нового API: «Спроектируй REST API для системы заказов: пользователи, заказы, позиции в заказе, статусы. Дай URL-схему, методы, примеры запросов/ответов.»
Каверзный вопрос«Если REST stateless, как работает авторизация?» — Авторизация передаётся в каждом запросе: токен в Authorization-заголовке или кука. Сервер не «помнит» сессию между запросами в смысле бизнес-состояния, но может валидировать токен (JWT — без БД, OAuth — с проверкой в БД/Redis). Stateless ≠ «без авторизации», а «без хранения контекста запросов».
07 GraphQL · gRPC api styles
L1 · Основы

Кроме REST есть ещё два популярных стиля API. Когда какой использовать:

GraphQL
Один эндпоинт. Клиент сам пишет запрос и указывает, какие поля ему нужны. Меньше over-/under-fetching. Хорош для гибких UI.
gRPC
Бинарный протокол поверх HTTP/2 с контрактом через Protobuf. Быстрый, типизированный. Идеален для общения между микросервисами.
АналогияREST — REST-эндпоинты типа /users/42 с заранее определённой структурой ответа. GraphQL — SQL-style: клиент указывает поля, как SELECT name, email FROM users WHERE id=42. gRPC — Unix domain sockets для общения между процессами: быстро, бинарно, типизированно, не для browser.
L2 · Детали

GraphQL — один эндпоинт (/graphql), клиент сам пишет запрос. По сети это обычный POST с JSON-телом, где запрос лежит в поле query:

// Запрос (реальное HTTP-тело)
POST /graphql Content-Type: application/json { "query": "{ user(id: 42) { name email } }", "variables": {} }
// Ответ
{ "data": { "user": { "name": "Иван", "email": "i@ex.com" } } }

Важно для тестирования: GraphQL — это POST с JSON, где сам запрос — строка в поле query (+ variables для параметров). В Postman/requests шлёшь обычный JSON, а не «графкуэловский синтаксис» напрямую.

  • Query — чтение, Mutation — изменение, Subscription — real-time через WebSocket
  • Schema — типизированный контракт всех типов и операций
  • Плюс: нет over-/under-fetching, версионирование не нужно. Минус: сложнее кэшировать, N+1 проблема, сложнее rate-limiting

gRPC — бинарный RPC от Google. Контракт описывается в .proto-файле:

// user.proto service UserService { rpc GetUser(GetUserRequest) returns (User); rpc ListUsers(ListUsersRequest) returns (stream User); } message User { int32 id = 1; string name = 2; string email = 3; }
Что такое .proto файлЭто текстовый файл с описанием gRPC-сервиса на языке Protocol Buffers. В нём определяют: service — какие методы (RPC) есть, message — структуры данных и типы полей. Цифры (= 1, = 2) — это номера полей для бинарной сериализации, а не значения. Из .proto компилятор protoc генерирует готовый код клиента и сервера на нужном языке. По сути это контракт — аналог OpenAPI для REST или WSDL для SOAP, только строже и компактнее.
  • HTTP/2 + Protobuf — бинарная сериализация, в разы меньше JSON по размеру
  • Генерация кода — из .proto генерируются клиенты и серверы на 10+ языках. Полностью типизировано
  • Streaming — поддержка bi-directional streaming из коробки
  • Где: внутреннее общение между микросервисами (Google, Netflix, Uber, Yandex)
  • Плюс: быстро, компактно, строго типизировано, контракт встроен
  • Минус: бинарь нельзя посмотреть глазами в DevTools, плохо работает с браузерами без gRPC-Web прокси
L3 · Senior

Когда какой стиль выбирать — итоговая таблица:

Особенности тестирования каждого:

В реальном миреСовременная типичная архитектура: REST/GraphQL наружу (для веба и мобайла) + gRPC внутри (между микросервисами). GraphQL хорош, но требует зрелой команды — иначе даёт проблемы производительности (N+1 на бэке, гигантские payload'ы от слишком жадных клиентов). Для нового внешнего API REST + OpenAPI 3.x по-прежнему самый частый выбор.
Делегируй LLMКонвертация контрактов: «Преобразуй этот OpenAPI / .proto в набор тест-сценариев pytest + requests.»

Объяснение незнакомого протокола: вставь .proto или GraphQL schema → «Что делает этот сервис? Какие методы есть, какие риски при тестировании?»

Генерация GraphQL-запросов: «По этой schema сгенерируй: 1) минимальный валидный query, 2) запрос со всеми полями, 3) запрос с превышением сложности (для тестирования rate limit).»
Каверзный вопрос«Может ли GraphQL вернуть HTTP 200 при ошибке?» — Да, и это норма. GraphQL обычно возвращает 200 даже если внутри есть ошибки — они описываются в поле "errors" в теле ответа:
{"data": null, "errors": [{"message": "..."}]}
Это сбивает с толку — проверять надо не только HTTP-код, но и наличие errors в ответе. На собесе можешь упомянуть как пример «GraphQL ломает привычные REST-паттерны».
08 WebSockets · SSE real-time
L1 · Основы

Обычный HTTP — это «клиент спросил → сервер ответил». Но иногда нужно, чтобы сервер сам отправлял данные клиенту (чаты, котировки, нотификации). Для этого есть два основных подхода:

SSE (Server-Sent Events)
Сервер открывает поток и шлёт сообщения клиенту в одну сторону. Поверх обычного HTTP. Авто-reconnect из коробки.
WebSocket
Постоянное двунаправленное соединение. После handshake клиент и сервер могут слать друг другу что угодно когда угодно.
АналогияSSE — как tail -f logfile: сервер пишет в поток, ты читаешь по мере появления, обратной связи нет. WebSocket — обычный TCP-сокет: full-duplex, оба могут писать в любой момент.
L2 · Детали

SSE — поток сообщений по HTTP:

# Запрос GET /events Accept: text/event-stream # Ответ — поток, не закрывается HTTP/1.1 200 OK Content-Type: text/event-stream data: {"order_id": 789, "status": "paid"} data: {"order_id": 790, "status": "shipped"} event: alert data: {"level": "warning", "msg": "..."}
  • Однонаправленный (сервер → клиент)
  • Авто-reconnect: браузер сам переподключается, если разорвалось
  • Простой текстовый формат, легко отлаживать
  • Работает через прокси (это просто HTTP)
  • Где: ленты новостей, нотификации, прогресс долгих операций, streaming-ответы от LLM (ChatGPT, Claude используют SSE)

WebSocket — двунаправленное соединение:

# Handshake — обычный HTTP с Upgrade GET /chat HTTP/1.1 Host: chat.com Upgrade: websocket Connection: Upgrade Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ== # Ответ сервера HTTP/1.1 101 Switching Protocols Upgrade: websocket Connection: Upgrade # После этого — двоичный или текстовый обмен # через ws:// (или wss:// — TLS)
  • Двунаправленный, постоянное соединение
  • Любой формат сообщений (текст, JSON, бинарь)
  • Низкий overhead — после handshake нет повторных HTTP-заголовков
  • Где: чаты, многопользовательские игры, биржевые тикеры, collaborative editing (Figma, Google Docs)

Сравнительная таблица:

L3 · Senior

Что тестировать в real-time соединениях:

  • Установление соединения: успешный handshake, отказ при невалидной авторизации
  • Доставка сообщений: все сообщения дошли, в правильном порядке, без дубликатов
  • Разрыв и переподключение: сервер падает / сеть прерывается → клиент переподключается с exponential backoff, не уходит в бесконечный цикл
  • Heartbeat / ping-pong: соединение не убивается прокси из-за idle timeout
  • Нагрузка: 1000–10000 одновременных соединений на сервер
  • Безопасность: авторизация при handshake (не в URL!), ограничение скорости сообщений, защита от flood
  • Очистка состояния: при дисконнекте — освобождение ресурсов, обновление статуса «online/offline»
Типичный багАутентификация через токен в URL: wss://chat.com/?token=eyJ.... Проблема — токен попадает в логи прокси, истории браузера, server access logs. Правильно: передать через subprotocol-заголовок при handshake или через первое сообщение после подключения.

Инструменты для тестирования:

  • Postman — поддерживает WebSocket с 2022
  • websocat — CLI «curl для WebSocket»
  • Python: websockets — для автотестов
  • k6 — нагрузочное тестирование WebSocket
  • wscat — простой Node.js клиент
В реальном миреSSE неожиданно получил вторую жизнь благодаря LLM-стримингу (ChatGPT, Claude шлют ответы по символам именно через SSE). WebSocket остаётся стандартом для чатов и интерактивных приложений. Часто на верхнем уровне используется не «голый» WebSocket, а абстракции: Socket.IO, SignalR (.NET), Phoenix Channels (Elixir). Они дают надёжность, переподключение, комнаты — но добавляют свой протокол поверх.
Делегируй LLMАнализ дампа WebSocket-сессии: вставь логи обмена → «Найди аномалии: пропущенные сообщения, дубликаты, неправильный порядок.»

Генерация нагрузочного сценария: «Напиши k6-скрипт для тестирования WebSocket-чата: 500 пользователей одновременно, каждый шлёт сообщение раз в 5 секунд, проверка получения от других.»
Каверзный вопрос«Когда использовать SSE вместо WebSocket?» — Когда нужен только односторонний поток сервер→клиент. Преимущества SSE: 1) проще, 2) поверх обычного HTTP — работает через любые прокси / CDN, 3) авто-reconnect из коробки, 4) текстовый формат — легко дебажить. WebSocket нужен, если клиент тоже должен слать данные постоянно (чат, игра). Для нотификаций или real-time лент — SSE достаточно и удобнее.
09 Брокеры сообщений: Kafka, RabbitMQ, PubSub async messaging
L1 · Основы

REST, GraphQL, gRPC — это синхронная связь: запросил → подождал → получил ответ. Но часто сервисам не нужно ждать друг друга. Брокер сообщений — это посредник, через который сервисы обмениваются сообщениями асинхронно: один кладёт сообщение, другой забирает, когда сможет.

  • Producer — кто отправляет сообщение
  • Broker — посредник, который хранит и доставляет
  • Consumer — кто получает и обрабатывает
АналогияСинхронный вызов (REST) — телефонный звонок: ждёшь, пока возьмут трубку и ответят. Брокер сообщений — это очередь задач в фоне: положил задачу в очередь (как queue.put() между потоками), и пошёл дальше. Worker разберёт её, когда дойдут руки. Отправитель не блокируется.
Зачем1) Развязка сервисов — producer не знает и не ждёт consumer. 2) Сглаживание нагрузки — пик запросов кладётся в очередь, обрабатывается ровно. 3) Надёжность — если consumer упал, сообщения не теряются, ждут в брокере. 4) Масштабирование — несколько consumer'ов разбирают очередь параллельно.
L2 · Детали

Три самых частых брокера. Они решают похожую задачу, но устроены по-разному:

RabbitMQ
Классическая очередь сообщений (message queue). Брокер активно раздаёт сообщения consumer'ам и удаляет их после подтверждения (ack). Гибкая маршрутизация через exchange. Протокол AMQP.
Kafka
Лог событий (event log), а не очередь. Сообщения не удаляются после прочтения — лежат в топике N дней. Consumer сам читает с нужной позиции (offset). Заточен под огромные объёмы и повторное чтение.
Google Pub/Sub
Облачный managed-брокер по модели publish/subscribe. Не надо поднимать инфраструктуру — это сервис в GCP. Похож на Kafka идеологически, но полностью управляемый. Аналоги: AWS SNS/SQS, Azure Service Bus.

Модели доставки — общие для всех брокеров:

  • Point-to-point (очередь): одно сообщение получает один consumer. Несколько consumer'ов делят нагрузку
  • Publish/Subscribe: одно сообщение получают все подписчики. Например, событие «заказ оплачен» слушают и склад, и аналитика, и нотификации

Гарантии доставки — важно для тестировщика:

  • At most once — максимум один раз (может потеряться). Быстро, но ненадёжно
  • At least once — минимум один раз (могут быть дубли). Самый частый режим
  • Exactly once — ровно один раз. Идеал, но сложно и дорого (Kafka умеет с оговорками)
L3 · Senior

Что тестировать в системах с брокерами:

  • Дубликаты (at least once): брокер может доставить сообщение дважды. Consumer должен быть идемпотентным — обработка дубля не должна списать деньги повторно
  • Порядок сообщений: пришли ли события в правильном порядке? В Kafka порядок гарантирован только внутри партиции — события одного заказа должны идти в одну партицию (по ключу)
  • Потеря сообщений: consumer упал во время обработки — сообщение вернулось в очередь или потерялось?
  • Dead Letter Queue (DLQ): сообщение, которое не удалось обработать N раз, уходит в отдельную очередь. Проверить, что «ядовитые» сообщения не блокируют очередь навечно
  • Consumer lag: consumer не успевает за producer'ом — очередь растёт. Метрика, за которой следят в проде
  • Retry и backoff: при ошибке обработки — повтор с задержкой, а не бесконечный цикл
Eventual consistencyГлавная боль тестирования асинхронных систем: после действия результат появляется не сразу. Положил заказ → событие ушло в Kafka → склад обработал через 2 секунды. Автотест, который сразу проверяет статус склада, упадёт. Решение: polling с таймаутом («жди до 10 секунд, пока статус не станет shipped»), а не мгновенная проверка. Это причина большей части flaky-тестов в микросервисах.

Как тестировать практически:

  • Поднять брокер в Docker (Testcontainers) — реальный Kafka/RabbitMQ в тесте
  • Producer: проверить, что сообщение ушло в топик с правильным телом и ключом
  • Consumer: положить сообщение в топик → проверить, что обработалось правильно
  • End-to-end: REST-запрос → событие в брокере → изменение в другом сервисе (с polling)
В реальном миреГрубо: RabbitMQ — задачи и команды с гибкой маршрутизацией, Kafka — потоки событий большого объёма и повторное чтение, Pub/Sub / SNS+SQS — managed-вариант в облаке. Но для QA важнее не «какой брокер», а тестовые риски: дубли, порядок, потеря сообщений, retry и eventual consistency — именно из-за них ломаются тесты.
Делегируй LLMГенерация теста с polling: «Напиши pytest-тест: POST создаёт заказ, событие уходит в Kafka, статус склада обновляется асинхронно. Используй polling с таймаутом 10 секунд вместо time.sleep.»

Анализ проблемы: «Consumer обрабатывает одно сообщение дважды, списывает деньги повторно. Как сделать обработку идемпотентной? Предложи варианты.»
Каверзный вопрос«Чем Kafka отличается от RabbitMQ в двух словах?» — RabbitMQ это очередь: сообщение прочитали — оно исчезло. Kafka это лог: сообщение прочитали — оно осталось, можно перечитать с любого места. Отсюда всё остальное: Kafka лучше для аналитики и больших потоков, RabbitMQ — для задач и команд с гибкой маршрутизацией.
10 Cookies, Session, localStorage, sessionStorage client state
L1 · Основы

HTTP сам по себе stateless — между запросами сервер ничего не помнит о клиенте. Чтобы хранить состояние (логин, корзину, настройки), есть четыре места:

Cookie
Маленький текстовый файлик, который сервер просит браузер хранить. Автоматически отправляется с каждым запросом на этот домен в заголовке Cookie:. Размер до ~4 КБ.
Session (серверная)
На клиенте — только session_id (обычно в cookie). Сами данные — на сервере (в Redis / БД). Stateless для клиента, stateful для сервера.
localStorage
Хранилище в браузере, до ~5–10 МБ. На сервер не отправляется автоматически. Живёт между перезапусками браузера, пока вручную не очистить.
sessionStorage
То же, что localStorage, но живёт только пока открыта вкладка. Закрыл вкладку — данные пропали. Другая вкладка с тем же сайтом видит свои данные.
АналогияCookie = переменные окружения для процесса — подставляются автоматически в каждый запрос. Session = session_id это PID процесса, а сами данные — в shared memory на сервере. localStorage = файл на диске для домена, переживает перезагрузку. sessionStorage = переменная в памяти процесса — умирает с процессом (вкладкой).
L2 · Детали

Cookie — как сервер устанавливает и как браузер отправляет:

Что и как тестировать в cookie-сценариях — частый Senior-вопрос:

  • Auth-token в cookie: HttpOnly стоит? Secure стоит на HTTPS-сайте? SameSite не None?
  • Logout: cookie действительно очищена? Token инвалидирован на сервере?
  • Multi-domain: cookie с Domain=.shop.com доступна на cart.shop.com? Не утекает на partner.com?

В реальном миреВокруг third-party cookies много лет идёт сворачивание: Safari и Firefox их блокируют по умолчанию. В новых проектах закладывают: 1) first-party cookies (на своём домене), 2) server-side tracking вместо клиентских пикселей, 3) CHIPS (Cookies Having Independent Partitioned State) для встраиваемых виджетов. Третий вариант — IndexedDB: полноценная NoSQL-БД в браузере, до сотен МБ, для PWA и offline-приложений.

Каверзный вопрос«Я залогинился, токен в HttpOnly cookie. Как фронтенду понять, что я залогинен?» — Фронтенд не видит сам токен, но: 1) сервер возвращает endpoint GET /me — 200 если залогинен, 401 если нет, 2) при login сервер дополнительно ставит не-HttpOnly cookie is_logged_in=true для UI-флага (не для auth!), 3) клиент кэширует флаг в памяти после login-ответа. Главное — флаг «залогинен» != сам токен.
11 Аутентификация: Basic, Bearer, API Key authentication
L1 · Основы

Аутентификация (authentication, AuthN) — «кто ты?». Сервер проверяет, что клиент тот, за кого себя выдаёт. Не путать с авторизацией («что тебе можно?») — это следующий шаг.

В REST API три популярные схемы:

Basic Auth
Authorization: Basic <base64(login:password)>. Простейшая, но логин/пароль шлются с каждым запросом. Только для HTTPS и internal API.
Bearer Token
Authorization: Bearer <token>. Токен заранее выдан сервером (по логин/паролю или OAuth). Не нужно слать пароль каждый раз.
API Key
Уникальный ключ для клиента. Передаётся в заголовке, query или теле. Часто для machine-to-machine, без пользователей.
АналогияBasic = SSH с паролем — пароль шлётся с каждой командой. Bearer = SSH-ключ — один раз настроил, дальше работаешь по ключу. API Key = personal access token в GitHub — выдан конкретному клиенту/сервису, можно отозвать отдельно.
L2 · Детали

Basic Auth — самая простая схема, описана в RFC 7617:

# Логин:пароль = "admin:secret123" # base64("admin:secret123") = "YWRtaW46c2VjcmV0MTIz" GET /api/users HTTP/1.1 Authorization: Basic YWRtaW46c2VjcmV0MTIz
  • Base64 — это НЕ шифрование, а кодирование. Любой за секунду декодирует
  • Без HTTPS — пароль уйдёт открытым текстом
  • Браузер кеширует Basic Auth — выйти можно только закрытием браузера или специальным трюком
  • Если сервер ждёт Basic, но клиент не передал — отвечает 401 + заголовок WWW-Authenticate: Basic realm="..."
  • Где встречается: internal admin-панели, легаси, базовая защита dev-окружений

Bearer Token — главная схема современных API:

# 1. Логинимся — получаем токен POST /api/auth/login {"login": "admin", "password": "secret123"} # Ответ: {"access_token": "eyJhbGciOiJIUzI1NiIs...", "token_type": "Bearer"} # 2. Дальше — ходим с токеном GET /api/users HTTP/1.1 Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
  • Токен — это «предъявитель этого билета — тот, кто его получил». Любой с токеном может ходить от твоего имени
  • Токены бывают opaque (просто строка, сервер проверяет по БД) или JWT (самодостаточная подписанная структура — см. раздел 10)
  • Обычно — пара: короткоживущий access_token + долгоживущий refresh_token для обновления

API Key — три способа передачи:

ГдеПримерКогда
В заголовкеX-API-Key: abc123Самый правильный способ. Не попадает в логи URL
В query?api_key=abc123Удобно для curl, но попадает в логи серверов и proxy
В теле{"api_key": "abc123", ...}Только для POST. Редкий вариант
L3 · Senior

Менее популярные, но важные схемы:

mTLS (mutual TLS)
И сервер, и клиент предъявляют сертификаты при handshake. Сервер не отдаст ничего без валидного клиентского сертификата. Используется в banking, internal microservices в zero-trust сетях, service mesh (Istio, Linkerd).
HMAC Signature
Клиент подписывает запрос секретным ключом (включая body, timestamp, метод, URL). Сервер проверяет подпись. Так работают AWS API, российские СБИС / Эльба. Защита от replay-атак за счёт timestamp.

Что тестировать в аутентификации:

  • Валидный логин — выдаёт ли токен с правильным TTL
  • Невалидный пароль — 401, без подсказки «логин верный, пароль нет»
  • Несуществующий логин — тот же 401, идентичное время ответа (защита от timing attack)
  • Brute-force — после N неуспешных попыток: rate limit / capture / временная блокировка аккаунта
  • Истекший токен — 401, клиент идёт за refresh
  • Отозванный токен (logout / смена пароля) — старый токен не работает на других устройствах
  • Tampered token (изменены символы) — 401, без утечки info об ошибке валидации
  • Замена своего токена на чужой — 403 при доступе к чужим ресурсам
  • Token в логах сервера / в URL / в Sentry-эвентах — недопустимо, должен быть отфильтрован
Типичные баги · Утечка в логах: nginx access.log пишет URL с ?api_key=...
· Слишком долгий TTL: access-token на месяц — украденный долго опасен
· Без rate limit: brute-force перебирает 10000 паролей за час
· Одинаковая ошибка при разных причинах: «invalid credentials» = и неверный пароль, и истёкший токен — клиенту непонятно, что делать
В реальном миреВ современных API дефолт — Bearer JWT для веб/мобайл + API Key в header для machine-to-machine. Basic Auth почти не используется (только internal). Для критичных внутренних коммуникаций — mTLS. OAuth 2.0 (раздел 10) — отдельная история, это не схема аутентификации, а протокол получения токенов.
Делегируй LLMГенерация тестов аутентификации: «Эндпоинт принимает JWT в Authorization: Bearer. Сгенерируй полный набор негативных тест-кейсов: без токена, истёкший, с изменённой signature, с alg:none, чужой токен, неверный issuer.»

Декодирование Basic Auth: «В логе видел Authorization: Basic YWRtaW46c2VjcmV0MTIz — что это значит и какой пароль?» (Спойлер: admin:secret123)
Каверзный вопрос«В чём разница между 401 и 403 в контексте аутентификации?» — 401 «не аутентифицирован»: токен не предъявлен, истёк, невалиден. Клиент идёт логиниться / обновлять токен. 403 «аутентифицирован, но не имеет прав»: токен валиден, ты — это ты, но к этому ресурсу тебе нельзя. На login делать не имеет смысла, результат тот же.
12 Авторизация: JWT, OAuth 2.0 authorization
L1 · Основы

Авторизация (authorization, AuthZ) — «что тебе можно?». После того, как сервер узнал, кто ты (аутентификация), он решает, к каким ресурсам и операциям ты имеешь доступ.

Две главные технологии в современных API:

JWT (JSON Web Token)
Самодостаточный подписанный токен. Внутри уже лежит информация о пользователе и его правах — серверу не нужно ходить в БД, чтобы проверить, кто это. Формат header.payload.signature.
OAuth 2.0
Не схема аутентификации, а протокол делегирования доступа. «Я разрешаю приложению X читать мою почту в Gmail, не давая ему мой пароль». Стандарт за «войти через Google / GitHub / VK».
АналогияJWT = self-signed сертификат с данными внутри: «предъявителю по фамилии Иванов разрешено читать /orders до 18:00». Никуда не нужно звонить — проверил подпись, прочитал данные. OAuth = протокол выдачи временных пропусков: вместо того чтобы дать чужому приложению ключи от квартиры, выдаёшь ограниченный пропуск «только в гостиную, только сегодня, могу отозвать в любой момент».
L2 · Детали

JWT — структура. Три части, разделённые точками: eyJhbGc....eyJzdWI....SflKxw...

// HEADER (base64-encoded JSON)
{ "alg": "HS256", "typ": "JWT" }
// PAYLOAD (base64-encoded JSON)
{ "sub": "user_42", "name": "Иван", "role": "admin", "iat": 1735689600, "exp": 1735693200 }
// SIGNATURE = HMAC-SHA256( // base64(header) + "." + base64(payload), // secret // )
  • Header — алгоритм подписи и тип
  • Payload — claims: данные о пользователе, правах, времени
  • Signature — подпись от header + payload секретным ключом. Без неё JWT — это просто открытый base64

Стандартные claims (RFC 7519):

ClaimСмысл
issIssuer — кто выдал токен
subSubject — кому выдан (user_id)
audAudience — для какого сервиса предназначен
iatIssued At — когда выдан (Unix timestamp)
expExpiration — когда истекает
nbfNot Before — раньше этого времени невалиден
jtiJWT ID — уникальный id токена (для blacklist)

OAuth 2.0 — четыре роли:

  • Resource Owner — пользователь (ты)
  • Client — приложение, которое хочет доступ (Trello)
  • Authorization Server — выдаёт токены (Google OAuth)
  • Resource Server — где лежат данные (Gmail API)

Главный flow — Authorization Code with PKCE (для веб/мобайл):

  1. Trello редиректит юзера на Google: «я хочу access к календарю»
  2. Google показывает экран consent: «Trello просит read access. Разрешить?»
  3. Юзер соглашается → Google редиректит обратно в Trello с временным code
  4. Trello обменивает code на access_token + refresh_token (запрос идёт между серверами, не через браузер)
  5. Trello ходит в Gmail API с access_token

JWT vs Session — когда что выбрать:

JWTSession (server)

Проверкаsignature, без БДлезем в Redis по session_id

Уязвимости JWT:

  • alg: none — атакующий меняет header на {"alg":"none"} и удаляет signature. Плохо написанная библиотека «верит» и пропускает. Защита: явно валидировать ожидаемый алгоритм
  • Key confusion (RS256 → HS256): сервер ждёт RS256 (асимметричный), атакующий шлёт HS256, используя публичный ключ как HMAC-секрет. Защита: жёсткая проверка алгоритма
  • Слабый секрет: HMAC-ключ "secret" или "123456" подбирается за минуты. Защита: длинные random-секреты (≥32 байт), ротация ключей
  • Чувствительные данные в payload: JWT — НЕ шифрование. password, PAN карты в payload — катастрофа
  • Долгий TTL без revocation: украденный токен на месяц = месяц доступа. Защита: короткий access (15 мин) + refresh с возможностью отзыва

RBAC vs ABAC vs ReBAC — модели контроля доступа:

RBAC (Role-Based)
Роли: admin, user, guest. У каждой — список разрешений. Простая, понятная. Минус: ролей становится много (admin_payments, admin_users, admin_readonly...).
ABAC (Attribute-Based)
Решение по атрибутам: «user может читать orders, если order.owner == user.id и время рабочее». Гибче, сложнее в реализации. Используется в AWS IAM.
ReBAC (Relationship-Based)
Доступ через связи: «user может редактировать doc, если он owner или editor этого doc, или owner родительской папки». Google Drive, Notion, Figma. Базовый инструмент — Google Zanzibar.

Что тестировать в авторизации:

  • Vertical privilege escalation: user пытается выполнить admin-операцию → 403
  • Horizontal privilege escalation (IDOR): user A пытается прочитать данные user B через GET /orders/789 → 403 или 404
  • Force browsing: попытка попасть на admin-URL без admin-роли
  • Token replay: использовать токен после logout — не должно работать
  • Token expiration: что происходит точно в момент exp, чуть раньше, чуть позже
  • Refresh flow: можно ли использовать refresh_token дважды (rotation)? Старый должен инвалидироваться
  • Scope в OAuth: если токен выдан с scope=read, попытка POST → 403
В реальном миреСовременный стек авторизации в типичном продукте: OAuth 2.0 + PKCE для веб/мобайл, OIDC для «войти через X», JWT как формат access-токена, refresh-token rotation для безопасного обновления. Для зрелых продуктов с тонкими правами — OPA (Open Policy Agent) или SpiceDB вместо ручного RBAC в коде. Authentication часто делегируют SaaS: Auth0, Clerk, Keycloak, WorkOS — не пишут свой login с нуля.
Делегируй LLMДекодирование JWT: вставь токен → «Расшифруй header и payload, проверь срок действия, оцени риски: чувствительные данные в payload? Слабый alg?»

Аудит OAuth-flow: вставь Postman-коллекцию или последовательность запросов → «Найди проблемы: hardcoded client_secret, отсутствие state-параметра против CSRF, redirect_uri без валидации.»

Генерация тестов: «Сгенерируй полный набор тестов для авторизации с RBAC ролями user/manager/admin. Включи vertical/horizontal escalation, IDOR, force browsing.»
Каверзный вопрос«Как инвалидировать JWT до его exp?» — В чистом виде — никак, в этом боль JWT vs session. Варианты: 1) blacklist по jti — но это уже state на сервере, теряем главное преимущество JWT, 2) версия ключа в payload — при logout инкрементируем версию у юзера, старые токены становятся невалидны, 3) короткий TTL + refresh-token rotation — самый используемый подход: access живёт 15 минут, refresh можно отозвать в БД.
13 Тестирование REST API на практике api testing
L1 · Основы

Тестирование API — это проверка, что на запрос приходит правильный ответ. В каждом ответе проверяем пять вещей:

  • Статус-код — 200 / 201 / 404 / тот, что ожидаем
  • Тело — правильные данные, правильные значения полей
  • Схема — структура и типы полей соответствуют контракту
  • Заголовки — Content-Type, Location, кэширование
  • Время ответа — укладывается в SLA (например, <500мс)

В Python основной инструмент — библиотека requests:

import requests r = requests.get( 'https://api.shop.com/users/42', headers={'Authorization': 'Bearer token'}, timeout=5 ) assert r.status_code == 200 data = r.json() assert data['id'] == 42
АналогияAPI-тест как unit-тест функции, только функция живёт на сервере. Вход — запрос (метод, URL, заголовки, тело), выход — ответ (статус, тело, заголовки). Принцип тот же: подал известный вход → проверил ожидаемый выход.
L2 · Детали

requests — что нужно знать для тестов:

# Session — переиспользует соединение и заголовки session = requests.Session() session.headers.update({'Authorization': 'Bearer token'}) # Retry на нестабильных эндпоинтах from requests.adapters import HTTPAdapter from urllib3.util import Retry retries = Retry(total=3, backoff_factor=0.5, status_forcelist=[502, 503, 504]) session.mount('https://', HTTPAdapter(max_retries=retries)) # Timeout — ВСЕГДА указывай, иначе тест может висеть вечно r = session.post(url, json=payload, timeout=10)

Структура автотеста — паттерн AAA (Arrange-Act-Assert):

def test_create_order_returns_201(): # Arrange — подготовка данных payload = {'product_id': 42, 'quantity': 2} # Act — выполняем действие r = session.post('/api/orders', json=payload, timeout=5) # Assert — проверяем результат assert r.status_code == 201 assert r.json()['status'] == 'pending' assert 'Location' in r.headers

  • Изоляция: тесты не должны зависеть друг от друга. Каждый создаёт свои данные
  • Очистка: упавший тест не должен оставлять данные, которые сломают следующий прогон
  • Уникальность: генерируй уникальные email/имена (user_{uuid}@test.com), чтобы не было конфликтов при параллельном запуске

Дебаг 500, когда UI показывает только «Something went wrong». Классическая ситуация — клиент прячет детали, надо докопаться до причины:

  1. Смотри реальный ответ API, а не UI: DevTools Network или прокси (Charles) — там часто есть тело ошибки с деталями
  2. Возьми X-Request-ID из заголовков ответа — по нему найдёшь запрос в логах сервиса
  3. Логи сервиса в централизованной системе (Kibana / Datadog / Grafana Loki) — поиск по request-id даёт стектрейс, где именно упало
  4. Трейсинг (Jaeger) — если микросервисы, видно, на каком сервисе в цепочке сломалось
  5. Sentry — необработанные исключения с полным контекстом часто уже там
  6. Воспроизведи напрямую: тот же запрос через curl/Postman, поиграй с входными данными — локализуй, что именно вызывает 500
Суть500 без деталей в UI — это не «нет информации», а «информация в другом месте». Цепочка: ответ API → request-id → логи → трейс → стектрейс. Senior знает этот путь и не гадает.
Каверзный вопрос«Как проверить, что API не сломается при изменении бэка?» — Не только функциональными тестами. Нужны: 1) проверка схемы (JSON Schema / OpenAPI validation) — ловит изменения структуры, 2) контрактные тесты (Pact) — ловят breaking changes между сервисами до деплоя, 3) schemathesis — генерит тысячи запросов по спеке. 14 Контрактное тестирование и mocking contracts & mocks
L1 · Основы

Часто тестируемый сервис зависит от других: платёжного шлюза, сервиса доставки, внешнего API. Гонять их по-настоящему в тестах — медленно, дорого и нестабильно. Решение — mock: подменяем зависимость на заглушку с заранее известным ответом.

# responses — мокаем HTTP-вызов на уровне Python import responses @responses.activate def test_order_with_mocked_payment(): responses.add( responses.POST, 'https://payments.com/charge', json={'status': 'success'}, status=200 ) # Теперь наш код, вызывая платёжку, получит этот ответ result = create_order_and_pay(...) assert result.paid is True
АналогияMock как stub-функция в unit-тесте: вместо реального обращения к БД или сети подставляешь def get_user(): return {'id': 42}. Тестируешь свою логику в изоляции, не завися от того, работает ли сейчас внешний сервис.
L2 · Детали

Mock / Stub / Fake / Spy / Dummy — терминология, которую часто путают:

ТипЧто делает
DummyПустышка для заполнения параметра. Не используется, просто чтоб код не падал
StubВозвращает заранее заданные ответы. «На любой вызов верни 200 OK»
MockStub + проверяет, как его вызвали: с какими аргументами, сколько раз
SpyОборачивает реальный объект и записывает вызовы. Реальная логика работает, но мы видим, что вызвалось
FakeРабочая, но упрощённая реализация. In-memory словарь вместо реальной БД

Уровни мокинга API:

На уровне кода (Python)
responses, requests-mock — перехватывают HTTP-вызовы внутри процесса. Быстро, для unit/integration тестов.
Мок-сервер (отдельный процесс)
WireMock, mountebank, Prism — поднимают настоящий HTTP-сервер, отдающий заданные ответы. Для E2E, когда нужно мокать внешний сервис целиком.

Валидация схемы (JSON Schema) — проверяем, что структура ответа не сломалась:

from jsonschema import validate schema = { 'type': 'object', 'properties': { 'id': {'type': 'integer'}, 'name': {'type': 'string'}, 'email': {'type': 'string', 'format': 'email'} }, 'required': ['id', 'name'] } validate(instance=response.json(), schema=schema)

Схему можно не писать руками, а взять из OpenAPI / Swagger — единого описания API. Тогда тесты проверяют ответ против официального контракта.

L3 · Senior

Контрактное тестирование (Pact) — решает проблему «сервис A обновился и сломал сервис B». Подход consumer-driven contracts:

  1. Consumer (сервис B) описывает, что он ожидает от Provider (сервиса A): «на GET /users/42 жду поле name типа string»
  2. Из этого генерируется pact-файл (контракт)
  3. Provider прогоняется против pact-файла на своём CI
  4. Если A убрал поле name — его тесты краснеют до деплоя, не дожидаясь, пока B сломается в проде
Зачем это в микросервисахПолноценные integration-тесты между всеми сервисами — медленно и хрупко. Контрактные тесты дают 80% уверенности за 20% стоимости: каждый сервис тестируется изолированно, но контракт между ними гарантирован.

schemathesis — property-based тестирование API. Берёт OpenAPI-спеку и сам генерирует тысячи запросов, ища: 500-е ошибки, ответы, не соответствующие схеме, падения на граничных данных. Находит то, что вручную не догадаешься проверить.

Backward compatibility — как тестировать. API меняется, но старые клиенты (мобильные приложения, партнёры) не обновляются мгновенно. Изменение не должно их ломать:

  • Что НЕ ломает (backward compatible): добавить новое опциональное поле, добавить новый эндпоинт, добавить новое значение enum (если клиент его игнорирует)
  • Что ЛОМАЕТ (breaking change): удалить/переименовать поле, сделать опциональное поле обязательным, изменить тип поля, изменить структуру ответа, ужесточить валидацию
  • Как проверять: прогнать тесты старого клиента против новой версии API. Сравнить старую и новую OpenAPI-спеку диффером (есть инструменты вроде oasdiff). Контрактные тесты (Pact) краснеют на breaking changes автоматически
  • Если breaking change неизбежен — версионирование (/v2/), параллельная поддержка старой версии, период депрекации с уведомлением клиентов
Когда мок вредитЗамокать можно так, что тесты зелёные, а в проде инцидент. Риски: 1) мок устарел — реальный API изменился, а мок отдаёт старый формат, 2) замокали то, что надо было тестировать — мок собственной БД скрывает реальные баги SQL, 3) over-mocking — тест проверяет моки, а не систему. Правило: мокай чужое и дорогое, не мокай своё ядро.
В реальном миреСовременный стек проверки контрактов: OpenAPI 3 как единый источник правды → авто-валидация ответов против него → Pact для межсервисных контрактов в CI → schemathesis для фаззинга по спеке. Мок-серверы (Prism, WireMock) часто поднимают прямо из OpenAPI-спеки — фронтенд начинает работать против мока ещё до того, как бэкенд написан (parallel development).
Делегируй LLMГенерация JSON Schema: вставь пример ответа API → «Сгенерируй JSON Schema для валидации этого ответа, с правильными типами и required-полями.»

Генерация моков: «По этой OpenAPI-спеке сгенерируй WireMock-стабы для всех эндпоинтов с реалистичными ответами.»

Разбор контракта: вставь два pact-файла → «Найди breaking changes между этими версиями контракта.»
Каверзный вопрос«В чём разница между mock и stub?» — Stub просто возвращает заданный ответ — ты проверяешь состояние (что вернула твоя функция). Mock дополнительно проверяет взаимодействие — что внешний сервис был вызван правильно: с теми аргументами, столько раз. Грубо: stub отвечает на «что вернулось?», mock — на «правильно ли я вызвал зависимость?».
15 Инструменты: Postman, Charles, Swagger tooling
L1 · Основы

Три категории инструментов, которые QA по API использует ежедневно:

Postman
Клиент для ручной работы с API: отправить запрос, посмотреть ответ, собрать коллекцию запросов, написать проверки. Самый популярный.
Charles / прокси
Перехват и подмена трафика между приложением и сервером. Видишь, что реально уходит по сети, и можешь это изменить на лету.
Swagger / OpenAPI
Документация и спецификация API. Описывает все эндпоинты, параметры и ответы. Источник правды о контракте.
АналогияPostman = REPL для API: пробуешь запросы интерактивно, как команды в консоли Python. Charles = breakpoint в дебаггере, но для сетевого трафика: останавливаешь запрос, смотришь и меняешь. Swagger = автодокументация типа docstrings, только для всего API сразу, с возможностью тут же дёрнуть метод.
L2 · Детали

Postman — больше, чем «отправить запрос»:

  • Коллекции — сохранённые наборы запросов, сгруппированные по фиче / сервису
  • Окружения (environments) — переменные {{base_url}}, {{token}} для dev / staging / prod. Один клик — и та же коллекция бьёт по другому стенду
  • Тесты на JS — после запроса можно проверить ответ: pm.test("status 200", () => pm.response.to.have.status(200))
  • Pre-request scripts — код перед запросом: получить токен, сгенерировать данные
  • Newman — CLI-runner: гоняет Postman-коллекции в CI/CD
  • Мок-серверы — Postman поднимает мок по твоим примерам ответов

Charles и подмена трафика — что это вообще. Прокси встаёт между приложением и сервером, весь трафик идёт через него:

Приложение → [ Charles / прокси ] → Сервер ↑ видит и может изменить запрос и ответ

Что это даёт QA:

ВозможностьЗачем
InspectПосмотреть реальные запросы/ответы — особенно у мобильного приложения, где DevTools нет
BreakpointsОстановить запрос/ответ, изменить и отпустить дальше
Map LocalПодменить ответ сервера локальным файлом
Map RemoteПеренаправить запрос на другой сервер (например, на staging)

Главный сценарий QAПротестировать, как клиент реагирует на разные ответы сервера, не трогая бэкенд. Подменил ответ на 500 — приложение показывает ошибку красиво? Подменил на пустой список — отрисовывается «ничего не найдено»? Замедлил сеть — крутится лоадер? Всё это без единой строчки изменений на сервере.

  • Swagger Editor — редактор спеки с превью
  • Swagger Codegen / OpenAPI Generator — генерация клиентов и серверных заглушек из спеки

HTTPS и подмена трафика. «Как прокси видит зашифрованный HTTPS?» Сам по себе — никак. Charles работает как man-in-the-middle: 1) на устройство ставится корневой сертификат Charles, 2) приложение доверяет ему, 3) Charles расшифровывает трафик, показывает/меняет, заново шифрует и шлёт серверу. Без установленного сертификата увидишь только зашифрованные байты.

Нюанс мобильного тестированияС Android 7+ приложения по умолчанию не доверяют пользовательским сертификатам — нужен networkSecurityConfig в debug-сборке или root. Многие банковские приложения используют certificate pinning — намеренно отказываются работать через любой прокси (защита от перехвата). Это не баг, это by design — на собесе можно упомянуть как пример осознанной защиты от MITM.

Каверзный вопрос«Как протестировать, что приложение правильно обрабатывает ошибку 500, если сервер её никогда не отдаёт?» — Через подмену трафика. Ставишь прокси (Charles/mitmproxy), настраиваешь Map Local или breakpoint, подменяешь реальный ответ на 500 → смотришь, как приложение реагирует. Не нужно ломать бэкенд или просить разработчиков — меняешь ответ на лету у себя.

Открытые вопросы вроде «Спроектируй стратегию тестирования для этого API», «Построй фреймворк API-тестов с нуля», «Как тестировать платёжный API?» проверяют структурированное мышление, а не заученный ответ.

L2 · Детали

Дизайн API-test фреймворка с нуля. Что назвать:

  • Слои: HTTP-обёртка → клиенты по доменам → тесты → фабрики данных (см. раздел 12)
  • Конфигурация: окружения (dev/staging/prod) через переменные, секреты не в коде
  • Авторизация: получение токена в фикстуре, переиспользование между тестами
  • Тест-данные: factory_boy для генерации, очистка через teardown
  • Отчётность: Allure с шагами, аттачами запросов/ответов
  • CI: запуск на каждый PR, параллелизация (pytest-xdist), флак-контроль
  • Контракты: проверка схемы + Pact в пайплайне

L3 · Senior

Кейс — «Протестируй платёжный API». Структурированный разбор:

Функциональность
Успешный платёж, отклонение (нет денег, fraud, expired card), частичный/полный refund, статусы транзакции.
Идемпотентность
Повтор с тем же Idempotency-Key → один платёж. С другим ключом → новый. Без ключа → как обрабатывается?
Асинхронность
Webhook о статусе: не пришёл / пришёл дважды (дедуп) / пришёл раньше нашего ответа. Eventual consistency — статус обновляется не сразу (polling в тестах).
Безопасность
PAN не в логах, PCI DSS, токенизация карты, чужой токен → 403, нельзя увидеть чужие транзакции (IDOR).
Надёжность
Сетевой обрыв на каждом шаге, retry, reconciliation с банком (сверка наших транзакций с реальными списаниями).
Контракт
Схема ответа, версионирование (платёжки часто меняют API), мок платёжного провайдера для тестов.
Что выделит Senior-ответНе перечисление «проверю успешный платёж и отклонение». А: «Я бы попросил OpenAPI-спеку платёжного провайдера и архитектурную схему. Замокал бы провайдера через WireMock для функциональных тестов, реальную интеграцию проверял бы отдельно в sandbox-окружении. Особое внимание — идемпотентности и обработке webhook'ов, потому что это про деньги пользователя при ретраях.»
В реальном миреSenior QA по API половину времени тратит не на написание тестов, а на: 1) ревью контрактов (OpenAPI) на этапе дизайна — поймать проблему до кода, 2) настройку мониторинга API в проде (latency, error rate по эндпоинтам в Grafana), 3) synthetic monitoring — боты дёргают критичные эндпоинты каждые N минут. Функциональные автотесты — это база, но реальная ценность Senior — в стратегии и в shift-left/shift-right.
Делегируй LLMДизайн стратегии: вставь OpenAPI-спеку → «Спроектируй стратегию тестирования: что на уровне schema, что contract, что integration, что E2E. Обоснуй приоритеты по риску.»

На собесе про AI: «Я генерирую базовые API-тесты из OpenAPI через Claude за минуты, затем добавляю доменные сценарии — идемпотентность, race conditions, бизнес-логику, которую LLM не знает. Освободившееся время трачу на стратегию и contract-тесты.»
Каверзный вопрос«У тебя API из 200 эндпоинтов и неделя времени. Что делаешь?» — Не «покрою все 200». А: 1) risk-based — выделю revenue-critical (платежи, auth, заказы), 2) на них — полные функциональные + contract тесты, 3) на остальные — быструю schema-validation из OpenAPI (дёшево, ловит структурные поломки), 4) schemathesis по всей спеке для авто-фаззинга, 5) договорюсь о приоритетах с командой. Покрыть осознанно важное лучше, чем поверхностно всё.
