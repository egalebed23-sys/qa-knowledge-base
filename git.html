01 Основы и базовые команды basics
L1 · Основы

Git — система контроля версий: хранит всю историю изменений кода, позволяет откатываться и работать команде параллельно. Базовый цикл из трёх команд:

git add file.py # добавить изменения в staging git commit -m "fix login" # зафиксировать снимок git push # отправить на сервер (origin)
АналогияCommit — как точка сохранения в игре: снимок всего проекта в этот момент, к которому можно вернуться. Staging area — это «корзина перед оформлением»: складываешь туда изменения, которые войдут в следующий снимок, остальное остаётся не зафиксированным.
L2 · Детали

Три состояния файла и где они живут:

  • Modified — изменён в рабочей директории, ещё не отмечен
  • Staged — добавлен в staging (index) через git add, войдёт в следующий коммит
  • Committed — зафиксирован в локальном репозитории

Команды, которые нужны каждый день:

КомандаЧто делает
git clone urlсклонировать репозиторий
git statusчто изменено, что в staging
git add .добавить все изменения в staging
git commit -m "..."зафиксировать снимок
git push / pullотправить / забрать с сервера
git logистория коммитов
git diffчто именно изменилось
L3 · Senior

Что такое коммит на самом деле: снимок состояния всех файлов + уникальный хеш (по умолчанию SHA-1; Git также поддерживает переход на SHA-256), автор, время, ссылка на родительский коммит. Git хранит снимки целиком (с дедупликацией неизменённых файлов), а не дельты.

HEAD — указатель на текущий коммит / ветку, где ты сейчас находишься. origin — стандартное имя удалённого репозитория.

В реальном миреQA-автоматизатору хватает уверенного владения базовым циклом + ветками + разрешением конфликтов + PR. Глубокое устройство Git (объектная модель, blob/tree/commit) знать не обязательно — спрашивают редко. Зато git log, git diff и умение разобраться, «какой коммит внёс регрессию», — ежедневный инструмент.
Каверзный вопрос«Чем git fetch отличается от git pull?» — fetch только скачивает изменения с сервера, но не трогает твою рабочую ветку. pull = fetch + merge (сразу вливает в текущую ветку). fetch безопаснее: сначала посмотреть, что прилетело, потом решить, как влить.
02 Ветки branches
L1 · Основы

Ветка (branch) — параллельная линия разработки. Делаешь фичу в своей ветке, не мешая main, потом вливаешь обратно.

git branch feature-login # создать ветку git switch feature-login # переключиться (или git checkout) git switch -c feature-login # создать и переключиться сразу
АналогияВетка — это просто указатель (ссылка) на коммит, как переменная, хранящая адрес. main и feature — две ссылки на разные коммиты. Переключение ветки — это смена того, на что смотрит HEAD. Создать ветку дёшево: это новая ссылка, а не копия всего проекта.
L2 · Детали

Типичный рабочий цикл с веткой:

git switch -c feature-x # 1. создал ветку от main # ... делаешь изменения, коммитишь ... git push -u origin feature-x # 2. запушил ветку на сервер # 3. открыл Pull Request на GitHub/GitLab # 4. ревью → merge в main git switch main git pull # 5. забрал обновлённый main git branch -d feature-x # 6. удалил локальную ветку

Полезное про ветки:

  • git branch — список локальных веток
  • git branch -a — включая удалённые
  • git branch -d name — удалить ветку
  • origin/main — remote-tracking ветка: где main на сервере по последним данным
L3 · Senior

Detached HEAD — состояние, когда ты переключился не на ветку, а на конкретный коммит (git checkout abc123). Коммиты в этом состоянии «висят в воздухе» — без ветки их легко потерять. Выход: создать ветку отсюда (git switch -c new-branch).

В реальном миреСовременный workflow: короткоживущие feature-ветки (1–2 дня) → PR → merge в main. Долгие ветки на недели повышают риск конфликтов и поздней интеграции: чем дольше живёт ветка, тем больнее потом конфликты при слиянии. Это прямая связь с trunk-based development (раздел 05).
Каверзный вопрос«Чем switch отличается от checkout?» — checkout перегружен: и ветки переключает, и файлы восстанавливает, и на коммиты прыгает. В новых версиях Git его разделили: switch — только для веток, restore — для восстановления файлов. Логика та же, но switch/restore понятнее и безопаснее.
03 Merge, Rebase, Fast-forward главная тема
L1 · Основы

Три способа объединить изменения из одной ветки в другую:

Merge
Объединяет две ветки, создавая merge-коммит. История сохраняется как есть, со всеми ответвлениями.
Rebase
Переносит коммиты твоей ветки поверх другой, как будто ты ответвился от неё только что. История линейная.
Fast-forward
Частный случай merge: если в main не было новых коммитов, Git просто перематывает указатель вперёд, без merge-коммита.
АналогияMerge — соединить две линии истории в одну точку, оставив развилку видимой. Rebase — переписать свою историю так, будто ты начал работу от самой свежей версии main: коммиты «перевешиваются» на новое основание. Fast-forward — просто сдвинуть закладку вперёд, потому что развилки и не было.
L2 · Детали

Merge — вливаем feature в main:

git switch main git merge feature-x # создаст merge-коммит (если были расхождения)

Rebase — переносим свои коммиты поверх свежего main:

git switch feature-x git rebase main # коммиты feature-x встают ПОВЕРХ main

Когда что использовать:

MergeRebase
Историясохраняется со всеми развилкамилинейная, чистая
Merge-коммитсоздаётсянет
Когдавливание feature в main, общие веткиподтянуть свежий main в свою личную ветку
Рискбезопаснопереписывает историю — опасно для общих веток

Fast-forward происходит автоматически, когда возможно. Заставить создать merge-коммит всегда: git merge --no-ff. Это частая настройка в командах — чтобы в истории было видно, где была feature-ветка.

L3 · Senior
Золотое правило rebaseНикогда не делай rebase ветки, которой пользуется кто-то ещё (например, общий main или ветка, которую кто-то уже забрал). Rebase переписывает историю — меняет хеши коммитов. Если ветка публичная, у других сломается история, и у команды возникнут расхождения в истории. Rebase — только для своих локальных веток до пуша.

Interactive rebase (git rebase -i) — переписать историю своих коммитов перед PR: объединить (squash), переименовать, переупорядочить, удалить. Часто перед мержем «схлопывают» пачку мелких коммитов в один осмысленный.

В реальном миреХоливар merge vs rebase решается командной договорённостью. Частый компромисс: rebase для подтягивания main в свою ветку (чистая локальная история) + squash-merge при вливании PR (одна фича = один коммит в main). Главное — единые правила в команде, а не «индивидуальные предпочтения без общей договорённости».
Каверзный вопрос«В чём разница merge и rebase?» — Merge объединяет ветки и сохраняет историю как было, создавая merge-коммит (видна развилка). Rebase переносит твои коммиты поверх другой ветки, делая историю линейной, но переписывая хеши. Merge безопасен для общих веток, rebase — только для личных. Простыми словами: merge честно показывает «как было», rebase делает «как будто было красиво».
04 Конфликты слияния conflicts
L1 · Основы

Конфликт возникает, когда две ветки изменили одно и то же место в файле, и Git не может решить сам, какую версию оставить. В файле появляются маркеры:

<<<<<<< HEAD твоя версия строки ======= версия из вливаемой ветки >>>>>>> feature-x

Нужно вручную выбрать правильный вариант (или скомбинировать), убрать маркеры и сохранить.

L2 · Детали

Как разрешить конфликт по шагам:

git merge feature-x # возник конфликт # 1. открываешь файл, видишь маркеры <<< === >>> # 2. редактируешь: оставляешь нужное, убираешь маркеры git add file.py # 3. помечаешь как разрешённый git commit # 4. завершаешь merge # (при rebase вместо commit — git rebase --continue)
  • IDE решают конфликты удобнее — VS Code, IntelliJ показывают обе версии side-by-side с кнопками «accept current / incoming / both»
  • git merge --abort / git rebase --abort — отменить и вернуться как было, если запутался
L3 · Senior

Как избегать конфликтов (лучше, чем уметь их разруливать):

  • Мелкие частые мержи — чаще подтягивай main в свою ветку, чем реже — тем больше расхождений накопится
  • Короткие ветки — фича на 1–2 дня конфликтует меньше, чем на 2 недели
  • Не форматируй чужой код заодно — массовый реформат = конфликт в каждой строке
  • Договорённости о структуре — если все правят один большой файл, конфликты неизбежны
Каверзный вопрос«Запутался в конфликте при rebase, история или рабочая директория оказались в неконсистентном состоянии. Как откатиться?» — git rebase --abort вернёт в состояние до rebase. Если уже завершил и нужно откатиться — git reflog покажет, где ты был до операции, и git reset --hard HEAD@{N} вернёт туда. reflog — спасательный круг (см. раздел 06).
05 Branching strategies workflows
L1 · Основы

Стратегия ветвления — договорённость команды о том, как организованы ветки и как код попадает в прод. Три популярных подхода:

Git Flow
Много веток: main, develop, feature, release, hotfix. Мощно, но сложно. Для релизов по расписанию.
GitHub Flow
Просто: main + короткие feature-ветки → PR → merge. Для continuous deployment.
Trunk-based
Всё в main, очень короткие ветки, незаконченное прячут за feature flags. Для CI/CD и частых релизов.
L2 · Детали
СтратегияВеткиКогда подходит
Git Flowmain + develop + feature/* + release/* + hotfix/*продукты с версиями и релизами по расписанию (десктоп, мобайл-стор)
GitHub Flowmain + feature-веткивеб-сервисы, частые деплои, небольшие команды
Trunk-basedmain + очень короткие веткизрелые команды с сильным CI/CD, релизы много раз в день

GitHub Flow (самый частый сегодня): ответвился от main → сделал фичу → PR → автотесты + ревью → merge → деплой. Просто и достаточно для большинства команд.

L3 · Senior
В реальном миреТренд — от сложного Git Flow к простому GitHub Flow / trunk-based. Git Flow с его develop и release-ветками всё чаще считают избыточным для веб-сервисов: он создан под эпоху редких релизов. Современный дефолт — короткие ветки от main + сильный CI + feature flags для незаконченного. Чем проще модель ветвления, тем меньше конфликтов и быстрее доставка (прямая связь с CI/CD и DORA-метриками).
Каверзный вопрос«Зачем нужны feature flags, если есть ветки?» — Долгая feature-ветка = сложное и рискованное слияние и поздняя интеграция. Feature flag позволяет влить незаконченный код в main сразу (спрятав за флагом), интегрировать его постоянно, а включить — когда готово. Это и есть суть trunk-based: интеграция через флаги, а не через долгоживущие ветки.
06 Полезные команды toolbox
L1 · Основы

Команды, которые выручают в типичных ситуациях:

git stash # отложить незакоммиченные изменения «в карман» git stash pop # вернуть их обратно git cherry-pick abc # взять один конкретный коммит в свою ветку
Аналогияstash — как буфер обмена для незавершённой работы: сложил изменения в карман, переключился на срочную задачу, потом достал обратно. cherry-pick — скопировать один коммит из другой ветки, как взять одну строку из чужого файла, а не весь файл.
Pull Request workflow — ежедневная рутинаДля QA это встречается чаще, чем rebase -i. Типичный цикл: 1) открыть PR (feature → main), 2) дождаться CI (тесты, линт — зелёные), 3) пройти code review, 4) исправить замечания (новые коммиты в ту же ветку), 5) squash & merge (слить в один коммит) или обычный merge, 6) удалить ветку после слияния. CI + review — это и есть quality gate перед попаданием в main.
restore — безопасная отмена измененийСовременная замена перегруженного checkout для файлов: git restore file.py — откатить изменения в файле к последнему коммиту; git restore --staged file.py — убрать файл из staging (не теряя правок). Понятнее, чем старый git checkout -- file.py.
L2 · Детали

stash — когда нужно срочно переключиться, но коммитить полуготовое не хочется:

git stash # временно сохранил незакоммиченные изменения, рабочая директория чистая git switch main # переключился, сделал срочное git switch feature-x git stash pop # вернул отложенное

cherry-pick — перенести конкретный коммит (например, хотфикс) в другую ветку без полного мержа.

reset vs revert — два способа «отменить», принципиально разные:

resetrevert
Что делаетдвигает указатель назад, убирает коммитысоздаёт новый коммит, отменяющий изменения
Историяпереписывает (коммиты исчезают)сохраняет (добавляет «обратный» коммит)
Безопасностьопасно для общих ветокбезопасно для общих веток
L3 · Senior

reset с режимами:

  • --soft — убрать коммит, изменения остаются в staging
  • --mixed (по умолчанию) — убрать коммит, изменения в рабочей директории
  • --hard — убрать коммит и изменения совсем (опасно, данные теряются)

git commit --amend — поправить последний коммит (сообщение или забытый файл). Тоже переписывает историю — не делай для запушенных коммитов.

reflog — спасательный кругgit reflog показывает историю всех перемещений HEAD — даже после reset --hard, неудачного rebase или «потерянных» коммитов. Почти всё, что кажется удалённым, можно восстановить: найди нужное состояние в reflog и git reset --hard HEAD@{N}. Запомни эту команду — она помогает восстановиться после ошибочных операций.
Правило безопасностиВсё, что переписывает историю (reset, rebase, amend, push --force), безопасно только для локальных коммитов, которые ещё никто не забрал. Для уже запушенного в общую ветку — только revert (новый коммит). Это то же золотое правило, что для rebase.
Как найти, какой коммит внёс регрессиюЧастая QA-задача — «тест/фича сломались, когда?»:
  • git log --oneline — список коммитов кратко, посмотреть что менялось недавно
  • git show <commit> — что именно изменил конкретный коммит
  • git diff main...feature — разница между ветками
  • git blame file.py — кто и в каком коммите менял каждую строку файла
  • git bisect (L3/optional) — бинарный поиск коммита с багом: помечаешь «хороший» и «плохой» коммиты, Git делит диапазон пополам, ты проверяешь — за log(N) шагов находишь виновный коммит
Force push безопасноЕсли после rebase своей ветки нужен форс-пуш — используй git push --force-with-lease, а не голый --force. Разница: --force-with-lease проверяет, что в удалённой ветке нет чужих новых коммитов, которые ты не видел, и откажется перезаписывать их. Голый --force затрёт чужую работу без предупреждения.
.gitignore и секретыВ .gitignore кладут то, что не должно попадать в репозиторий: .env и файлы с секретами, артефакты сборки, кэши, отчёты тестов, виртуальные окружения (.venv). Правило: секрет, попавший в Git, считается скомпрометированным — даже если удалить его следующим коммитом, он останется в истории.
Каверзный вопрос«Закоммитил пароль в репозиторий. Что делать?» — Не просто git rm в новом коммите — он останется в истории. Нужно: 1) немедленно отозвать/сменить сам секрет (он уже скомпрометирован), 2) вычистить из истории (git filter-repo — современный предпочтительный инструмент, BFG — альтернатива; устаревший filter-branch использовать не стоит), 3) форс-пуш. Но главное — секрет считается скомпрометированным в момент пуша, поэтому первым делом его меняют.
