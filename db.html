01 Реляционные БД, SQL vs NoSQL basics
L1 · Основы

Реляционная БД хранит данные в таблицах — строки (записи) и столбцы (поля). Таблицы связаны между собой через ключи. Язык общения с такой БД — SQL.

  • Таблица — например, users: каждая строка — пользователь, столбцы — id, name, email
  • Первичный ключ (PK) — уникальный идентификатор строки (id). Не повторяется, не пустой
  • Внешний ключ (FK) — ссылка на строку в другой таблице. В orders поле user_id ссылается на users.id
Аналогия и как запомнитьТаблица — как список объектов одного класса: строка = экземпляр, столбцы = поля. Первичный ключ — «кто я» (уникальный id строки). Внешний ключ — «на кого я ссылаюсь» (id строки в другой таблице). Пример: в t_employee поле loc_id — внешний ключ, ссылается на t_location.loc_id (первичный ключ там). Так сотрудник «знает» свою локацию, не дублируя её название.
Зачем FK нуженВнешний ключ не только связывает таблицы для JOIN, но и обеспечивает целостность данных: СУБД не даст вставить сотрудника с loc_id = 99, если такой локации нет, и (при настройке) не даст удалить локацию, пока на неё ссылаются сотрудники. Защита от «висячих» ссылок.
L2 · Детали

SQL vs NoSQL — два больших семейства баз:

SQL (реляционные)NoSQL
Структуратаблицы, жёсткая схемадокументы / ключ-значение / графы, гибкая схема
Связичерез ключи и JOINобычно вложенные данные
ПримерыPostgreSQL, MySQLMongoDB (документы), Redis (ключ-значение)
Когдасвязанные данные, транзакции, отчётыгибкие данные, кэш, огромный объём, скорость

В большинстве проектов основная БД — реляционная (Postgres/MySQL), а NoSQL — для конкретных задач: Redis для кэша и сессий, MongoDB для гибких документов.

L3 · Senior

Типы связей между таблицами:

  • Один-к-одному (1:1) — пользователь и его профиль
  • Один-ко-многим (1:N) — один пользователь, много заказов. Самая частая
  • Многие-ко-многим (M:N) — студенты и курсы. Реализуется через промежуточную таблицу (student_courses)
В реальном миреQA-автоматизатор почти всегда работает с реляционной БД — чтобы проверить, что данные после операции записались правильно. NoSQL встречается реже и чаще как кэш (Redis). Для QA обычно достаточно прикладного уровня: хватает уверенного SELECT с JOIN и понимания связей таблиц.
Каверзный вопрос«Чем PK отличается от UNIQUE?» — Оба гарантируют уникальность. Но в таблице один primary key constraint (хотя он может состоять из нескольких колонок), он не может быть NULL и обычно по нему строится основной индекс. UNIQUE-ограничений может быть несколько. Поведение UNIQUE + NULL зависит от СУБД: в PostgreSQL NULL по умолчанию считаются различными, поэтому несколько NULL допускаются (есть опция NULLS NOT DISTINCT).
02 Базовый SQL queries
L1 · Основы

Главный запрос — SELECT: «выбери данные». Базовая структура:

SELECT name, email -- какие столбцы FROM users -- из какой таблицы WHERE city = 'Москва' -- условие фильтра ORDER BY name -- сортировка LIMIT 10; -- сколько строк вернуть

SELECT * — все столбцы. Но в реальных запросах лучше перечислять нужные.

L2 · Детали

Фильтры в WHERE:

WHERE age >= 18 -- сравнение WHERE city IN ('Москва', 'Питер') -- из списка WHERE name LIKE 'Иван%' -- по шаблону (начинается с) WHERE age BETWEEN 18 AND 65 -- диапазон WHERE email IS NULL -- пустое значение WHERE age > 18 AND city = 'Москва' -- несколько условий

Агрегатные функции — считают по множеству строк:

SELECT COUNT(*) -- количество строк, SUM(amount) -- сумма, AVG(amount) -- среднее, MIN(amount) -- минимум, MAX(amount) -- максимум FROM orders;
Порядок выполненияSQL пишется как SELECT ... FROM ... WHERE ... ORDER BY, но выполняется иначе: сначала FROM (откуда), потом WHERE (фильтр), потом SELECT (что вернуть), и в конце ORDER BY и LIMIT. Поэтому в WHERE нельзя использовать алиасы из SELECT.
L3 · Senior

DISTINCT — убрать дубликаты: SELECT DISTINCT city FROM users вернёт каждый город один раз.

INSERT / UPDATE / DELETE — QA иногда готовит тестовые данные и убирает их за собой:

INSERT INTO users (name, email) VALUES ('Test', 't@x.io'); UPDATE orders SET status = 'paid' WHERE id = 42; DELETE FROM users WHERE email LIKE '%@test.io';
Осторожно с UPDATE/DELETE без WHEREЗабыл WHERE — изменишь/удалишь всю таблицу. На тестовом стенде это потеря данных, на проде — катастрофа. Привычка: сначала SELECT с тем же условием (посмотреть, что затронется), потом UPDATE/DELETE.

Подзапросы — запрос внутри запроса:

-- пользователи, у которых есть хотя бы один заказ SELECT name FROM users WHERE id IN (SELECT user_id FROM orders);
NULL — отдельная ловушкаNULL ломает SQL-задачи чаще всего. Что помнить: NULL != NULL (сравнение даёт «неизвестно», не True) → нельзя = NULL, только IS NULL / IS NOT NULL. COUNT(column) не считает NULL (а COUNT(*) считает все строки). NOT IN с подзапросом, где есть NULL, может вернуть пустой результат неожиданно. При сложении/конкатенации NULL «заражает» результат (нужен COALESCE).
Каверзный вопрос«Чем WHERE отличается от HAVING?» — WHERE фильтрует строки до группировки, HAVING — группы после агрегации. Нельзя в WHERE написать COUNT(*) > 5 — для этого есть HAVING (см. раздел 04).
03 JOIN главная тема
L1 · Основы

JOIN объединяет строки из двух таблиц по общему ключу. Данные лежат раздельно (users и orders), а нужно показать их вместе — имя пользователя рядом с его заказом.

SELECT users.name, orders.amount FROM users JOIN orders ON users.id = orders.user_id;

ON — условие, по которому связываем: users.id совпадает с orders.user_id.

АналогияJOIN — как объединить два списка объектов по общему полю. Есть список users и список orders со ссылкой user_id. JOIN склеивает их в одну таблицу, где у каждого заказа подставлены данные его пользователя — как merge по ключу.
L2 · Детали

Четыре вида JOIN — главное, что спрашивают:

JOINЧто возвращает
INNER JOINтолько строки, где есть совпадение в обеих таблицах
LEFT JOINвсе строки из левой таблицы + совпавшие из правой (где нет — NULL)
RIGHT JOINвсе из правой + совпавшие из левой
FULL JOINвсе строки из обеих таблиц

INNER JOIN (по умолчанию, можно писать просто JOIN) — пользователи, у которых есть заказы:

SELECT u.name, o.amount FROM users u JOIN orders o ON u.id = o.user_id; -- пользователи без заказов НЕ попадут в результат

LEFT JOIN — все пользователи, даже без заказов (у них amount будет NULL):

SELECT u.name, o.amount FROM users u LEFT JOIN orders o ON u.id = o.user_id; -- Иван без заказов: name='Иван', amount=NULL
Алиасыusers u и orders o — короткие псевдонимы таблиц, чтобы не писать длинные имена. Очень частая практика.
L3 · Senior

LEFT JOIN + IS NULL — частый приём: найти пользователей без заказов:

SELECT u.name FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE o.id IS NULL; -- остаются те, для кого совпадений в orders не нашлось
В реальном миреНа собесах QA в 90% случаев нужен INNER JOIN и LEFT JOIN — этого достаточно. RIGHT JOIN используют редко (его всегда можно переписать через LEFT), FULL JOIN — ещё реже. Если уверенно пишешь INNER и LEFT с пониманием разницы — собес по SQL ты проходишь.
Каверзный вопрос«В чём разница INNER и LEFT JOIN?» — INNER вернёт только тех, у кого есть пара в обеих таблицах. LEFT вернёт всех из левой таблицы, а где пары нет — подставит NULL. Простой способ запомнить: LEFT = «все слева + что нашлось справа».
04 GROUP BY и HAVING aggregation
L1 · Основы

GROUP BY группирует строки по значению столбца и считает агрегат для каждой группы. «Сколько заказов у каждого пользователя?»

SELECT user_id, COUNT(*) AS orders_count FROM orders GROUP BY user_id;
// результат
user_idorders_count
15
23
31
АналогияGROUP BY — как сгруппировать список объектов по полю и посчитать агрегат в каждой группе. В Python это itertools.groupby или Counter: разложил заказы по user_id и посчитал, сколько в каждой пачке.
L2 · Детали

HAVING — фильтр для групп после агрегации. «Пользователи, у которых больше 3 заказов»:

SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id HAVING COUNT(*) > 3;

WHERE vs HAVING — ключевое различие:

  • WHERE фильтрует строки до группировки
  • HAVING фильтрует группы после агрегации
-- заказы за 2026, сгруппировать по юзеру, оставить тех у кого сумма > 10000 SELECT user_id, SUM(amount) AS total FROM orders WHERE created_at >= '2026-01-01' -- фильтр строк ДО GROUP BY user_id HAVING SUM(amount) > 10000; -- фильтр групп ПОСЛЕ
L3 · Senior

Правило GROUP BY: в SELECT можно класть только столбцы из GROUP BY либо агрегатные функции. Нельзя выбрать «голый» столбец, которого нет в группировке — СУБД не знает, какое из значений группы показать.

GROUP BY + JOIN — частая комбинация. «Имя пользователя и количество его заказов»:

SELECT u.name, COUNT(o.id) AS orders_count FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.id, u.name; -- LEFT JOIN, чтобы попали и юзеры с 0 заказов
Каверзный вопрос«Почему COUNT(*) и COUNT(column) могут дать разный результат?» — COUNT(*) считает все строки. COUNT(column) считает только строки, где column не NULL. Поэтому в LEFT JOIN лучше COUNT(o.id), а не COUNT(*) — иначе юзер без заказов даст 1 вместо 0.
05 SQL-задачи с собеса practice
L1 · Основы

Типовые задачи, которые реально дают на собесах. Везде две таблицы: users (id, name, city) и orders (id, user_id, amount, created_at).

Задача 1. Все пользователи из Москвы, отсортированные по имени.

SELECT * FROM users WHERE city = 'Москва' ORDER BY name;

Задача 2. Количество заказов у каждого пользователя.

SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id;
L2 · Детали

Задача 3. Имя пользователя и сумма его заказов (JOIN + GROUP BY).

SELECT u.name, SUM(o.amount) AS total FROM users u JOIN orders o ON u.id = o.user_id GROUP BY u.id, u.name;

Задача 4. Пользователи без единого заказа (LEFT JOIN + IS NULL).

SELECT u.name FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE o.id IS NULL;

Задача 5. Найти дубликаты email (GROUP BY + HAVING).

SELECT email, COUNT(*) AS cnt FROM users GROUP BY email HAVING COUNT(*) > 1;

Задача 6. Топ-5 пользователей по сумме заказов.

SELECT u.name, SUM(o.amount) AS total FROM users u JOIN orders o ON u.id = o.user_id GROUP BY u.id, u.name ORDER BY total DESC LIMIT 5;
L3 · Senior

Задача 7. Второй по величине заказ — классика «со звёздочкой». Без оконных функций, через LIMIT / OFFSET:

SELECT amount FROM orders ORDER BY amount DESC LIMIT 1 OFFSET 1; -- пропустить 1 строку (максимум), взять следующую
ПодвохЭтот запрос вернёт вторую строку, а не вторую уникальную сумму. Если у двух заказов одинаковый максимум, «вторым» окажется тот же максимум. Нужна вторая уникальная сумма — добавь DISTINCT: SELECT DISTINCT amount FROM orders ORDER BY amount DESC LIMIT 1 OFFSET 1. На собесе это стоит проговорить — показывает внимание к дубликатам.
Оконные функции (L3 / optional)Современный способ для задач «второй по величине», «последняя запись по пользователю», «топ-N в группе». ROW_NUMBER() — уникальный номер строки, RANK() — ранг с пропусками при равенстве (1,1,3), DENSE_RANK() — без пропусков (1,1,2). Пример:
SELECT name, amount, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn FROM orders; -- rn=1 → последний заказ каждого пользователя
Для QA это L3/optional, но современно и часто решает задачи элегантнее, чем подзапросы.

Задача 8. Заказы с именем пользователя за последний месяц — комбинация JOIN + WHERE по дате:

SELECT u.name, o.amount, o.created_at FROM orders o JOIN users u ON u.id = o.user_id WHERE o.created_at >= CURRENT_DATE - INTERVAL '1 month' ORDER BY o.created_at DESC;

Задача 9. Исключение по списку (реальная с собеса) — сотрудники, у которых локация не «Водный» и не «ТЦР». Связь employee → location:

SELECT e.* FROM t_employee e JOIN t_location l ON e.loc_id = l.loc_id WHERE l.loc_name NOT IN ('Водный', 'ТЦР'); -- эквивалент: WHERE l.loc_name != 'Водный' AND l.loc_name != 'ТЦР'
Подвох с NULLЕсли в loc_name бывают NULL, NOT IN их не включит в результат без явной ошибки: сравнение с NULL даёт «неизвестно», не True. На собесе это стоит проговорить — «при наличии NULL добавил бы OR loc_name IS NULL, если их надо включить». Показывает, что думаешь про граничные случаи.

Задача 10. Фильтр до агрегации (реальная с собеса) — клиенты со средним успешным платежом > 2000. Связь идёт через счёт: payments.account_id → account.id → client_id:

SELECT a.client_id, AVG(p.amount) AS avg_payment FROM t_account a JOIN t_payments p ON p.account_id = a.id WHERE p.success_flg = 'Y' -- только успешные, ДО агрегации GROUP BY a.client_id HAVING AVG(p.amount) > 2000;
Два ключевых момента1) Связь через счёт — платёж привязан к account_id, а не напрямую к клиенту; нужен JOIN к t_account, чтобы добраться до client_id. 2) WHERE до агрегации — неуспешный платёж (success_flg = 'N') надо исключить до подсчёта AVG, иначе среднее будет рассчитано некорректно. Это и есть разница WHERE (фильтр строк до GROUP BY) vs HAVING (фильтр групп после).
Делегируй LLMSQL-запросы — удобно делегировать, но на собесе писать придётся самому. Для подготовки:

«Дай 10 SQL-задач уровня собеса QA на таблицах users и orders, с решениями. Только SELECT, JOIN, GROUP BY, HAVING — без оконных функций.»

«Объясни, что делает этот запрос построчно, и есть ли в нём ошибка: [запрос]»
Совет на собесеПрежде чем писать запрос — проговори вслух структуру: «беру orders, джойню users по user_id, группирую по имени, считаю сумму, сортирую, беру топ-5». Интервьюер оценивает ход мысли, а не идеальный синтаксис. Забыл точное имя функции — скажи логику, тебе помогут.
06 БД в работе QA practice
L1 · Основы

Зачем QA обращается к базе: проверить, что данные записались правильно после действия. Создал заказ через API → проверил в БД, что он реально появился с нужными полями.

# Псевдо-сценарий автотеста # 1. Создаём заказ через API response = api.post('/orders', json={...}) order_id = response.json()['id'] # 2. Проверяем в БД, что записалось правильно row = db.query("SELECT status FROM orders WHERE id = %s", order_id) assert row.status == 'pending'
L2 · Детали

Основные задачи QA с БД:

  • Проверка результата — после операции данные в БД корректны (статусы, суммы, связи)
  • Подготовка тест-данных — заполнить БД нужным состоянием перед тестом (seed)
  • Очистка — удалить созданное после теста, чтобы не мешало следующим прогонам
  • Поиск причины бага — посмотреть, что реально в данных, когда UI/API ведут себя странно

Инструменты в Python:

  • psycopg (psycopg3) — драйвер для PostgreSQL; psycopg2 по-прежнему часто встречается как legacy-вариант. Сырые SQL-запросы
  • SQLAlchemy — ORM, работа с таблицами как с объектами
  • GUI-клиенты для ручной проверки: DBeaver, pgAdmin, DataGrip
L3 · Senior

Расхождение UI / API / DB — частая задача. Одни и те же данные в трёх местах могут различаться. Как локализовать, где возникает расхождение:

  • UI показывает X, API возвращает Y → проблема на фронте (кэш, неверный рендеринг)
  • API возвращает Y, в БД Z → проблема в бэке (кэш, неверная логика, рассинхрон)
  • В БД Z, а должно быть W → проблема в записи (бизнес-логика, транзакция не прошла)

Проверяешь по цепочке UI → API → DB, находишь, на каком слое значение расходится с ожидаемым.

Подводный каменьEventual consistency: в системах с кэшем или асинхронной обработкой данные в БД появляются не сразу. Автотест, который проверяет БД мгновенно после API-запроса, может упасть. Решение — polling с таймаутом («жди до 10 сек, пока статус не станет shipped»), а не моментальная проверка.
АнтипаттернНе делай тесты зависимыми от конкретных данных в проде/стейдже («там точно есть юзер с id=5»). Тест должен сам создать нужные данные и сам убрать. Иначе он сломается, как только кто-то почистит базу.
Миграции схемы — что проверять QAМиграция меняет структуру БД (новые колонки, типы, ограничения). Риски, которые проверяет QA:
  • Rollback — миграцию можно откатить без потери данных, если что-то пошло не так
  • Backward compatibility — старая версия приложения работает с новой схемой во время раскатки (zero-downtime деплой: сначала схема, потом код)
  • nullable → NOT NULL — добавление ограучения на колонку с существующими NULL упадёт; нужен дефолт или предварительное заполнение
  • Дефолты — у новой колонки корректное значение по умолчанию для старых строк
  • Сохранность данных — после изменения схемы данные на месте и не повреждены (счёт записей, агрегаты, выборочная сверка)
Каверзный вопрос«Как проверить, что миграция данных прошла корректно?» — 1) Кол-во записей до и после совпадает (или изменилось ожидаемо), 2) выборочная сверка ключевых полей, 3) проверка, что связи (FK) не побились, 4) контрольные суммы / агрегаты (SUM, COUNT) до и после, 5) граничные и «странные» записи (NULL, юникод, длинные значения) перенеслись правильно. Плюс для схемных миграций — проверить rollback и что старая версия кода ещё работает (backward compatibility).
07 Реже на собесе: транзакции, NoSQL, теория rare
Зачем это здесьТемы, которые на собесах QA всплывают редко: транзакции, устройство индексов, нормализация, NoSQL/Redis и академическая терминология. Держим коротко, чтобы было что ответить, если спросят.

① Транзакции и ACID

L1 · Основы

Транзакция — группа операций, которая выполняется как единое целое: либо всё успешно, либо ничего (откат). Классический пример — перевод денег:

BEGIN; UPDATE accounts SET balance = balance - 1000 WHERE id = 1; UPDATE accounts SET balance = balance + 1000 WHERE id = 2; COMMIT; -- если между двумя UPDATE упадёт сервер — ROLLBACK, -- деньги не «потеряются» и не «удвоятся»
АналогияТранзакция — как atomic-операция или commit в git: либо все изменения применяются вместе, либо откатываются целиком. Нельзя оказаться в состоянии «деньги списались, но не зачислились» — это как наполовину применённый коммит, которого не бывает.
L2 · Детали

ACID — четыре свойства, которые гарантирует транзакция:

СвойствоЧто значит
Atomicity (атомарность)всё или ничего — частичного выполнения не бывает
Consistency (согласованность)данные валидны до и после (соблюдены все правила и ограничения)
Isolation (изоляция)параллельные транзакции не мешают друг другу
Durability (долговечность)после COMMIT данные сохранены даже при сбое питания

COMMIT — применить изменения. ROLLBACK — откатить всё с начала транзакции.

L3 · Senior

Уровни изоляции — компромисс между корректностью и скоростью. Чем строже, тем безопаснее, но медленнее:

  • Read Uncommitted — видны даже незакоммиченные чужие изменения (грязное чтение)
  • Read Committed — видны только закоммиченные (дефолт в PostgreSQL)
  • Repeatable Read — в рамках транзакции данные не меняются при повторном чтении
  • Serializable — самый строгий, транзакции как будто выполняются по очереди

Deadlock — взаимная блокировка: транзакция A держит ресурс 1 и ждёт ресурс 2, а B держит ресурс 2 и ждёт ресурс 1. СУБД обнаруживает это и откатывает одну из транзакций.

В реальном миреQA редко настраивает уровни изоляции, но понимать их нужно для тестирования конкурентности: два одновременных списания с одного счёта, два refund'а, повторный webhook. Именно изоляция и транзакции защищают от «charged twice» и гонок. На собесе в финтехе про это спрашивают часто.
Каверзный вопрос«Два пользователя одновременно покупают последний товар на складе. Что должно произойти?» — Только один должен успешно купить. Это решается транзакцией + блокировкой строки (SELECT ... FOR UPDATE) или проверкой остатка внутри транзакции. Задача QA — проверить, что не возникает «продали 2 товара при остатке 1».

② Индексы

L1 · Основы

Индекс ускоряет поиск по таблице. Без него БД проверяет каждую строку подряд (full scan). С индексом — находит сразу, как по оглавлению.

АналогияИндекс — как оглавление в книге или отсортированный указатель. Без него, чтобы найти слово, читаешь всю книгу страница за страницей. С указателем — открываешь нужную страницу сразу. Технически индекс — это отдельная упорядоченная структура с указателями на строки в таблице.
L2 · Детали
CREATE INDEX idx_users_email ON users(email); -- теперь поиск WHERE email = '...' будет быстрым

Минусы индексов (почему не создавать индексы без необходимости):

  • Замедляют запись: при каждом INSERT/UPDATE индекс тоже надо обновить
  • Занимают место на диске — каждый индекс это доп. структура

Правило: индексы — на столбцы, по которым часто ищут и джойнят (PK индексируется автоматически, FK почти всегда стоит проиндексировать).

L3 · Senior

Чаще всего используется B-tree индекс (сбалансированное дерево) — хорош для сравнений и диапазонов. Есть и другие: hash (точное совпадение), GIN (полнотекстовый поиск, JSON).

EXPLAIN — команда, показывающая план выполнения запроса: использует ли он индекс или делает full scan. Главный инструмент для разбора медленных запросов.

Каверзный вопрос«Добавили индекс, а запрос не ускорился. Почему?» — Варианты: 1) в таблице мало строк (full scan и так быстр), 2) запрос всё равно читает большую часть таблицы, 3) индекс не подходит под условие (например, LIKE '%text' с ведущим % не использует обычный индекс), 4) статистика БД устарела. Проверяется через EXPLAIN.

③ Нормализация

L1 · Основы

Нормализация — организация данных так, чтобы убрать дублирование. Вместо того чтобы в каждой строке заказа писать имя и адрес пользователя, хранят user_id — ссылку, а данные пользователя лежат один раз в таблице users.

L2 · Детали

Три базовые нормальные формы (для собеса достаточно знать суть):

  • 1NF — в каждой ячейке атомарное значение (не список). Не «телефоны: 123, 456», а отдельные строки
  • 2NF — нет частичной зависимости: все поля зависят от всего первичного ключа
  • 3NF — нет транзитивных зависимостей: поля зависят только от ключа, а не друг от друга
L3 · Senior

Денормализация — намеренное дублирование данных ради скорости чтения. В аналитике и высоконагруженных системах часто сознательно нарушают нормализацию: лучше хранить копию, чем делать тяжёлый JOIN на каждый запрос.

В реальном миреQA нормализацию почти не трогает — это забота архитекторов и бэкендеров. Знать определения нужно ровно для собеса. На практике важнее понимать связи между таблицами (через что джойнить), чем уметь доказать, что схема в 3NF.

④ NoSQL и Redis

L1 · Основы

NoSQL — базы без жёсткой табличной схемы. Несколько семейств:

  • Key-value — простые пары ключ→значение. Redis, DynamoDB
  • Документные — хранят JSON-подобные документы. MongoDB
  • Wide-column — Cassandra (большие объёмы, распределённость). Для аналитических колоночных БД примеры другие: ClickHouse, BigQuery, Redshift
  • Графовые — для связей (соцсети, рекомендации). Neo4j
АналогияРеляционная БД — это таблица в Excel со строгими столбцами. Key-value (Redis) — это обычный словарь dict в памяти: cache["user:42"] = {...}. Документная (Mongo) — список JSON-объектов, где у каждого может быть свой набор полей.
L2 · Детали

Redis — самая частая NoSQL, которую встретит QA. Это in-memory key-value хранилище: данные лежат в оперативной памяти, поэтому очень быстрое (микросекунды).

Где применяется:

  • Кэш — хранить результат тяжёлого запроса, чтобы не обращаться к БД каждый раз
  • Сессии — session_id → данные пользователя
  • Rate limiting — счётчик запросов с TTL
  • Очереди — простые задачи между сервисами
  • Счётчики, лидерборды — атомарные инкременты, сортированные множества

Поддерживает структуры: strings, hashes, lists, sets, sorted sets. У каждого ключа можно задать TTL — время жизни, после которого ключ автоматически удаляется (идеально для кэша и сессий).

SET user:42 "{...}" EX 3600 # записать с TTL 1 час GET user:42 # прочитать INCR page_views # атомарно +1 (счётчик) TTL user:42 # сколько осталось жить
L3 · Senior

Что тестировать с Redis:

  • Кэш работает: первый запрос идёт в БД, повторный — из кэша (быстрее, БД не дёргается)
  • Инвалидация кэша: после изменения данных кэш сбросился, не отдаёт устаревшее
  • TTL: ключ исчезает после истечения времени
  • Сессии: после logout сессия удалена из Redis
Подводный каменьТипичная проблема кэша — устаревшие данные (stale cache). Обновили запись в БД, а Redis отдаёт старое значение, потому что забыли инвалидировать. Классический баг: «поменял имя в профиле, а везде показывается старое». При тестировании всегда проверяй пару «изменение → отражается ли сразу».
Каверзный вопрос«Redis — это база данных или кэш?» — И то, и другое. Чаще используется как кэш (быстрый слой перед основной БД), но может быть и основным хранилищем. Persistence есть: RDB (снапшоты на диск) и AOF (лог операций). Но если данные критичны и их нельзя терять — основной БД обычно остаётся реляционная.

⑤ Академическая терминология

L1 · Основы

Теоретические термины из реляционной модели Кодда. На практике говорят «таблица / строка / столбец», но академические названия такие:

Академический терминНа практике
Отношение (relation)таблица
Кортеж (tuple)строка / запись
Атрибут (attribute)столбец / поле
Домен (domain)тип и множество допустимых значений атрибута
ЗапомнитьКортеж = строка таблицы. Название идёт от математики: строка — это упорядоченный набор значений (как tuple в Python). Если спросят «что такое кортеж» — это просто строка в реляционной БД.
