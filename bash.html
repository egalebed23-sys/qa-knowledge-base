01 Bash: команды и навигация cli basics
L1 · Основы

Почти все серверы и CI-раннеры работают на Linux, а общение с ними — через shell (чаще bash). QA заходит в консоль, чтобы посмотреть логи, проверить сервис, запустить тесты. Базовая навигация:

pwd # где я сейчас (путь) ls -la # список файлов, включая скрытые и права cd /var/log # перейти в директорию cat app.log # вывести содержимое файла cp / mv / rm # копировать / переместить / удалить mkdir tests # создать директорию
АналогияShell — это REPL операционной системы: вводишь команду, получаешь результат, как в интерактивном Python. Файловая система — дерево: / корень, ветки — директории, листья — файлы. cd — переход по дереву, ls — посмотреть, что в текущем узле.
L2 · Детали

Пути: / — корень (абсолютный путь), ~ — домашняя директория, . — текущая, .. — родительская.

Права доступа (ls -l показывает rwxr-xr-x):

  • r (read=4), w (write=2), x (execute=1) — для владельца, группы, остальных
  • chmod 755 file — владельцу всё (7=rwx), остальным чтение+выполнение (5=r-x)
  • chmod +x script.sh — сделать файл исполняемым
  • chown user file — сменить владельца

find — поиск файлов:

find . -name "*.log" # все .log в текущем дереве find /var -mtime -1 # изменённые за последний день
L3 · Senior

Пайпы и редиректы — основа мощи shell: соединяешь простые команды в цепочки.

command > file.txt # вывод в файл (перезапись) command >> file.txt # дописать в конец command 2> errors.txt # перенаправить ошибки (stderr) command 2>&1 # stderr вместе со stdout cmd1 | cmd2 # вывод cmd1 на вход cmd2 (пайп) cmd1 && cmd2 # cmd2 только если cmd1 успешна
АналогияПайп | — это композиция функций: cmd2(cmd1(input)). Выход одной команды становится входом следующей, как в функциональном программировании. cat log | grep error | wc -l = «возьми лог → отфильтруй ошибки → посчитай» — конвейер обработки данных.
Каверзный вопрос«Что значит 2>&1?» — Перенаправить stderr (поток ошибок, дескриптор 2) туда же, куда идёт stdout (вывод, дескриптор 1). Нужно, чтобы поймать и обычный вывод, и ошибки в один файл/пайп: ./script.sh > all.log 2>&1.
Безопасная работа в консолиCLI не прощает ошибок — особенно на чужих/боевых машинах:
  • Перед rm — проверь, где ты: pwd и ls, что именно удаляешь
  • rm -rf — осознанно: один неверный путь сносит дерево без корзины и без отмены
  • cp -i, mv -i — режим с подтверждением перезаписи (safer)
  • Не запускать curl ... | bash из интернета, не прочитав, что внутри — это выполнение чужого кода
  • На production — сначала read-only диагностика (смотреть логи/статус), а не команды, меняющие состояние
02 Логи и текст главное для QA
L1 · Основы

Главное, зачем QA в консоли — разбирать логи. Когда тест упал или сервис ведёт себя странно, ответ обычно в логах. Базовый инструментарий:

grep "error" app.log # найти строки со словом error tail -f app.log # следить за логом в реальном времени tail -n 100 app.log # последние 100 строк head app.log # первые строки less app.log # листать большой файл
Аналогияgrep — это filter() по строкам файла: оставляет только те, что подходят под условие. tail -f — как подписка на поток событий: новые строки появляются на экране по мере записи, как live-лента.
L2 · Детали

grep с флагами — рабочая лошадка:

ФлагЧто делает
-iигнорировать регистр (Error = error)
-rрекурсивно по всем файлам в директории
-vинвертировать — строки БЕЗ совпадения
-nпоказать номера строк
-cпосчитать количество совпадений
-A 3 / -B 3 / -C 33 строки после / до / вокруг совпадения

Пайпы для фильтрации логов — комбинируешь команды:

# посчитать, сколько ошибок в логе grep -c "ERROR" app.log # следить за логом, но только за строками с ошибками tail -f app.log | grep -i error # найти ошибки и показать 5 строк контекста вокруг grep -C 5 "Exception" app.log

Обработка текста: wc -l (посчитать строки), sort (сортировать), uniq -c (уникальные + счётчик).

# топ-10 самых частых ошибок в логе grep "ERROR" app.log | sort | uniq -c | sort -rn | head
L3 · Senior

awk и sed — для более сложной обработки (знать базово):

  • awk '{print $1}' — вывести первый столбец каждой строки
  • sed 's/old/new/g' — заменить текст
jq — работа с JSON (важнее awk/sed для QA)API отдаёт JSON, structured-логи в JSON — jq вытаскивает поля без регулярок:
# достать поле из ответа API curl -s api/orders/1 | jq '.status' # отфильтровать массив: только неуспешные curl -s api/orders | jq '.[] | select(.status != "paid")' # из structured-логов вытащить request_id ошибок cat app.log | jq 'select(.level=="error") | .request_id'
Полезнее, чем углублять awk/sed: большинство данных у QA — это JSON (API, логи).

Реальный кейс — найти ошибку по request-id (связка с дебагом 500 из раздела API):

# 1. из ответа API взял X-Request-ID: abc-123 # 2. ищем все логи этого запроса через все сервисы grep -r "abc-123" /var/log/services/ # 3. нашли строку с трейсом — видно, где упало

Централизованное логирование — на реальном проде grep по файлам почти не используют. Логи всех сервисов стекаются в одну систему с поиском через UI (сервисов десятки, логи в контейнерах исчезают при перезапуске). Основные системы — одной таблицей:

СистемаКратко
ELK / KibanaElasticsearch + Kibana-UI. Классика, часто self-hosted
Grafana Lokiлоги рядом с метриками в Grafana, популярен с Kubernetes
Datadogоблачный all-in-one (логи + метрики + трейсы + APM), платный
Splunkэнтерпрайз, мощный язык запросов, дорогой

Принцип у всех один — тот же, что у grep, но через UI и по всем сервисам сразу: поиск по тексту, фильтр по полям (service:payment AND level:error), time range, дашборды/алерты, корреляция по request-id.

Structured loggingСовременные сервисы пишут логи не строками, а JSON: {"level":"error","service":"payment","request_id":"abc-123","msg":"..."}. Это позволяет фильтровать по полям (level:error), а не парсить текст регулярками. Для QA — намного удобнее искать: точечные фильтры вместо grep по сырому тексту.
В реальном миреQA-автоматизатор использует оба подхода. Kibana / Datadog / Grafana — для дебага в проде и на staging: найти ошибку по request-id, посмотреть всплеск 500-х, собрать путь запроса через микросервисы. grep / tail на сервере — когда централизованного сбора нет: CI-раннер, локальный контейнер, быстрый дебаг. Принцип одинаковый (поиск → фильтр → контекст), меняется только инструмент. Знать стоит и то, и другое.
Делегируй LLMГенерация команд: «Напиши команду: найти в app.log все строки с ERROR за сегодня, сгруппировать по тексту ошибки и показать топ-5 по частоте.»

Разбор awk/sed: вставь непонятную команду → «Объясни, что делает эта awk-команда построчно.»
03 Процессы, порты, сеть diagnostics
L1 · Основы

Вторая частая задача — проверить, жив ли сервис и что с ним. Процессы:

ps aux | grep python # найти процессы python top # или htop — что потребляет CPU/память в реальном времени kill 1234 # мягко остановить процесс по PID kill 1234 # обычное завершение (SIGTERM) — сначала пробуем это kill -9 1234 # принудительное (SIGKILL) — только если не завершается
АналогияПроцесс — это запущенный экземпляр программы, как объект — экземпляр класса. У каждого свой PID (как id объекта). kill — отправить сигнал процессу; kill -9 — жёсткое завершение, как принудительно завершить зависший поток.
L2 · Детали

Порты — какой процесс на каком порту слушает (частая проверка «занят ли порт», «поднялся ли сервис»):

lsof -i :5432 # кто слушает порт 5432 ss -tlnp # все слушающие TCP-порты + процессы # (netstat — старый аналог ss)

Сеть и диагностика:

КомандаЗачем
curl urlвыполнить запрос к API из консоли, посмотреть ответ
curl -vс деталями: заголовки, статус, TLS
ping hostдоступен ли хост по сети
dig / nslookupрезолвится ли домен в IP (DNS)
nc -zv host 443открыт ли порт на хосте
# проверить API прямо из консоли curl -X POST https://api.shop.com/orders \ -H "Authorization: Bearer token" \ -H "Content-Type: application/json" \ -d '{"product_id": 42}'
L3 · Senior

SSH — подключиться к удалённому серверу (смотреть логи, проверить сервис):

ssh user@server.com # подключиться (по ключу, не паролю) scp file.txt user@server:/tmp # скопировать файл на сервер
В реальном миреДля QA-автоматизатора этот набор — must-have: curl (проверить API), grep/tail (логи), lsof/ss (порты), ssh (на сервер). Это не «системное администрирование», а ежедневный дебаг. Глубокое системное администрирование (настройка демонов, iptables, тюнинг ядра) знать не нужно — это зона DevOps/SRE.
Каверзный вопрос«Сервис не отвечает. Как по шагам диагностируешь в консоли?» — 1) процесс жив? ps aux | grep service, 2) слушает порт? ss -tlnp | grep :8080, 3) отвечает локально? curl localhost:8080/health, 4) что в логах? tail -n 100 app.log | grep -i error, 5) если в другом сервисе — DNS/сеть: dig, nc. Идёшь от процесса к сети к логам.
04 Bash-скрипты scripting
L1 · Основы

Скрипт — набор команд в файле, чтобы не вводить их каждый раз. Для QA — автоматизация рутины: поднять окружение, прогнать тесты, собрать логи.

#!/bin/bash # shebang — чем исполнять echo "Running tests..." pytest tests/ echo "Done"

Запуск: chmod +x run.sh (сделать исполняемым) → ./run.sh.

L2 · Детали

Переменные, условия, циклы:

#!/bin/bash NAME="staging" # переменная (без пробелов вокруг =) echo "Env: $NAME" # использование через $ if [ "$NAME" = "prod" ]; then echo "Production environment selected" fi for file in *.log; do # цикл по файлам echo "Processing $file" done

Спецпеременные: $1, $2 — аргументы скрипта, $? — код возврата последней команды (0 = успех), $# — число аргументов.

L3 · Senior

Реальный кейс QA — скрипт поднимает окружение, ждёт сервис, гоняет тесты:

#!/bin/bash set -e # падать сразу при первой ошибке docker compose up -d # поднять зависимости # ждём, пока БД ответит (polling вместо sleep) until nc -z localhost 5432; do echo "Ждём БД..."; sleep 1 done pytest tests/ --maxfail=1 RESULT=$? # сохранили код возврата docker compose down # остановили окружение exit $RESULT
Полезные привычкиset -e — скрипт падает при первой ошибке (а не идёт дальше на сломанном). set -u — падать при использовании необъявленной переменной. set -o pipefail — без него пайп возвращает код последней команды и может скрыть ошибку ранней: в failing_cmd | tee log упадёт первая, но код будет 0 от tee. Together: set -euo pipefail — стандартная защитная шапка скрипта. Ожидание сервиса — через polling (until nc -z), а не sleep 30 наугад.
Exit codes — почему CI «видит» падениеКаждая команда возвращает код: 0 = успех, не 0 = ошибка. CI считает шаг проваленным, если команда вернула non-zero — поэтому pytest с упавшими тестами возвращает не 0, и пайплайн краснеет. Проверить код последней команды: echo $?. В скрипте именно поэтому сохраняют RESULT=$? и пробрасывают через exit $RESULT.
Делегируй LLMBash-синтаксис легко забывается — отличная задача для LLM:

«Напиши bash-скрипт: поднять docker compose, дождаться готовности Postgres на 5432, прогнать pytest, в конце остановить compose и вернуть код тестов.»
05 Docker containers
L1 · Основы

Docker упаковывает приложение со всем окружением в изолированный контейнер, который одинаково работает везде. Для QA — поднять БД/сервис для тестов одной командой. Базовые команды:

docker ps # запущенные контейнеры docker ps -a # все, включая остановленные docker logs container_id # логи контейнера docker run -e POSTGRES_PASSWORD=test postgres:16 # запустить (Postgres без пароля не стартует) docker stop <container> # остановить контейнер docker rm <container> # удалить остановленный контейнер
АналогияКонтейнер — как venv, но для всего окружения: ОС, библиотеки, сама программа. Образ (image) = класс, контейнер = объект (запущенный экземпляр). Один образ → много контейнеров.
L2 · Детали

Запуск с параметрами:

# поднять Postgres для тестов docker run -d \ # -d = в фоне -p 5432:5432 \ # проброс порта host:container -e POSTGRES_PASSWORD=test \ # переменная окружения --name test-db \ # имя контейнера postgres:16

docker compose (V2 — единая команда, не старый docker-compose) — поднять несколько сервисов вместе (app + БД + Redis) одним файлом:

docker compose up -d # поднять всё из docker-compose.yml docker compose down # остановить и удалить ресурсы docker compose logs -f # логи всех сервисов

Dockerfile — рецепт сборки образа:

FROM python:3.12 WORKDIR /app COPY requirements.txt . RUN pip install -r requirements.txt COPY . . CMD ["pytest"]
L3 · Senior

Дебаг контейнера — частая задача, когда «в контейнере не работает»:

docker exec -it container bash # зайти ВНУТРЬ контейнера docker logs -f --tail 100 cont # смотреть логи в реальном времени docker inspect container # детали: сеть, env, volumes

Зайдя внутрь через exec, проверяешь изнутри: переменные окружения (env), доступность других сервисов (curl, nc), файлы конфигов.

Testcontainers — библиотека, поднимает реальные зависимости (Postgres, Kafka) в Docker прямо из автотеста и останавливает после теста. Тест работает с настоящей БД, а не моком. Это связка раздела с CI/CD (раздел 06).

Каверзный вопрос«Контейнер сразу падает после запуска. Как дебажить?» — 1) docker logs container — чаще всего причина видна в логах (упал процесс, не найден файл, нет env), 2) docker ps -a — посмотреть exit code, 3) запустить с переопределением команды на bash и зайти внутрь контейнера и проверить состояние, 4) проверить docker inspect — правильные ли env и volumes. Контейнер живёт, пока жив его главный процесс — если процесс упал, контейнер останавливается.
06 Kubernetes orchestration
L1 · Основы

Kubernetes (k8s) — оркестратор контейнеров. Когда контейнеров десятки и сотни, кто-то должен их запускать, перезапускать упавшие, масштабировать, распределять по серверам. Это и делает k8s.

АналогияDocker запускает один контейнер на одной машине. Kubernetes — как оркестратор/супервизор для целого парка: следит, чтобы нужное число копий всегда работало, перезапускает упавшие, распределяет нагрузку. Как process manager (типа supervisord), но для контейнеров на кластере машин.
L2 · Детали

Ключевые понятия:

  • Pod — минимальная единица, один или несколько контейнеров вместе. Обычно = один экземпляр сервиса
  • Deployment — управляет репликами подов: «держи 3 копии сервиса всегда живыми»
  • Service — стабильная точка доступа к подам (поды меняются, адрес остаётся)
  • Namespace — логическое разделение (dev / staging / prod в одном кластере)

kubectl — команда для общения с кластером, основное для QA:

kubectl get pods # список подов kubectl logs pod-name # логи пода kubectl logs -f pod-name # следить за логами kubectl describe pod pod-name # детали, события, почему не стартует
L3 · Senior

Что реально нужно QA в k8s — не писать манифесты, а уметь диагностировать:

kubectl exec -it pod -- bash # зайти внутрь пода kubectl get pods -w # следить за статусами подов kubectl port-forward pod 8080:80 # пробросить порт пода к себе

port-forward особенно полезен — пробрасываешь порт сервиса из кластера на свою машину и тестируешь его локально через curl/Postman.

В реальном миреKubernetes — зона DevOps/SRE, QA редко настраивает его сам. От QA-автоматизатора ждут понимания «что это и зачем» + умения kubectl logs, kubectl describe (посмотреть, почему под не поднялся), kubectl exec (зайти внутрь), port-forward (протестировать сервис локально). Писать YAML-манифесты и настраивать кластер — не задача QA.
Каверзный вопрос«Под в статусе CrashLoopBackOff. Что это и что делать?» — Под постоянно падает и перезапускается по кругу. Действия: kubectl logs <pod> --previous (логи упавшего контейнера — текущий мог уже перезапуститься, поэтому важен --previous), kubectl describe pod (события: нехватка ресурсов, не прошёл health-check, не скачался образ), kubectl get events (события неймспейса — проблемы scheduling/probe/image pull). Частые причины: ошибка в коде при старте, неверный конфиг/env, failed liveness probe.
