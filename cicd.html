01 CI/CD: что это basics
L1 · Основы

CI/CD — автоматизация сборки, тестирования и доставки кода. Три понятия, которые часто путают:

  • CI (Continuous Integration) — каждый коммит автоматически собирается и прогоняется через тесты. Баги ловятся сразу, а не «под релиз»
  • CD (Continuous Delivery) — код всегда готов к деплою, выкатка — по кнопке (ручное подтверждение)
  • CD (Continuous Deployment) — деплой в прод полностью автоматический, без ручного шага
АналогияCI — как автопрогон тестов при каждом сохранении файла, только для всей команды: каждый коммит сразу проверяется. Pipeline — конвейер функций, где выход одной стадии (собранный билд) идёт на вход следующей (тесты → деплой).
L2 · Детали

Зачем это нужно:

  • Каждый разработчик делает мелкие изолированные коммиты — меньше конфликтов при слиянии
  • Фокус на бизнес-логике, а не на ручном деплое (он автоматизирован)
  • Стандартизация: каждый коммит проходит одни и те же стадии (build, lint, тесты)
  • Ранний поиск дефектов — баг виден через минуты после коммита, а не через неделю

Delivery vs Deployment — разница в последнем шаге: Delivery доводит до состояния «готово, нажми кнопку», Deployment жмёт кнопку сам. Большинство команд — на Delivery (ручное подтверждение прод-релиза).

Инструменты: GitHub Actions, GitLab CI, Jenkins, CircleCI, TeamCity. Принцип у всех один — описываешь стадии в конфиге, система их выполняет на каждый коммит/PR.

L3 · Senior
В реальном миреCI/CD неразрывно связан с trunk-based development (см. раздел теории): мелкие частые коммиты в main, автотесты на каждый PR, релизы несколько раз в день. Здоровье процесса меряют DORA-метриками: deployment frequency, lead time for changes, change failure rate и failed deployment recovery time (раньше назывался MTTR). В последние годы DORA добавила пятую — rework / failed deployment recovery — суть та же: как быстро и надёжно команда доставляет изменения. Для QA это значит одно главное требование к автотестам — они должны быть быстрыми и стабильными, иначе становятся тормозом всего пайплайна.
Каверзный вопрос«В чём разница Continuous Delivery и Continuous Deployment?» — В обоих случаях код автоматически проходит все стадии и готов к проду. Разница в финале: Delivery требует ручного нажатия кнопки «выкатить», Deployment выкатывает в прод автоматически, без человека. Deployment требует очень зрелого тестирования и мониторинга.
02 Pipeline и стадии pipeline
L1 · Основы

Pipeline — последовательность стадий, через которые проходит код от коммита до прода. Если стадия упала — пайплайн останавливается, дальше код не идёт.

build→ lint→ unit→ integration→ deploy

Типичные стадии: собрать (build) → проверить стиль кода (lint) → быстрые тесты (unit) → тесты посложнее (integration) → выкатить (deploy).

L2 · Детали

На что запускается пайплайн (триггеры):

  • push — на каждый коммит в ветку
  • pull/merge request — при создании или обновлении PR (главный момент для прогона тестов)
  • schedule — по расписанию (ночной полный регресс)
  • manual — по кнопке (деплой в прод)

Артефакты — то, что передаётся между стадиями или сохраняется: собранный билд, отчёты тестов, логи, скриншоты. Стадия deploy берёт артефакт стадии build.

Простой пример конфига (GitHub Actions):

name: CI on: [push, pull_request] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: '3.12' cache: 'pip' # кэш зависимостей между прогонами - run: pip install -r requirements.txt - run: python -m pytest --maxfail=1
L3 · Senior

Оптимизация пайплайна:

  • Кэширование зависимостей — не качать пакеты заново каждый раз (кэш pip/npm)
  • Параллельные стадии — lint и unit могут бежать одновременно
  • Матричные сборки — прогнать на разных версиях (Python 3.10/3.11/3.12) параллельно
  • Fail fast — --maxfail=1, чтобы не ждать весь прогон, если уже большая часть набора уже падает
В реальном миреГлавная метрика пайплайна для команды — время от коммита до зелёного билда. Если CI идёт 40 минут, разработчики переключаются на другие задачи и теряют контекст. Цель — держать основной прогон в пределах 10–15 минут: быстрые тесты на каждый PR, тяжёлый полный регресс — ночью по расписанию.
Делегируй LLMГенерация конфига: «Напиши GitHub Actions workflow: на каждый PR установить зависимости, прогнать pytest с параллелизацией и собрать Allure-отчёт.»

Дебаг пайплайна: вставь лог упавшей стадии → «Почему упало и как починить?»
03 Автотесты и quality gates quality gates
L1 · Основы

Автотесты встраиваются в пайплайн как стадия. Quality gate — условие, которое должно выполниться, чтобы код прошёл дальше (например, в merge или deploy).

tests→ quality gate→ merge / deploy
АналогияQuality gate — это if в пайплайне, который пускает дальше только при выполнении условий. Не прошёл тесты или упало покрытие — return, merge заблокирован. Как assert перед критичной операцией.
L2 · Детали

Где какие тесты гонять (по скорости и стоимости):

КогдаЧто гоняем
На каждый коммит/PRlint + unit + быстрые integration (должны быть быстрыми, 5–10 мин)
После merge в mainполный integration, smoke на staging
Ночью по расписаниюполный regression, E2E, нагрузочные
Перед prod-деплоемsmoke на prod-like окружении

Типичные quality gates перед merge:

  • Все тесты прошли (зелёные)
  • Code coverage не упал ниже порога
  • Линтер и форматтер чисты
  • Нет critical/high уязвимостей (security scan)
  • Код-ревью одобрено (минимум N апрувов)
L3 · Senior

Smoke / sanity / regression в CI — когда что:

  • Smoke — 5–10 критичных проверок, гоняются на каждый деплой. «Сервис вообще жив?»
  • Sanity — проверка конкретной зоны после фикса
  • Regression — полный набор, тяжёлый. Ночью или перед релизом, не на каждый PR

Pre-merge vs post-merge: быстрые тесты — до merge (блокируют вливание сломанного кода). Тяжёлые E2E — часто после merge, чтобы не тормозить разработчиков, но с алертом, если что-то упало.

БалансСлишком строгие gates → у команды появляется сопротивление процессу, пайплайн медленный, появляется соблазн «обойти». Слишком слабые → дефекты попадают в production. Senior ищет баланс: быстрый и обязательный pre-merge (unit, lint, smoke) + полный, но не блокирующий post-merge (E2E, regression).
Каверзный вопрос«Тесты идут 40 минут на каждый PR, разработчики жалуются. Что делаешь?» — Не «убрать тесты». А: 1) разделить на быстрые (pre-merge) и медленные (post-merge/ночью), 2) параллелизация (pytest-xdist, шардинг), 3) кэш зависимостей, 4) убрать дубли и flaky, 5) часть E2E заменить на более дешёвые contract/integration тесты.
04 Стратегии деплоя deployment
L1 · Основы

Как выкатить новую версию, не уронив всё для всех пользователей сразу. Четыре подхода:

Blue / Green
Две одинаковые среды. На «blue» работает текущая версия, на «green» разворачивают новую. Когда готово — переключают весь трафик на green. Откат = переключить обратно.
Canary
Новую версию выкатывают на маленький % пользователей (5%), наблюдают метрики. Всё ок → постепенно 50% → 100%. Проблема → откат, затронуты немногие.
Rolling
Обновляют инстансы по очереди: пока часть на новой версии, остальные ещё на старой. Без простоя, но обе версии какое-то время работают вместе.
Feature flags
Код новой фичи уже в проде, но спрятан за флагом. Включают для части юзеров без передеплоя. Выключают мгновенно, если что-то не так.
АналогияBlue/green — как переключение симлинка между двумя папками: current → v1 или current → v2, мгновенно туда и обратно. Feature flag — обычный if feature_enabled: в коде, который решает, показывать фичу или нет.
L2 · Детали

Rollback (откат) — возврат на предыдущую рабочую версию, когда новая сломалась. Чем быстрее и надёжнее откат, тем смелее можно выкатывать.

Как тестировать каждую стратегию:

  • Feature flag — проверить обе ветки: фича включена работает правильно, выключена — не ломает остальное и не видна
  • Canary — на канареечной версии гоняют smoke + следят за метриками (error rate, latency). Рост ошибок → авто-откат
  • Blue/green — прогнать тесты на green до переключения трафика
  • Rollback — проверить, что откат реально возвращает рабочее состояние, данные не теряются
Связь с теориейЭто shift-right тестирование: проверяем поведение уже в проде, на реальных пользователях, но контролируемо. Canary + feature flags + быстрый rollback — основа безопасных частых релизов.
L3 · Senior

Основной риск — миграции БД при rollback. Код откатить легко, а схему БД — нет. Если новая версия добавила/изменила колонку, откат кода на старую версию может сломаться о новую схему.

  • Решение: backward-compatible миграции. Сначала добавить колонку (старый код её игнорирует), потом выкатить код, который её использует. Удалять старое — только через несколько релизов
  • Это называется expand-contract (или parallel change): расширяем схему → мигрируем → сужаем
В реальном миреВ современных командах базовый набор — feature flags + canary + быстрый rollback. Флаги управляются через сервисы вроде LaunchDarkly, Unleash, Flagsmith. Часто фичу катят «тёмным запуском» (dark launch): код работает в проде, но результат скрыт от пользователя — так тестируют производительность и корректность на реальном трафике до публичного включения.
Каверзный вопрос«Как тестировать rollback?» — Не только «нажали откат — вернулось». Проверить: 1) приложение реально откатилось на прошлую версию, 2) данные, созданные на новой версии, не потерялись и читаются старой, 3) миграции БД обратно совместимы (старый код работает с новой схемой), 4) время отката укладывается в требования (RTO). Rollback, который теряет данные — хуже бага, который чинили.
05 Docker для QA containers
L1 · Основы

Docker упаковывает приложение со всем окружением (ОС, библиотеки, зависимости) в контейнер — изолированную коробку, которая одинаково работает на любой машине. «У меня работает, а на сервере нет» — Docker решает эту проблему.

АналогияКонтейнер — как venv в Python, но не только для пакетов, а для всего окружения целиком: ОС, системные библиотеки, сама программа. Запускается одинаково на ноутбуке и на сервере, потому что несёт окружение с собой.

Зачем QA: поднять БД или сервис для тестов одной командой, иметь одинаковое окружение локально и в CI, не засорять свою машину.

L2 · Детали

Ключевые понятия:

  • Образ (image) — шаблон, «рецепт» с приложением и окружением
  • Контейнер (container) — запущенный экземпляр образа (образ : контейнер = класс : объект)
  • Dockerfile — инструкция, как собрать образ
  • docker-compose — поднять несколько контейнеров вместе (app + БД + Redis) одной командой
# Поднять Postgres для тестов одной командой docker run -d -p 5432:5432 \ -e POSTGRES_USER=test -e POSTGRES_PASSWORD=test -e POSTGRES_DB=test \ postgres:16 # Или несколько сервисов через compose docker compose up -d

Testcontainers — библиотека, которая поднимает реальные зависимости (Postgres, Kafka, Redis) в Docker прямо из автотеста и останавливает после. Тест работает с настоящей БД, а не с моком.

L3 · Senior

Docker в CI — пайплайн поднимает контейнеры с зависимостями, гоняет интеграционные тесты против них, останавливает. Изолированное чистое окружение на каждый прогон — нет данных, оставшихся от предыдущих прогонов.

Kubernetes (k8s) — оркестратор контейнеров: запускает, масштабирует, перезапускает упавшие, распределяет по серверам. QA напрямую трогает редко, но знать стоит:

  • Pod — минимальная единица, один или несколько контейнеров
  • Deployment — управляет репликами подов (сколько копий держать)
  • Service — стабильная точка доступа к подам
В реальном миреQA-автоматизатору достаточно уметь: запустить контейнер, прочитать docker-compose, поднять зависимости для тестов, посмотреть логи (docker logs). Kubernetes — это уже зона DevOps; от QA ждут понимания «что это и зачем», а не умения писать манифесты. Testcontainers — самый практичный навык: реальные зависимости в тестах без ручного поднятия.
Каверзный вопрос«Зачем поднимать реальную БД в Docker, если можно замокать?» — Мок проверяет твою логику, но не ловит реальные проблемы: SQL-синтаксис, схему, поведение транзакций, особенности конкретной СУБД. Testcontainers даёт настоящую БД — тесты ближе к проду. Мокать стоит чужие внешние сервисы, а свою БД лучше поднимать реальную.
06 CI/CD в работе QA practice
L1 · Основы

Главные задачи QA-автоматизатора в CI/CD:

  • Встроить автотесты в пайплайн как стадию
  • Настроить quality gates (что блокирует merge/deploy)
  • Держать тесты быстрыми и стабильными
  • Настроить отчёты, чтобы падения было легко разбирать
Что QA смотрит в CI после паденияКогда билд красный, разбор идёт по порядку: failed stage (на какой стадии упало — lint/unit/e2e), test report (какие тесты, с какой ошибкой), artifacts (скриншоты, видео, request/response, browser traces — для UI-тестов незаменимо), logs приложения и теста, commit diff (что менялось — часто причина там), flaky history (падал ли этот тест раньше нестабильно), owner / assignee (кто отвечает за фикс). Хороший CI даёт всё это из коробки — иначе разбор превращается в чтение сырых логов.
L2 · Детали

Flaky-тест в main pipeline — тест, который иногда падает без причины. Это типичная проблема CI, потому что подрывает доверие к тестам («опять упало, перезапусти»). Что делать:

  • Не игнорировать — flaky-тест хуже отсутствия теста, он учит команду игнорировать красное
  • Изолировать — пометить (quarantine), убрать из блокирующих, чинить отдельно
  • Найти причину: гонки, зависимость от порядка тестов, sleep вместо ожиданий, общие тест-данные, внешние сервисы
  • retry — временная мера, а не устранение причины: pytest-rerunfailures прячет проблему, но не решает

Параллелизация — главный способ ускорить suite:

  • pytest-xdist — запуск тестов в несколько процессов (pytest -n 4)
  • Шардинг — разбить тесты на части и гонять на разных раннерах CI параллельно
  • Требование: тесты должны быть изолированы (не зависеть друг от друга и от общих данных)
Environments (окружения)Типичная цепочка: dev → test/qa → staging → pre-prod → prod. Staging — максимально похож на prod (та же конфигурация, близкие данные), но «у нас на staging зелёное» не гарантирует prod: отличаются объём и реальность данных, нагрузка, сторонние интеграции (в staging часто моки/sandbox), секреты и фича-флаги. Поэтому после деплоя в prod всё равно нужны проверки (smoke, мониторинг), а не только зелёный staging.
Quarantine policy для flaky — формализуйКарантин flaky-теста — не «убрал и забыл», а процесс: 1) flaky временно не блокирует merge (вынесен из gate), 2) сразу заводится задача на исправление, 3) есть SLA на починку (например, неделя), 4) нельзя держать в карантине бесконечно — иначе карантин становится свалкой мёртвых тестов. Без правил «временно отключим» превращается в «навсегда».
L3 · Senior

Как ускорить suite, который идёт 2 часа — частый Senior-вопрос:

  • Параллелизация — xdist + шардинг по раннерам (самый большой выигрыш)
  • Разделить по уровням — быстрые на каждый PR, медленные E2E ночью
  • Убрать дубли — E2E, повторяющие то, что уже покрыто integration
  • Сдвинуть вниз по пирамиде — заменить часть медленных E2E на быстрые contract/integration
  • Кэш зависимостей и тестовых данных
  • Убрать sleep — заменить на явные ожидания с polling

Allure — главный инструмент отчётности в CI: показывает прошедшие/упавшие тесты, шаги, прикладывает запросы, ответы, скриншоты, логи. Без хорошего отчёта разбор падений в CI превращается в чтение сырых логов.

Test impact / selective testingСовременная идея для больших suite: запускать не всё подряд, а релевантный набор — тесты, затронутые изменёнными файлами/сервисами. Экономит время на каждом PR. Полный регресс при этом гоняют по расписанию (ночью) или перед релизом. Не обязательно к собесу, но знать концепцию — плюс.
Deployment verification — связка с observabilityПосле деплоя проверка не заканчивается. Что делают: smoke (критичные сценарии сразу после выката), health checks (сервис поднялся и отвечает), synthetic checks (боты гоняют ключевые пути), error rate / latency (метрики не выросли против baseline), rollback criteria (заранее заданные пороги: если error rate > X% — откат). Это мост между CI/CD и observability (раздел Архитектура): деплой «закончен», когда метрики подтвердили здоровье, а не в момент выката.
Главная мысль про valueЗелёный пайплайн ≠ качество. Автотесты дают ценность, только если: ловят реальные баги (не просто «зелёная галочка»), стабильны (им доверяют), быстры (не тормозят разработку). Suite из 5000 flaky-тестов, которые все перезапускают не глядя, — такой набор снижает доверие к автоматизации и требует пересмотра.
Каверзный вопрос«Как понять, что автотесты реально дают value, а не просто зелёную галочку?» — Смотреть на: 1) escape rate — сколько багов прошло тесты и нашлось в проде (мало = тесты ловят), 2) доверие команды — красный билд реально останавливает релиз или его игнорируют, 3) flaky rate — низкий, 4) тесты падали на реальных регрессиях, а не только на своих изменениях. Если suite всегда зелёный, но дефекты попадают в production — он ничего не проверяет.
