UI и E2E-автоматизация

// 6 подтем · ручное UI/UX + Playwright на Python

О разделе Два пласта в одном: ручное UI/UX-тестирование (что проверять глазами, против дизайна) и UI-автоматизация на Playwright (Python-биндинг). Playwright сейчас на подъёме и часто требуется как E2E-инструмент — теснит Selenium. L1/L2/L3 — уровни глубины.
01 UI/UX: что проверять manual
L1 · Основы

UI-тестирование — проверка интерфейса глазами и руками: видит ли пользователь то, что должен, и работает ли это так, как ожидается. Базовые категории:

  • Функциональность — кнопки, формы, ссылки делают то, что должны
  • Визуал — вёрстка не поехала, элементы на месте, нет наложений
  • Состояния — загрузка, пустой экран, ошибка, успех
  • Ввод — валидация полей, сообщения об ошибках
ГлавноеUI ≠ только «красиво ли». Это про то, может ли пользователь выполнить задачу без лишних затруднений. Кнопка есть, но не нажимается; форма отправилась, но нет фидбэка; ошибка отображается, но непонятно что делать — это UI-баги, даже если «пиксели на месте».
L2 · Детали

Состояния UI — то, что часто забывают проверить (а баги живут именно тут):

СостояниеЧто проверить
Loadingспиннер/скелетон, не зависает, нет «прыжков» контента
Emptyпустой список — понятное «ничего нет», а не белый экран
Errorошибка сети/сервера — внятное сообщение + что делать
Successдействие прошло — есть подтверждение
Edgeочень длинный текст, много элементов, нет данных

Формы — отдельный фокус: валидация (вживую и при сабмите), сообщения об ошибках у нужных полей, поведение кнопки (disabled пока невалидно), что происходит при двойном клике.

Microcopy — частый источник UX-баговТексты интерфейса проверяют отдельно: понятны ли сообщения об ошибках («что-то пошло не так» vs «email уже занят, войдите»), осмысленны ли тексты кнопок (не «OK/Отмена» в опасном действии), есть ли внятный текст в empty states, понятны ли confirmation-сообщения (после действия ясно, что произошло). Технически работает, но непонятный текст = UX-баг.
L3 · Senior
В реальном миреSenior QA не просто «проверяет по тикету», а смотрит на продукт глазами пользователя: где затруднение в флоу, что неинтуитивно, где можно срезать шаг. Это product-orientation — ценится выше, чем умение находить съехавший пиксель. Полезная привычка — регулярно проходить ключевые сценарии как живой пользователь и сравнивать с конкурентами (как у них решён этот флоу).
Каверзный вопрос«С чего начнёшь тестировать новый экран?» — Не «проверю элементы без приоритизации». Сначала: 1) что это за экран, какая задача пользователя, 2) happy path — основной сценарий работает, 3) состояния (loading/empty/error), 4) ввод и границы, 5) визуал против дизайна, 6) что будет «не так» (нет сети, медленно, чужие данные). От задачи пользователя — к деталям, а не наоборот.
02 Mockups, cross-browser, responsive visual
L1 · Основы

Тестирование против mockups — сверка реализации с макетом дизайнера (Figma и т.п.). Проверяешь, что разработчик собрал то, что нарисовали: отступы, шрифты, цвета, размеры, состояния элементов.

Что сверять — не только пикселиПиксель-перфект — не главная ценность. Важнее: соответствие дизайн-системе (те же компоненты, токены, отступы из общей библиотеки), все ли состояния реализованы (hover, active, disabled, error), и отсутствие регрессий при изменениях. Точное совпадение каждого пикселя вторично — система и состояния первичны.
L2 · Детали

Cross-browser — проверка в разных браузерах (Chrome, Firefox, Safari, Edge). Движки рендерят по-разному: Safari/WebKit часто имеет отличия в рендеринге и поддержке API. Что чаще ломается: CSS-флексы/гриды, шрифты, даты, видео/аудио, новые API.

Responsive / адаптив — поведение на разных размерах: десктоп, планшет, мобильный. Проверять на брейкпоинтах (точках, где меняется вёрстка) и между ними. Частые баги: горизонтальный скролл, наложение элементов, обрезанный текст, нерабочие тач-зоны.

Mobile-специфика веб-интерфейсаНа мобильном вебе свои частые баги: touch targets (кнопки достаточного размера для пальца, не слипаются), экранная клавиатура перекрывает input (поле уезжает под клавиатуру), sticky header/footer (не ломаются при скролле и клавиатуре), orientation change (поворот экрана не ломает верстку и не теряет данные), safe areas на iOS (контент не заходит под «чёлку»/индикатор).

Инструменты: DevTools device mode (быстрая проверка размеров), реальные устройства (точнее эмуляции), облачные фермы (BrowserStack, Sauce Labs) — много браузеров/устройств без своего парка.

L3 · Senior

Visual regression testing — автоматическая сверка скриншотов: эталон vs текущее состояние, инструмент подсвечивает пиксельные различия. Ловит случайные визуальные поломки, которые человек пропустит. Инструменты: Percy, Applitools, встроенный скриншот-сравниватель Playwright.

Подводный камень visual testingСкриншот-тесты хрупкие: меняются шрифты на разных ОС, анимации, динамические данные (даты, аватары) → ложные срабатывания. Лечится: маскировать динамические зоны, фиксировать данные и шрифты, задавать порог различий. Без этого visual-тесты быстро начинают нестабильно падать (flaky) и им перестают доверять.
Каверзный вопрос«Стоит ли автоматизировать сверку с дизайном?» — Частично. Пиксельную сверку отступов/цветов делает visual regression (Percy/Applitools) — это автоматизируемо. Но «удобно ли пользоваться, логичен ли флоу» автомат не оценивает — это ручная и продуктовая работа. Автоматизируй стабильный визуал, оставь UX-суждение человеку.
03 Accessibility и локализация a11y · l10n
L1 · Основы

Accessibility (a11y) — доступность для людей с ограничениями: незрячие (скринридеры), слабовидящие, без мыши (только клавиатура). Базовое, что проверяют:

  • Клавиатура — всё доступно через Tab, виден фокус, нет ловушек
  • Контраст — текст читается (есть нормы WCAG по контрасту)
  • Alt-тексты — у картинок есть описания для скринридеров
  • Семантика — кнопка это <button>, а не <div> с onclick
L2 · Детали

Локализация (l10n) — работа приложения на разных языках и в разных регионах. Что проверять:

АспектЧто ломается
Длина текстанемецкий длиннее английского на ~30% → текст не влезает, обрезается, ломает вёрстку
RTLарабский/иврит справа налево → весь layout зеркалится
Форматыдаты (MM/DD vs DD/MM), числа (1,000.5 vs 1 000,5), валюты
Непереведённоезахардкоженные строки, оставшиеся на английском
Кодировкиюникод, эмодзи, спецсимволы языков
WCAG 2.2 — конкретные критерииW3C рекомендует WCAG 2.2 как актуальный стандарт. Проверяемые вещи: контраст 4.5:1 для обычного текста (3:1 для крупного), видимый focus на всех интерактивных элементах, полное управление с клавиатуры, focus not obscured (элемент в фокусе не перекрыт другим, например sticky-хедером), достаточный target size (минимальный размер кликабельной зоны). Это конкретные пункты, а не «сделайте доступно».

a11y-инструменты: axe DevTools, Lighthouse (встроен в Chrome) — автоматический аудит доступности и базовых проблем.

L3 · Senior
В реальном миреЛокализация — частый отдельный навык в вакансиях (особенно у продуктов с глобальной аудиторией). Ключевая мысль: псевдолокализация — заменить текст на удлинённую версию с акцентами (Ĥéĺĺö Ŵöŕĺđ) ещё до реального перевода, чтобы заранее поймать обрезанный текст и захардкоженные строки. Автоматизировать можно проверку, что все строки вынесены в файлы переводов и нет пропущенных ключей.
Каверзный вопрос«Как тестировать локализацию, не зная всех языков?» — Не нужно знать языки. Проверяй: 1) длину — на самом длинном языке (немецкий) текст влезает, 2) RTL — арабский не ломает layout, 3) непереведённое — нет английских строк там, где должен быть перевод, 4) форматы — даты/числа/валюты под регион, 5) спецсимволы — юникод отображается. Псевдолокализация ловит большинство без знания языка.
04 Playwright: основы (Python) automation
L1 · Основы

Playwright — современный инструмент UI-автоматизации от Microsoft. Управляет браузером из кода: открывает страницы, кликает, вводит, проверяет. Есть Python-биндинг (playwright-python) + плагин pytest-playwright.

def test_login(page): # page — фикстура pytest-playwright page.goto("https://app.com/login") page.fill("#email", "user@test.com") page.fill("#password", "secret") page.click("button[type=submit]") expect(page).to_have_url("**/dashboard")
Чем хорошГлавное преимущество — авто-ожидания: Playwright сам ждёт, пока элемент появится и станет кликабельным, перед действием. Это убирает большинство sleep и ручных ожиданий, на которых спотыкается Selenium. Меньше flaky из коробки.
L2 · Детали

Ключевые объекты:

  • Browser — запущенный браузер (Chromium, Firefox, WebKit — все три из коробки)
  • Context — изолированная сессия (свои куки, storage). Как «новый инкогнито» — для изоляции тестов
  • Page — вкладка, с которой работаешь

Сравнение с Selenium (частый вопрос):

PlaywrightSelenium
Ожиданияавто-ожидания из коробкинадо настраивать (waits)
БраузерыChromium/Firefox/WebKit встроеныдрайверы отдельно
Скоростьбыстреемедленнее
Где силёнсовременный web и E2Eраспространён в enterprise/legacy

Не «Selenium устарел»: Playwright часто удобнее для modern web и E2E, Selenium всё ещё широко используется в enterprise и legacy-проектах с большой экосистемой.

L3 · Senior

Фишки, которые ценятся:

  • Trace viewer — запись прогона с DOM-снимками на каждом шаге. Упал тест в CI — открыл трейс, видишь что было на экране. Спасение для дебага нестабильных тестов
  • Artifacts on failure — настроить screenshot, video, trace при падении. В CI это превращает «тест упал» в «вот скриншот, видео и трейс момента падения» — диагностика без воспроизведения
  • Codegen — записывает действия в браузере и генерит код теста (черновик, не финал)
  • Перехват сети — мокать API-ответы прямо в UI-тесте (page.route): протестировать, как UI реагирует на 500 от бэка, не трогая бэк
  • Projects — конфигурации под разные браузеры/viewport (Chromium/Firefox/WebKit, desktop/mobile-профили) одним прогоном
  • Параллелизм — из коробки через pytest-xdist + изоляция через context
Каверзный вопрос«Почему Playwright меньше флакает, чем Selenium?» — Главное — авто-ожидания (auto-waiting): перед каждым действием Playwright проверяет actionability — элемент visible, stable, receives events, enabled — и ждёт этого сам. В Selenium ожидания надо прописывать руками (explicit waits), забыл — получил нестабильное падение на таймингах. Плюс Playwright управляет браузерами через собственные browser-server механизмы (для Chromium это близко к CDP), а не через отдельный WebDriver.
05 Локаторы, ожидания, Page Object core
L1 · Основы

Локатор — как найти элемент на странице. Playwright рекомендует искать по тому, что видит пользователь, а не по хрупким CSS/XPath:

page.get_by_role("button", name="Войти") # по роли + тексту (лучшее) page.get_by_text("Добро пожаловать") # по тексту page.get_by_label("Email") # по label поля page.get_by_test_id("submit") # по data-testid page.locator(".btn-primary") # CSS (хрупче)
L2 · Детали

Иерархия локаторов от устойчивого к хрупкому:

  • get_by_role / get_by_label — по доступности и тексту, не ломается при рефакторинге вёрстки
  • get_by_test_id (data-testid) — выделенный атрибут для тестов, стабильный
  • CSS-селекторы — ок, но ломаются при смене классов/вёрстки
  • XPath по позиции (div[3]/span[2]) — самое хрупкое, избегать

Ожидания через expect — web-first assertions с авто-retry:

expect(page.get_by_text("Успешно")).to_be_visible() # expect сам ждёт появления до таймаута, не нужен sleep

Это убивает классическую болезнь UI-тестов — гонки и sleep. Не «подожди 3 секунды», а «дождись, пока появится».

L3 · Senior

Page Object Model (POM) — распространённый паттерн UI-автоматизации: страница = класс с локаторами и методами. Тесты не знают про селекторы, работают с понятными действиями (в Playwright нередко обходятся и более лёгкими fixtures/helpers вместо тяжёлых Page Object):

class LoginPage: def __init__(self, page): self.page = page self.email = page.get_by_label("Email") self.submit = page.get_by_role("button", name="Войти") def login(self, email, pwd): self.email.fill(email) ... self.submit.click() # в тесте: LoginPage(page).login("user", "pass") — читается как сценарий
Зачем POMТа же идея, что client-слой в API-тестах (раздел Pytest): спрятать детали за понятным интерфейсом. Поменялся локатор — правишь в одном месте (Page Object), а не в 50 тестах. Тесты читаются как бизнес-сценарии, а не как клики по селекторам.
Каверзный вопрос«Какие локаторы выбирать и почему?» — Приоритет — по тому, что видит пользователь: get_by_role, get_by_label, get_by_text. Они устойчивы к рефакторингу вёрстки и заодно проверяют доступность. Если не получается — data-testid (стабильный, но требует поддержки в коде). CSS/XPath по позиции — крайний случай, они ломаются от любой смены вёрстки и дают flaky.
06 UI-автоматизация на практике practice
L1 · Основы

Главный вопрос UI-автоматизации — что автоматизировать, а что нет. UI-тесты дорогие и хрупкие, поэтому их держат мало (вершина пирамиды, раздел Теория).

Что автоматизироватьКритичные сквозные пути, которые: 1) важны для бизнеса (логин, оплата, ключевой флоу), 2) стабильны (UI не переделывают каждую неделю), 3) дорого проверять руками каждый релиз. Остальное — ручная и API-проверка.
L2 · Детали

Почему UI-тесты флакают и как лечить:

Причина flakyЛечение
гонки / таймингиавто-ожидания и expect вместо sleep
хрупкие локаторыget_by_role / data-testid вместо XPath по позиции
зависимость от данныхсоздавать свои данные через API перед тестом
нестабильный бэкендмокать сеть (page.route) для UI-логики
зависимость тестовизоляция через свежий context

Ускорение setup через API: не кликать логин в UI на каждый тест, а залогиниться через API и подложить storage state — UI-тест начинается уже авторизованным. Сильно быстрее и стабильнее.

L3 · Senior
В реальном миреЗрелый подход — UI-тестов мало, и они про сквозные пути, а не про каждую кнопку. Логику проверяют на уровне API (быстро, стабильно), UI-автоматизацией — что критичный флоу в принципе работает в браузере. Частая ошибка джунов — автоматизировать всё через UI: получается медленный флакающий suite, которому никто не доверяет. Setup (логин, тест-данные) — через API, не через клики.
Делегируй LLMГде помогает: черновик Page Object по HTML, генерация теста по сценарию, разбор хрупкого локатора, по screenshot + DOM предложить устойчивые локаторы и возможные a11y-проблемы.
Пример промпта: «Вот HTML формы [код]. Сделай Page Object на Playwright Python с устойчивыми локаторами (get_by_role/label) и метод login().»
Что проверить руками: устойчивость локаторов на реальной странице и сценарии, которые LLM не знает (доменный флоу, edge cases).
Каверзный вопрос«Когда НЕ стоит автоматизировать UI?» — Когда: 1) UI часто меняется (тесты устаревают быстрее, чем окупаются), 2) то же можно проверить на API-уровне (дешевле и стабильнее), 3) разовая проверка (быстрее руками), 4) сложная визуальная/UX-оценка (автомат не оценивает «удобно ли»). Автоматизируй стабильные критичные пути, а не «всё подряд, потому что можем».
