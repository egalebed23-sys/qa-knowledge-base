01 Python для QA: структуры данных basics
L1 · Основы

На собесе для QA-автоматизатора кодинг обычно middle-уровня: разобрать строку, посчитать что-то в списке, объединить данные. Четыре базовые структуры:

СтруктураЧто этоКогда
listупорядоченный изменяемый списокпоследовательность, порядок важен
dictпары ключ→значениебыстрый доступ по ключу, группировка
setмножество уникальныхуникальность, быстрая проверка наличия
tupleнеизменяемый списокфиксированный набор (координаты, запись)
L2 · Детали

Главное для задач — выбор структуры влияет на скорость. Проверка «есть ли элемент»:

# Медленно: поиск в списке — O(n), проходит весь список if x in my_list: ... # Быстро: поиск в set/dict — O(1), мгновенно if x in my_set: ...
Аналогияset/dict — как индекс в БД: ищет сразу по хешу, не перебирая всё. list — как таблица без индекса: чтобы найти элемент, надо пройти все строки. Когда нужно много раз проверять наличие — складывай в set, а не в list.

Полезные коллекции из стандартной библиотеки:

  • collections.Counter — подсчёт частот (Counter(nums).most_common(3))
  • collections.defaultdict — словарь с дефолтным значением (удобно для группировки)
  • set операции — пересечение a & b, разность a - b, объединение a | b

Сложность операций — что спрашивают:

Операцияlistdict / set
доступ по ключу/индексуO(1) по индексуO(1) по ключу
поиск x inO(n)O(1)

вставка в началоO(n) (сдвиг)—
В реальном миреДля QA-автоматизатора алгоритмический хардкор (динамическое программирование, графы) почти не нужен — это для SWE/SDET в FAANG. Чаще проверяют: умеешь ли ты разобрать данные, выбрать подходящую структуру, написать читаемый код и оценить, не получится ли случайно O(n²) на больших данных. Идиоматичный Python (list comprehension, Counter, with) ценится выше «олимпиадных» решений.
02 Синтаксис и особенности языка language
L1 · Основы

Три базовых вопроса по самому языку:

  • Изменяемые vs неизменяемые типы. Mutable: list, dict, set. Immutable: int, str, tuple, frozenset. Неизменяемые нельзя поменять на месте — создаётся новый объект
  • is vs ==. == сравнивает значения (равны ли). is сравнивает идентичность (тот же объект в памяти)
  • None — единственный объект, проверять через is None, не == None
a = [1, 2]; b = [1, 2] a == b # True — значения равны a is b # False — разные объекты в памяти
L2 · Детали

Comprehensions — компактное создание коллекций, очень питоничны:

[x*2 for x in nums] # list [x for x in nums if x > 0] # с фильтром {k: v for k, v in pairs} # dict {x for x in nums} # set (x for x in nums) # generator (ленивый)

Срезы (slicing):

nums[1:3] # элементы с 1 по 2 nums[::-1] # разворот списка nums[::2] # каждый второй nums[-1] # последний

*args и **kwargs — переменное число аргументов: *args собирает позиционные в tuple, **kwargs — именованные в dict. Распаковка работает и наоборот: func(*list, **dict).

Полезное: f-strings (f"{name}: {x}"), enumerate (индекс + элемент), zip (пара списков), распаковка (a, *rest = nums).

L3 · Senior

Генераторы (yield) — ленивые: отдают значения по одному, не держат всё в памяти. Незаменимы для больших данных/логов:

def read_large_file(path): with open(path) as f: for line in f: yield line.strip() # по строке, не весь файл в память

Декораторы — функция, оборачивающая другую функцию (добавляет поведение, не меняя код). Ты их уже используешь: @pytest.fixture, @pytest.mark.parametrize — это декораторы.

Контекстные менеджеры (with) — гарантируют очистку (закрыть файл, соединение) даже при исключении. Под капотом — методы __enter__ / __exit__.

copy vs deepcopyПрисваивание b = a для списка — это та же ссылка, не копия: меняешь b — меняется a. copy.copy() — поверхностная копия (вложенные объекты общие). copy.deepcopy() — полная копия всего дерева. Частый источник багов с мутабельными структурами.
Каверзный вопрос«Что выведет a = [[]] * 3; a[0].append(1)?» — [[1], [1], [1]], а не [[1], [], []]. Потому что * 3 создаёт три ссылки на один и тот же вложенный список, а не три разных. Классическая ловушка с мутабельностью. Правильно: [[] for _ in range(3)].
03 ООП в Python oop
L1 · Основы

ООП нужно QA-автоматизатору на практике: фреймворк строится на классах (Page Object, client-слой, базовые классы тестов). Базовое:

class ApiClient: def __init__(self, base_url): # конструктор self.base_url = base_url # атрибут def get(self, path): # метод return requests.get(self.base_url + path) client = ApiClient("http://api") # объект (экземпляр)

self — ссылка на сам объект, через неё методы обращаются к атрибутам. __init__ — конструктор, вызывается при создании.

L2 · Детали

Наследование — класс берёт поведение родителя и расширяет:

class BaseTest: def setup(self): ... class OrderTest(BaseTest): # наследует BaseTest def setup(self): super().setup() # вызвать родительский # + своя логика

Три типа методов:

  • @staticmethod — не зависит от объекта (утилита внутри класса)
  • @classmethod — работает с классом, а не экземпляром (cls)
  • @property — метод, который вызывается как атрибут (obj.full_name без скобок)

Magic / dunder методы — управляют поведением объекта: __init__ (создание), __str__ (печать), __repr__ (отладка), __eq__ (==), __len__ (len()).

L3 · Senior

Зачем это QA на практике:

  • Page Object / client-слой — классы, прячущие детали за понятными методами (раздел Pytest)
  • Базовые классы тестов — общий setup через наследование
  • __eq__ — чтобы сравнивать объекты в ассертах по значению, а не по ссылке
  • Контекстные менеджеры — свой with для setup/teardown ресурса
В реальном миреГлубокое ООП (метаклассы, дескрипторы, MRO при сложном множественном наследовании) QA-автоматизатору почти не нужно — это для разработчиков библиотек. Достаточно: классы, наследование, super(), основные dunder-методы, @property/@staticmethod/@classmethod. Этого хватает, чтобы писать и читать тест-фреймворк.
Каверзный вопрос«Чем __str__ отличается от __repr__?» — __str__ — «человеческое» представление для пользователя (print(obj)). __repr__ — однозначное техническое для отладки, в идеале такое, чтобы по нему можно было воссоздать объект (repr(obj), вывод в консоли/логах). Если определён только один — определяй __repr__, он используется как fallback.
04 Строки и регулярные выражения часто на собесе
L1 · Основы

Половина live-coding задач для QA — про строки (валидация, парсинг). Строка неизменяема (immutable): методы возвращают новую строку, а не меняют исходную. Базовые методы:

s = "Hello World" s.lower(), s.upper() # 'hello world', 'HELLO WORLD' s.strip() # убрать пробелы по краям s.split(" ") # ['Hello', 'World'] " ".join(["a", "b"]) # 'a b' s.replace("l", "L") # 'HeLLo WorLd' s.startswith("He") # True "World" in s # True — подстрока
Проверки символовСпрашивают постоянно (валидация пароля, ввода): "123".isdigit() (только цифры), "abc".isalpha() (только буквы), "a1".isalnum() (буквы+цифры), ch.isupper() / ch.islower() (регистр). Возвращают bool.
L2 · Детали

any / all с генератором — главный приём для проверок «есть ли хоть один / все ли»:

# есть ли хоть одна цифра в строке any(ch.isdigit() for ch in s) # все ли символы — буквы all(ch.isalpha() for ch in s) # есть ли заглавная И строчная (валидатор пароля) has_upper = any(ch.isupper() for ch in s) has_lower = any(ch.islower() for ch in s)

Срезы (slicing): s[1:3] (с 1 по 2, stop не включается), s[::-1] (реверс — частый трюк для палиндрома), s[:2], s[-1] (последний).

Счётчик символов — типовой приём:

counts = {} for ch in "hello": counts[ch] = counts.get(ch, 0) + 1 # {'h':1, 'e':1, 'l':2, 'o':1} — или короче: Counter("hello")
L3 · Senior

Регулярные выражения (модуль re) — нужны для валидации (email, пароль, телефон) и парсинга (логи). Паттерны пишут как r"..." (raw string), чтобы не экранировать \ дважды.

import re re.search(r"\d+", text) # первое совпадение где угодно (или None) re.match(r"\d+", text) # только с НАЧАЛА строки re.findall(r"\d+", text) # все совпадения списком re.fullmatch(r"\d+", text) # совпадает ли ВСЯ строка bool(re.search(r"[A-Z]", s)) # есть ли заглавная
ПаттернЗначение
\d / \Dцифра / не цифра
\w / \Wбуква/цифра/_ / наоборот
\s / \Sпробельный / не пробельный
[abc] / [^abc]один из / не из
[a-z] / [A-Za-z]диапазон / буквы любого регистра
* / + / ?0+ / 1+ / 0 или 1
{3} / {2,5}ровно 3 / от 2 до 5
^ / $начало / конец строки
Важная оговоркаСложную валидацию email / URL / телефонов не пиши самодельным regex в проде — это классический источник багов (валидный email по RFC регулярка почти никогда не покрывает). Используй проверенную библиотеку или серверную валидацию. Regex хорош для логов и простых проверок, а не для «угадай все форматы email одной строкой».
Каверзный вопрос«Разница re.search, re.match, re.fullmatch?» — search ищет совпадение где угодно в строке, match — только с начала строки, fullmatch — совпадает ли вся строка целиком. Для валидации «строка целиком соответствует формату» нужен fullmatch (или ^...$ в search), иначе "123abc" пройдёт проверку на \d+ через search/match.
Делегируй LLMГде помогает: regex легко забыть — попроси построить и объяснить паттерн.
Пример промпта: «Напиши regex для валидации пароля: минимум 8 символов, хотя бы одна цифра, заглавная и спецсимвол. Объясни паттерн по частям.»
Что проверить руками: прогнать на граничных (пустая строка, юникод, очень длинный ввод) — regex часто пропускает edge cases.
05 Частые вопросы «в чём разница» tricky
L1 · Основы

Любимый формат Python-собеса — «чем X отличается от Y». Самое спрашиваемое:

arr = [1, 2] arr.append([3, 4]) # [1, 2, [3, 4]] — добавил ОДИН элемент (список) arr = [1, 2] arr.extend([3, 4]) # [1, 2, 3, 4] — добавил КАЖДЫЙ элемент
append vs extendappend(x) добавляет x целиком как один элемент (даже если x — список, он вложится). extend(iterable) добавляет каждый элемент из переданного. Запомнить: append — «положить коробку в конец», extend — «высыпать содержимое коробки».
L2 · Детали
ПараРазница
append vs extendдобавить элемент целиком vs добавить каждый элемент итерируемого
append vs insertв конец vs по индексу (insert(0, x) — в начало)
remove vs pop vs delпо значению vs по индексу (и вернуть) vs по индексу (без возврата)
sort() vs sorted()меняет список на месте (None) vs возвращает новый отсортированный
list vs tupleизменяемый vs неизменяемый (tuple можно ключом dict, list — нет)
is vs ==тот же объект в памяти vs равны по значению
d[key] vs d.get(key)KeyError если нет vs None (или дефолт) если нет
/ vs //деление с float vs целочисленное (7/2=3.5, 7//2=3)
L3 · Senior

Менее очевидные, но спрашивают:

  • list comprehension vs generator — [x for x in ...] создаёт весь список в памяти; (x for x in ...) — ленивый генератор, отдаёт по одному (экономит память на больших данных)
  • copy vs deepcopy — поверхностная копия (вложенные объекты общие) vs полная копия всего дерева (раздел «Синтаксис»)
  • and / or возвращают значение, не bool — a or b вернёт a, если оно truthy, иначе b (частый приём для дефолтов)
  • truthy / falsy — пустые [], {}, "", 0, None в условии = False; непустые = True
  • == vs is для чисел — мелкие int (-5..256) кэшируются, is случайно вернёт True; всегда сравнивай значения через ==, идентичность (is) — только для None
Каверзный вопрос«Что вернёт [1,2].append([3,4]) и чему равна переменная?» — Метод append (как и sort, extend, reverse) меняет список на месте и возвращает None. Так что x = arr.append(...) положит в x None, а не новый список — частая ошибка. Сам список станет [1, 2, [3, 4]]. Методы, меняющие на месте, возвращают None; те, что возвращают новое (sorted, срезы) — не меняют исходник.
06 Python для QA-фреймворка практика
L1 · Основы

То, что реально используешь в автотест-фреймворке каждый день — важнее алгоритмов. Начнём с исключений:

try: response = client.get("/orders") except ConnectionError as e: # ловим КОНКРЕТНОЕ, не голый except logger.error(f"Сеть недоступна: {e}") raise # пробрасываем дальше, не глотаем else: process(response) # если ошибки не было finally: cleanup() # выполнится всегда
Принцип для тестовВ тестах лучше падать явно, чем глотать ошибку. Голый except: прячет баги — тест «прошёл», хотя внутри всё сломалось. Лови конкретное исключение, а если поймал — либо обработай осмысленно, либо пробрось через raise. Молчаливый except: pass в тестах — это замаскированный непройденный тест.
L2 · Детали

Type hints (аннотации типов) — не обязательны, но в тестовом фреймворке делают код читаемым и ловят ошибки до запуска (через mypy):

def get_user(user_id: int) -> dict[str, Any]: ... def find(name: str) -> Optional[User]: # может вернуть User или None ...

Современный Python (3.11+): пиши list[str], dict[str, int] напрямую (не List из typing). Optional[X] = «X или None».

dataclass — удобно для тестовых DTO и моделей ответов API: класс данных без бойлерплейта:

from dataclasses import dataclass @dataclass class User: id: int name: str email: str = "" # значение по умолчанию u = User(1, "Иван") # __init__, __repr__, __eq__ — сгенерированы автоматически

Сравнение по значению (__eq__) из коробки — удобно в ассертах: assert actual == User(1, "Иван").

L3 · Senior

Pydantic — практичный инструмент для валидации моделей API: описываешь схему ответа как класс, Pydantic проверяет типы и структуру при парсинге. Незаменим для контроля контрактов в API-тестах:

from pydantic import BaseModel class OrderResponse(BaseModel): id: int status: str amount: float # парсит и ВАЛИДИРУЕТ ответ API — упадёт, если структура не та order = OrderResponse(**response.json()) assert order.status == "paid"
Зачем Pydantic в API-тестахВместо ручных проверок «есть ли поле, тот ли тип» — описываешь ожидаемую модель один раз, и Pydantic валидирует весь ответ: отсутствует поле, пришёл string вместо int, лишняя вложенность — тест падает с понятной ошибкой. Это контрактная проверка на стороне теста (см. раздел API → контракты). Используется в FastAPI, так что модели часто можно переиспользовать с бэкенда.

Окружение и зависимости — что нужно знать:

  • venv — изолированное окружение под проект (python -m venv .venv). Зависимости не ставят глобально — конфликты версий между проектами
  • pip + requirements.txt / pyproject.toml — фиксируют зависимости, чтобы у всех одинаковые версии
  • ruff / black / mypy — современный минимум качества: ruff (быстрый линтер), black (форматтер), mypy (проверка типов). Часто в pre-commit и CI

Логирование — в QA-фреймворке важнее, чем кажется:

import logging logger = logging.getLogger(__name__) logger.debug("детали запроса") # подробности для дебага logger.info("тест начат") # ход выполнения logger.warning("повтор попытки") logger.error("запрос упал") # ошибки
logging вместо printНе print() в тестах, а logging: уровни (debug/info/warning/error) позволяют фильтровать шум, логи идут в отчёт и CI. Критично: при логировании request/response не пиши токены, пароли, ключи в логи — они утекут в CI и хранилища. Маскируй чувствительные поля (связка с разделом Безопасность).
Современный Python для QAРабочий минимум — Python 3.11+ (актуальная ветка уже 3.14.x). Для собеса и работы не нужны новейшие фичи — достаточно владеть современным Python 3: f-strings, type hints, pathlib вместо os.path, dataclasses, comprehensions, pytest. Это и есть «современно и достаточно».
Делегируй LLMГде помогает: code review (найти mutable default, off-by-one, неверную сложность), рефакторинг простыни в читаемые функции, генерация Pydantic-моделей по примеру JSON-ответа.
Пример промпта: «Вот JSON-ответ API [пример]. Сгенерируй Pydantic-модель для валидации в тестах.»
Что проверить руками: опциональные/nullable поля и реальные граничные форматы — LLM часто делает все поля обязательными.
07 Сложность (Big O) complexity
L1 · Основы

Big O показывает, как растёт время (или память) с ростом данных. Не точные секунды, а характер роста:

СложностьНазваниеПример
O(1)константадоступ по индексу, dict lookup
O(log n)логарифмбинарный поиск
O(n)линейнаяодин проход по списку
O(n log n)линейно-лог.хорошая сортировка (sorted)
O(n²)квадратичнаявложенные циклы по одному списку
АналогияBig O — это «во сколько раз станет хуже, если данных станет в 10 раз больше». O(1) — без разницы. O(n) — в 10 раз дольше. O(n²) — в 100 раз. Поэтому случайный вложенный цикл на большом списке превращает быстрый тест в зависающий.
L2 · Детали

Как определить сложность по коду:

# O(n) — один цикл по n элементам for x in items: process(x) # O(n²) — цикл в цикле по тем же данным for x in items: for y in items: compare(x, y) # O(n) — ВАЖНО: in по списку внутри цикла = скрытый O(n²) for x in items: if x in other_list: # каждый in — O(n)! ...

Последний пример — частая ошибка: выглядит как один цикл, но in по списку внутри делает его O(n²). Замена other_list на set возвращает O(n).

L3 · Senior

Space complexity (память) — отдельно от времени. Создал set из всех элементов → O(n) памяти. Иногда жертвуют памятью ради скорости (кэш, словарь-индекс) — это осознанный trade-off.

Каверзный вопрос«Какая сложность у поиска дубликатов через два вложенных цикла и через set?» — Вложенные циклы: O(n²) время, O(1) память. Через set: O(n) время, O(n) память. Классический trade-off «время vs память»: set быстрее, но занимает место. На больших данных O(n) через set почти всегда выигрывает.

Задача 1. Валидные скобки. Проверить, что скобки ()[]{} правильно закрыты и вложены. "([])" → True, "([)]" → False.

def is_valid(s): stack = [] pairs = {')': '(', ']': '[', '}': '{'} for ch in s: if ch in '([{': stack.append(ch) elif ch in pairs: if not stack or stack.pop() != pairs[ch]: return False return not stack # стек пуст = все закрыты
ИдеяСтек (list как стек): открывающую скобку кладём, закрывающую — сверяем с вершиной. Аналогия — вложенные вызовы функций: последний открытый должен закрыться первым (LIFO). O(n) время, O(n) память.

Задача 2. Сравнить версии. "1.2.0" vs "1.10.1" → вернуть −1 / 0 / 1. Подвох: "1.10" > "1.2" (как числа, не строки).

def compare_versions(v1, v2): p1 = [int(x) for x in v1.split('.')] p2 = [int(x) for x in v2.split('.')] # дополнить нулями до равной длины: 1.2 vs 1.2.0 length = max(len(p1), len(p2)) p1 += [0] * (length - len(p1)) p2 += [0] * (length - len(p2)) for a, b in zip(p1, p2): if a != b: return 1 if a > b else -1 return 0

Ключевое — сравнивать как числа (через int), а не как строки, и дополнять короткую версию нулями.

L3 · Senior

Что показать на собесе помимо решения:

  • Уточни вход: могут ли быть буквы (1.2.0-rc1)? отрицательные? пустые строки?
  • Назови edge cases: пустая строка, разная длина, ведущие нули (01), невалидный формат
  • Оцени сложность: O(n) по длине строки
  • Сразу прикинь тесты (это плюс для QA): равные, больше/меньше, разная длина, edge cases
Каверзный вопрос«Почему "1.10" > "1.2" ломается при строковом сравнении?» — Потому что строки сравниваются посимвольно: '1' == '1', потом '1' < '2' → "1.10" считается меньше. А по смыслу версий 10 > 2. Поэтому версии всегда разбивают по точкам и сравнивают части как числа.

Задача 6. Объединить пересекающиеся интервалы. [[1,3],[2,6],[8,10]] → [[1,6],[8,10]].

def merge_intervals(intervals): intervals.sort(key=lambda x: x[0]) # сортируем по началу result = [intervals[0]] for start, end in intervals[1:]: if start <= result[-1][1]: # пересекается с последним result[-1][1] = max(result[-1][1], end) else: result.append([start, end]) return result
ИдеяСначала сортировка по началу (O(n log n)) — после неё достаточно одного прохода: если текущий интервал пересекается с последним в результате — расширяем, иначе добавляем новый. Сортировка — ключ ко многим интервальным задачам.
Уровень задачиЭто уже SDET/SWE-style вопрос. Для обычного QA Automation достаточно понимать принцип «сортировка + один проход», без углубления в алгоритмику. Если на собесе именно QA-роль — такие задачи редкость.
Каверзный вопрос«Какая сложность у merge intervals?» — O(n log n), и она определяется сортировкой, а не проходом. Сам проход после сортировки — O(n). Когда в задаче есть «пересечения / перекрытия интервалов» — почти всегда первый шаг это сортировка, и итоговая сложность O(n log n).
10 Качество кода и тестирование code quality
L1 · Основы

Напиши unit-тесты для функции — для QA это родная территория: применяешь те же техники тест-дизайна. Возьмём compare_versions из раздела 03:

import pytest @pytest.mark.parametrize("v1, v2, expected", [ ("1.0", "1.0", 0), # равны ("1.2", "1.10", -1), # 2 < 10 (не строки!) ("2.0", "1.9", 1), # больше ("1.0", "1.0.0", 0), # разная длина ]) def test_compare_versions(v1, v2, expected): assert compare_versions(v1, v2) == expected
ПринципПокрываешь по категориям: позитив (равны, больше, меньше), границы (разная длина, ведущие нули), негатив (пустая строка, невалидный формат). Это те же классы эквивалентности и граничные значения из раздела теории — просто применённые к коду.

«Найди баг в коде» — проверяют внимательность. Три классических Python-бага:

# БАГ 1: изменяемый дефолтный аргумент def add_item(item, items=[]): # список создаётся ОДИН раз! items.append(item) return items # add_item(1) → [1], add_item(2) → [1, 2] — накапливает! # Фикс: items=None, внутри if items is None: items = []
# БАГ 2: изменение списка во время итерации for x in items: if x < 0: items.remove(x) # пропускает элементы, портит итерацию # Фикс: items = [x for x in items if x >= 0]
# БАГ 3: off-by-one в диапазоне for i in range(len(items) - 1): # пропускает последний элемент process(items[i]) # Фикс: range(len(items)) если нужны все
L3 · Senior

«Как бы ты протестировал свой собственный код?» — сильный ответ показывает QA-мышление:

  • Happy path — работает на типичном входе
  • Границы — пустой вход, один элемент, очень большой вход
  • Негатив — невалидные данные, None, неверный тип
  • Edge cases — дубликаты, отрицательные, юникод, спецсимволы
  • Свойства — для сортировки: результат отсортирован и той же длины (property-based, Hypothesis)
В реальном миреПреимущество QA на кодинг-секции — не только написать функцию, но и сразу сказать, как её ломать. Многие кандидаты-разработчики пишут код и останавливаются. Когда ты добавляешь «вот тут я бы проверил пустой вход и переполнение» — это твоя сильная сторона, показывай её. Чистый код для QA: понятные имена, маленькие функции, без дублирования, ранний return вместо вложенных if.
Делегируй LLMГде помогает: генерация тестов к функции, поиск edge cases, ревью на баги.
Пример промпта: «Вот функция [код]. Напиши pytest-тесты: happy path, границы, негатив, edge cases. Используй parametrize. Подскажи, есть ли в коде баги.»
Что проверить руками: доменную логику и реальные граничные случаи продукта — LLM генерит шаблонные тесты, но не знает контекст твоей задачи.
Каверзный вопрос«Что не так с def f(x, lst=[])?» — Дефолтный список создаётся один раз при определении функции, а не на каждый вызов. Поэтому он сохраняет изменения между вызовами — частый источник «магических» багов. Правильно: def f(x, lst=None): if lst is None: lst = [].
11 Реальные задачи с собеса формат «почини класс»
L1 · Основы

Частый формат для QA — дан класс с багами или недостающими методами, нужно починить/реализовать. Две реальные задачи такого типа.

Задача A. Последовательные числа. Вернуть True, если в массиве есть 3+ подряд идущих числа (22, 23, 24). Менять можно только тело метода.

def test_method(self, array: list) -> bool: counter = 1 # одно число — уже "1 подряд" for i in range(len(array) - 1): # до предпоследнего: i+1 валиден if array[i+1] == array[i] + 1: counter += 1 if counter >= 3: return True else: counter = 1 # сброс при разрыве return False
L2 · Детали

Баги, которые надо было найти в задаче A (типовые):

  • range(1, len(array)) + обращение к array[i+1] → выход за границы (IndexError). Нужно range(len(array)-1)
  • counter стартовал с 0, но первая пара = уже 2 числа подряд → off-by-one
  • if counter == 3 — жёсткое равенство; безопаснее >= 3
  • Счётчик не сбрасывался при разрыве последовательности → ложные срабатывания

Тест-кейсы (проговорить как QA): пустой массив, 1 элемент, ровно 3 подряд, 4+ подряд, последовательность в начале/середине/конце, отрицательные, дубликаты ([1,2,2,3] — это 3 подряд?), убывающая.

L3 · Senior

Задача B. Валидатор пароля. Реализовать недостающие методы класса. Главная ловушка — обращение к атрибуту класса без self. (NameError) и сравнение строки с числом вместо len().

class Validator: CRITICAL_LENGTH = 8 def __init__(self, password): self.password = password def is_length_valid(self) -> bool: return len(self.password) >= self.CRITICAL_LENGTH # self.! и len()! def has_digits(self) -> bool: return any(ch.isdigit() for ch in self.password) def has_different_cases(self) -> bool: has_upper = any(ch.isupper() for ch in self.password) has_lower = any(ch.islower() for ch in self.password) return has_upper and has_lower def calculate_complexity(self) -> int: checks = [self.is_length_valid(), self.has_digits(), self.has_different_cases()] return sum(1 for ok in checks if ok) # True == 1
Две главные ловушки1) Внутри методов всегда self.x, не просто x — иначе NameError (это и был баг с CRITICAL_LENGTH). 2) self.password — это строка; сравнивать с числом нельзя, нужен len(), иначе TypeError. Эти два бага — суть задачи.
Каверзный вопрос«Зачем sum(1 for ok in checks if ok), можно ли проще?» — Можно sum(checks) напрямую: в Python True == 1, False == 0, поэтому сумма списка булевых = число True. Приём, который любят на собесе: sum([True, False, True]) вернёт 2. Знание, что bool — подтип int, отличает уверенный ответ.
Делегируй LLMГде помогает: тренировка формата «почини класс» — генерация похожих задач с багами.
Пример промпта: «Дай Python-класс с 3 типовыми багами (self, off-by-one, сравнение типов) и попроси меня их найти. Потом проверь мой ответ.»
Что проверить руками: свой разбор — называй баг, причину и минимальный фикс (не переписывай класс с нуля, как требуют на собесе).
