01 Pytest: основы basics
L1 · Основы

pytest — самый популярный фреймворк для тестов на Python. Тест — это обычная функция с assert:

# test_math.py def test_addition(): assert 2 + 2 == 4 def test_string(): assert "hello".upper() == "HELLO"

Запуск — команда pytest в терминале. Pytest сам находит файлы test_*.py и функции test_*.

Главное преимуществоОбычный assert вместо assertEqual/assertTrue из unittest. При падении pytest сам показывает, что было слева и справа: assert 2 + 2 == 5 → видно 4 != 5. Не нужно запоминать десятки методов проверки.
L2 · Детали

Соглашения об именах (pytest находит тесты по ним): файлы test_*.py, функции test_*, классы Test* (без __init__).

Полезные флаги запуска:

КомандаЧто делает
pytest -vподробный вывод (имя каждого теста)
pytest -k "login"запустить тесты, в имени которых есть login
pytest -xостановиться на первом падении
pytest --maxfail=2остановиться после 2 падений
pytest test_api.py::test_loginконкретный тест
pytest -sпоказывать print (не перехватывать вывод)

Структура теста — AAA (Arrange-Act-Assert):

def test_user_creation(): data = {"name": "Иван"} # Arrange user = create_user(data) # Act assert user.name == "Иван" # Assert
L3 · Senior

Проверка исключений:

import pytest def test_divide_by_zero(): with pytest.raises(ZeroDivisionError): result = 1 / 0
pytest vs unittestunittest есть в стандартной библиотеке, но многословный (классы, self.assertEqual). pytest — внешний пакет, но удобнее: голый assert, фикстуры (мощнее setUp/tearDown), параметризация, огромная экосистема плагинов (xdist, allure, mock, cov). В современных проектах де-факто стандарт — pytest.
Каверзный вопрос«Чем pytest лучше unittest?» — Голый assert с понятным выводом вместо методов assertEqual; фикстуры гибче setUp/tearDown (scope, переиспользование, зависимости); параметризация одним декоратором; богатая экосистема плагинов. unittest хорош тем, что в стандартной библиотеке — не нужно ставить.
02 Фикстуры главная тема
L1 · Основы

Фикстура (fixture) — подготовка того, что нужно тесту: данные, соединение с БД, авторизованный клиент. Объявляешь через @pytest.fixture, тест получает её как аргумент.

import pytest @pytest.fixture def user(): return {"name": "Иван", "role": "admin"} def test_user_is_admin(user): # fixture передаётся по имени assert user["role"] == "admin"
АналогияФикстура — это dependency injection для тестов: тест объявляет, что ему нужно (аргумент user), pytest сам создаёт и подставляет. Тест не занимается подготовкой — он просто получает готовое, как функция получает аргументы.
L2 · Детали

Setup/teardown через yield — код до yield это подготовка, после — очистка:

@pytest.fixture def db_connection(): conn = connect_to_db() # setup yield conn # отдаём тесту conn.close() # teardown (всегда выполнится)

Scope — как часто пересоздавать фикстуру:

ScopeСоздаётся
function (по умолчанию)на каждый тест заново
classодин раз на класс тестов
moduleодин раз на файл
sessionодин раз на весь прогон (дорогие ресурсы: БД, токен)

conftest.py — файл с общими фикстурами, доступными всем тестам в директории без импорта. Туда выносят переиспользуемое: клиент API, авторизацию, подключение к БД.

# conftest.py @pytest.fixture(scope="session") def api_client(): client = APIClient(base_url="...") client.login("user", "pass") return client # один авторизованный клиент на весь прогон
L3 · Senior

Фикстуры зависят от фикстур — выстраиваются в цепочку:

@pytest.fixture def order(api_client, user): # использует две другие фикстуры return api_client.create_order(user_id=user.id)

Фабрики — фикстура возвращает функцию, чтобы создавать объекты с разными параметрами:

@pytest.fixture def make_user(): def _make(role="user"): return create_user(role=role) return _make # в тесте: make_user(role="admin")

Встроенные фикстуры — даёт сам pytest, мини-примеры:

def test_file(tmp_path): # tmp_path — pathlib.Path, временная папка f = tmp_path / "data.txt" f.write_text("hello") def test_env(monkeypatch): # подмена env/атрибутов monkeypatch.setenv("API_URL", "http://test") def test_logs(caplog): # перехват логов для проверки do_something() assert "ошибка" in caplog.text

Используй tmp_path (на pathlib.Path), а не устаревший tmpdir.

autouse-фикстуры@pytest.fixture(autouse=True) применяется ко всем тестам в области без явного указания в аргументах. Удобно для сквозной подготовки (сброс состояния, общий лог). Опасно — создаёт «неявную магию»: тест зависит от того, чего не видно в его сигнатуре. Применяй для действительно сквозного и очевидного, иначе тесты становится трудно читать и дебажить.
Главное про teardownОчистка через yield выполнится даже если тест упал — это важно для изоляции. Создал данные в БД → удалил после, независимо от результата. Тесты не должны оставлять данные, которые сломают следующий прогон (связка с разделом БД).
Каверзный вопрос«Фикстура со scope=session создаёт юзера, тесты его меняют. В чём риск?» — Тесты перестают быть изолированными: один тест поменял состояние общего юзера → следующий падает или зависит от порядка. Дорогие ресурсы (соединение, токен) шарить можно, а изменяемые данные лучше создавать на каждый тест (function scope) или откатывать в teardown.
03 Параметризация parametrize
L1 · Основы

Параметризация — один тест прогоняется на множестве входных данных. Не дублируешь код, не пишешь 10 почти одинаковых тестов:

@pytest.mark.parametrize("value, expected", [ (2, 4), (3, 9), (5, 25), ]) def test_square(value, expected): assert value ** 2 == expected # pytest создаст 3 отдельных теста
АналогияПараметризация — как цикл по тест-кейсам, но каждая итерация — отдельный тест в отчёте. Упал один набор данных — видно, какой именно, остальные прошли. Это лучше, чем цикл внутри одного теста (там первое падение прячет остальные).
L2 · Детали

ids — читаемые имена для каждого набора (видны в отчёте):

@pytest.mark.parametrize("email, valid", [ ("user@mail.com", True), ("no-at-sign", False), ("", False), ], ids=["valid", "no_at", "empty"]) def test_email_validation(email, valid): assert is_valid_email(email) == valid

В отчёте будет test_email_validation[valid], [no_at], [empty] — сразу понятно, какой случай упал.

pytest.param — современный способ задать id и пометить отдельный набор (например, известный баг через xfail):

@pytest.mark.parametrize("value, expected", [ pytest.param(2, 4, id="positive"), pytest.param(0, 0, id="zero"), pytest.param(-1, 1, id="negative", marks=pytest.mark.xfail), ]) def test_square(value, expected): ...

Это прямое применение техник тест-дизайна (раздел теории): классы эквивалентности и граничные значения превращаются в наборы параметров.

L3 · Senior

Параметризованные фикстуры (indirect) — один тест прогоняется на разных конфигурациях (роли, окружения, клиенты). Особенно полезно для API-тестов:

@pytest.fixture def client(request): role = request.param # значение приходит из parametrize return make_client(role=role) @pytest.mark.parametrize("client", ["admin", "user"], indirect=True) def test_access(client): # тест прогонится для admin и user ...

indirect=True направляет параметр в фикстуру, а не в тест напрямую — фикстура успевает подготовить ресурс под каждый набор.

Тест-данные — где брать:

  • Inline — прямо в parametrize (простые случаи)
  • Фабрики — factory_boy для объектов с реалистичными полями
  • Faker — фейковые данные (имена, email, адреса)
  • Внешние файлы — JSON/CSV/YAML для больших датасетов
Уникальность данныхПри параллельном запуске тесты не должны конфликтовать за одни данные. Генерируй уникальное: f"user_{uuid4()}@test.com", а не хардкод "test@test.com" — иначе два теста создадут юзера с одним email и один упадёт.
Делегируй LLM«Сгенерируй parametrize-набор для теста валидации пароля: валидные, слишком короткие, без цифр, без спецсимволов, юникод, граничная длина. С ids через pytest.param.»

LLM хорошо превращает описание правил валидации в готовые параметризованные наборы по классам эквивалентности.
04 Моки в pytest mocking
L1 · Основы

Мок подменяет внешнюю зависимость (API, БД, время) на управляемую заглушку, чтобы тест не зависел от внешнего мира. Базовые инструменты — monkeypatch (встроенный) и mocker (плагин pytest-mock).

def test_with_monkeypatch(monkeypatch): # подменить переменную окружения monkeypatch.setenv("API_URL", "http://test") # подменить функцию monkeypatch.setattr(time, "time", lambda: 12345)
ЗачемТест должен проверять твою логику, а не работу чужого сервиса. Если код ходит в платёжку — мокаешь её ответ, и тест не зависит от того, доступна ли платёжка сейчас. Это делает тесты быстрыми и стабильными (см. также раздел API → контрактное тестирование).
L2 · Детали

pytest-mock — это плагин (не часть ядра pytest), даёт фикстуру mocker — обёртку над unittest.mock, удобнее в pytest:

def test_order_payment(mocker): # подменяем функцию, задаём что вернёт mock_pay = mocker.patch("app.payment.charge") mock_pay.return_value = {"status": "success"} result = create_order(amount=100) assert result.paid is True mock_pay.assert_called_once() # проверка вызова mock_pay.assert_called_with(amount=100)

Мок HTTP-вызовов — через библиотеку responses (как в разделе API):

@responses.activate def test_external_api(): responses.add(responses.GET, "https://api.ext.com/data", json={"key": "value"}, status=200) result = fetch_data() assert result["key"] == "value"
L3 · Senior
Patch там, где используетсяКлассическая ошибка: мокать функцию там, где она определена, а не там, где используется. Если app.service делает from app.payment import charge, то патчить надо app.service.charge, а не app.payment.charge. Иначе мок не сработает, тест ходит в реальный код.

Что мокать, а что нет (связка с разделом API):

  • Мокать: чужие внешние сервисы (платёжки, сторонние API), медленное (сеть), недетерминированное (время, random)
  • Не мокать: своё ядро (бизнес-логику, которую и тестируешь), свою БД (лучше реальная через Testcontainers)
Каверзный вопрос«Замокал всё, тесты зелёные — это хорошо?» — Не обязательно. Over-mocking опасен: тест проверяет моки, а не систему. Если замокал собственную бизнес-логику — тест зелёный, а реальный код сломан. Правило: мокай границы (внешнее, дорогое), а ядро тестируй по-настоящему. Зелёный тест на моках ≠ работающий код.
05 Организация фреймворка архитектура тестов
L1 · Основы

«Расскажи про свой автотест-фреймворк» — типовая структура E2E-проекта:

project/ ├── tests/ # сами тесты, разбиты по доменам/фичам │ ├── conftest.py # общие фикстуры (корневые) │ ├── api/ │ │ └── test_orders.py │ └── ui/ │ └── test_checkout.py ├── clients/ # обёртки над API (Client Object) │ └── order_client.py ├── pages/ # Page Object для UI │ └── checkout_page.py ├── data/ # фабрики и тест-данные │ └── factories.py ├── config/ # настройки окружений (dev/stage/prod) │ └── settings.py ├── utils/ # хелперы: логирование, ожидания, форматтеры ├── conftest.py # корневой conftest ├── pytest.ini # конфиг pytest, маркеры ├── pyproject.toml # зависимости, ruff/mypy ├── .env.example # шаблон переменных (без секретов!) └── README.md # как запустить
Главный принципТест отвечает на вопрос «что проверяем», всё остальное вынесено: как ходить (clients/pages), с какими данными (data), куда (config). Тест не должен содержать ни голых HTTP-запросов, ни селекторов, ни хардкода URL — он читается как бизнес-сценарий.
L2 · Детали

Что куда класть — best practices:

ПапкаЧто лежит / правило
tests/только проверки и сценарии. Разбивка по доменам/фичам, не «свалка». Имя теста описывает поведение
clients/ pages/взаимодействие с системой (API/UI). Меняется эндпоинт/локатор — правишь здесь, тесты не трогаешь
data/фабрики тест-данных, уникальные значения. Никаких хардкод-данных в тестах
config/настройки окружений (base_url, таймауты) через переменные. Переключение dev/stage без правки кода
conftest.pyфикстуры: корневой — общие (сессия, авторизация), вложенные — для своей группы тестов
utils/переиспользуемые хелперы — но не «склад всего подряд»

Слои (отделяют тесты от деталей): Client/Page-слой (как взаимодействуем) → Test-слой (что проверяем) → Data-слой (с чем). Зависимости идут в одну сторону: тест использует клиент и данные, но клиент не знает про тесты.

Маркеры — помечать и группировать:

@pytest.mark.smoke def test_critical_path(): ... # pytest -m smoke / pytest -m "not slow"

Config и секреты: окружения через переменные (.env локально, secrets в CI), .env в .gitignore, в репо только .env.example. Секреты — никогда в код (связка с разделом Безопасность).

L3 · Senior

Паттерны проектирования автотестов — частый senior-вопрос «какие паттерны используешь и зачем»:

ПаттернСуть и зачем
Page Object (POM)страница = класс с локаторами и методами. Тесты не знают про селекторы; поменялась вёрстка — правишь в одном месте. Для UI
Client / Service Objectто же для API: класс-обёртка над эндпоинтами домена. Тесты работают с client.create_order(), не с HTTP. Аналог POM для бэкенда
Factory / Builderсоздание тест-данных: фабрика генерит объект с дефолтами + уникальными полями, builder собирает сложный объект по частям. Убирает копипасту setup
Fixture / DIвстроен в pytest: подготовка и очистка ресурсов, внедрение зависимостей в тест через аргументы. Замена setup/teardown
Singletonодин экземпляр на прогон (конфиг, сессия). Осторожно с общим состоянием при параллели

Реже: Screenplay и Fluent interface — достаточно знать названия.

Общий принцип всех паттерновВсе они про одно — разделение ответственности и убирание дублирования: что проверяем (тест) отделено от как взаимодействуем (Page/Client Object) и с какими данными (Factory). Не нужно знать все паттерны наизусть — нужно понимать зачем: чтобы тесты читались как сценарии и менялись в одном месте, а не в пятидесяти.
Каркас ответа «расскажи про свой фреймворк»Структурируй так: 1) стек — pytest + requests, Allure для отчётов; 2) слои — клиенты по доменам (Client Object), тесты не знают про HTTP; 3) фикстуры — авторизация и тест-данные в conftest, очистка через teardown; 4) данные — фабрики, уникальные значения; 5) маркеры — smoke/regression для разделения прогонов; 6) CI — параллельный запуск, Allure-отчёт, gate на merge. Заверши тем, что улучшил: «сократил флак с X до Y», «ускорил прогон вдвое».
В реальном миреХороший фреймворк оценивают не по «красоте», а по тому, легко ли в нём писать новые тесты и читать чужие. Если для нового теста надо скопировать 20 строк настройки — слои плохие. Если тест читается как бизнес-сценарий (order = client.create(); assert order.status == "paid") — фреймворк удался.
Каверзный вопрос«Зачем client-слой, если можно просто requests в тесте?» — При росте до сотен тестов голые requests превращаются в ад поддержки: поменялся эндпоинт — правишь в 50 местах. Client-слой = единая точка: меняешь в одном месте. Плюс тесты читаются как сценарии, а не как HTTP-запросы. Это та же причина, по которой для UI делают Page Object.
Каверзный вопрос«Как ты организуешь E2E-проект, чтобы он не превратился в хаос?» — Best practices: 1) разделение слоёв — тесты (что), clients/pages (как), data (с чем), config (где); 2) тесты независимы — каждый сам готовит данные и убирает за собой, не зависит от порядка и от других тестов; 3) изоляция данных — уникальные значения через фабрики, setup через API (не через UI); 4) окружения через config, секреты вне кода; 5) E2E мало — только критичные сквозные пути, логику вниз по пирамиде на API; 6) стабильность — явные ожидания вместо sleep, борьба с flaky. Главный критерий зрелости: новый тест пишется быстро и читается без расшифровки.
Делегируй LLMГде помогает: разложить прототип в нормальную структуру фреймворка.
Пример промпта: «Вот тест с голыми requests [код]. Разнеси по слоям: вынеси HTTP в client-слой, данные в фабрики, предложи фикстуры и маркеры. Покажи итоговую структуру папок.»
Что проверить руками: где тесты не изолированы (общее состояние, зависимость от порядка) — LLM это часто пропускает, а именно тут рождается flaky.
06 Запуск, отчёты, CI execution
L1 · Основы

Тесты должны гоняться автоматически в CI на каждый PR (см. раздел CI/CD) и давать понятный отчёт. Базовые инструменты:

  • pytest-xdist — параллельный запуск
  • pytest-cov — покрытие кода
  • Allure — красивые отчёты с шагами и аттачами

Где конфигурируется pytest — частый вопрос. В pytest.ini или pyproject.toml (современнее, всё в одном файле):

# pyproject.toml [tool.pytest.ini_options] testpaths = ["tests"] # где искать тесты python_files = "test_*.py" # шаблон файлов с тестами addopts = "-v --strict-markers" # опции по умолчанию markers = [ "smoke: быстрые критичные тесты", "slow: долгие тесты", ]

--strict-markers — заставляет регистрировать все маркеры (защита от опечаток в @pytest.mark).

L2 · Детали

Параллельный запуск — главный способ ускорить:

pytest -n 4 # в 4 процесса (pytest-xdist) pytest -n auto # по числу ядер

Требование — изоляция тестов: не зависят друг от друга и от общих данных, иначе при параллели начнут конфликтовать.

Разделение прогонов маркерами (связка с CI/CD):

pytest -m smoke # быстрые на каждый PR pytest -m regression # полный набор ночью

Flaky-тесты — pytest-rerunfailures позволяет перезапускать упавшие (--reruns 2), но это костыль, не лечение: прячет проблему, а не решает (см. CI/CD).

skip / xfail — управление «непроходящими» тестами:

@pytest.mark.skip(reason="фича в разработке") def test_new(): ... @pytest.mark.skipif(sys.platform == "win32", reason="только Linux") def test_linux_only(): ... @pytest.mark.xfail(reason="известный баг JIRA-123") def test_known_bug(): ... # ожидаем падение
xfail — не способ прятать плохой тестxfail = «знаем, что падает, есть заведённый баг» — тест помечен честно, и если он вдруг пройдёт, pytest подсветит это (XPASS — баг починили). Это не то же, что закомментировать или skip сломанный тест, чтобы «не мешал». Используй xfail только с указанием реальной причины/тикета, иначе это маскировка проблемы.

Ускорить медленный suite на стороне pytest: параллелизация (-n auto), переиспользование дорогих фикстур (session scope), убрать sleep → явные ожидания. Процессная часть (разделить на pre-merge/nightly, сдвиг по пирамиде, value автотестов) — в разделе CI/CD.

Делегируй LLMГде помогает: генерация каркаса тестов по функции, конвертация ручных кейсов в код.
Пример промпта: «Напиши pytest-тесты для функции [код]: позитив, негатив, граничные, исключения. Используй parametrize и фикстуры.»
Что проверить руками: доменные кейсы и приоритизацию — их LLM не знает.
Каверзный вопрос«Тест иногда падает, иногда проходит. Что делаешь?» — Не --reruns и забыть. Ищу причину flaky: гонки/тайминги (sleep вместо ожидания), зависимость от порядка тестов, общие тест-данные при параллели, нестабильный внешний сервис (мокать), недетерминизм (время, random). retry — временная мера, пока чиню корень.
